## コンセプト

**PARAメソッド** をベースに、4つのカテゴリで知識を管理する。

> **P**rojects — **A**reas — **R**esources — **A**rchives

すべてのノートはこの4分類のどれかに属する。分類に迷ったときは「このノートはいつ使うか」で判断する。

| 問い | 分類 |
|---|---|
| 特定のゴールに向けて今動いている | Projects |
| 継続的に維持・学習しているトピック | Areas |
| いつか使えそうな参考資料 | Resources |
| 終わった・不要になった | Archives |

## フォルダ構成（PARA）

```
obsidian-vault/
├── 00_Inbox/                    ← キャプチャした未整理ノートの一時置き場
├── 10_Projects/                 ← 期限・ゴールのあるアクティブな作業
│   └── プロジェクト研究A/
│       └── 下書き/              ← 執筆ドラフト置き場
├── 20_Areas/                    ← 継続的に管理するトピック
│   └── 永続ノート/              ← Zettelkasten永続ノート
├── 30_Resources/                ← 将来再利用できる参考資料・論文
├── 40_Archives/                 ← 完了・不要になったノートの保管庫
├── 99_Templates/                ← Templaterテンプレート（実ノートではない）
├── Home.md                      ← ダッシュボード
├── CLAUDE.md                    ← Claude Code用の作業規約（自動参照）
├── VAULT_INDEX.md               ← Claudeのファイル索引（トークン節約用）
└── README.md                    ← このファイル
```

### 00_Inbox — 一時置き場

思いついたこと・気になったリンク・メモなどをとにかくここに書く。  
**週1回** レビューして、適切なフォルダに振り分けるか削除する。

### 10_Projects — プロジェクト

期限やゴールが明確な作業単位。プロジェクトごとにサブフォルダを作り、MOCで管理する。  
完了したプロジェクトは `40_Archives/` に移動する（削除しない）。

`下書き/` サブフォルダに執筆ドラフトを置き、研究ノートから成果物への橋渡しをする。

### 20_Areas — エリア

終わりのない継続的な関心事。スキル・習慣・知識領域など。

`永続ノート/` には Zettelkasten スタイルの原子的な概念ノートを蓄積する（詳細は後述）。

### 30_Resources — リソース

論文・書籍・外部資料のメモ。PDFは `30_Resources/` に置き、対応する読書メモ（`_メモ.md`）をセットで作成する。

### 40_Archives — アーカイブ

削除せずに保管しておく場所。検索はできるが基本的に参照しない。

### 99_Templates — テンプレート

Templater プラグインが使うテンプレートファイル。新規ノート作成時はTemplaterから呼び出す。

## 日々の使い方

### 典型的な1日の流れ

```
1. Home.md を開く
   → Inboxの未整理ノートと未完了タスクを確認

2. 作業する
   → 実験ノート・作業メモは 10_Projects/該当プロジェクト/ に作成
   → 気づき・参考リンクは 00_Inbox/ に一時保存
   → 論文を読んだら 30_Resources/ に読書メモを作成

3. 週1回：週次レビュー（月曜推奨）
   → Inboxを整理して PARA のどれかに振り分ける
   → 未完了タスクを確認・更新

4. 月1回：月次レビュー（月末推奨）
   → 今月の成果を振り返り、来月の目標を立てる
```

### 新しいノートの作り方

Templater を使う（`Ctrl+T` またはコマンドパレットから「Templater: Open Insert Template Modal」）。

| 状況 | 使うテンプレート | 保存先 |
|---|---|---|
| 論文を読む | `論文ノート用.md` | `30_Resources/該当トピック/` |
| 実験・作業ログを書く | `実験ノート用.md` | `10_Projects/プロジェクト名/` |
| 授業・勉強会のメモ | `授業ノート用.md` | `20_Areas/該当トピック/` |
| 概念を永続ノートにする | `永続ノート用.md` | `20_Areas/永続ノート/` |
| 新しいMOCを作る | `【MOC】.md` | 該当フォルダ |
| 週次レビュー | `週次レビュー用.md` | ルートまたは専用フォルダ |
| 月次レビュー | `月次レビュー用.md` | ルートまたは専用フォルダ |
| 執筆ドラフト | `執筆ドラフト用.md` | `10_Projects/プロジェクト名/下書き/` |

## ファイル命名規則

**大原則：時間軸のノートは日付を付け、知識のノートは付けない。**

- **知識ノート**（永続ノート・トピックノート・科目ノート・解説）＝ `概念名.md`（日付なし）。`[[概念名]]` で素直にリンクでき、内容が育っても名前が陳腐化しない。
- **時間軸ノート**（実験ログ・勉強会の各回・レビュー・日次）＝ 日付を付ける。

| 種別 | 命名規則 | 例 |
|---|---|---|
| 知識ノート（永続・トピック・科目・解説） | `概念名.md` | `決定木.md` / `K-means法.md` / `人工知能.md` |
| MOC | `【MOC】タイトル.md` | `【MOC】プロジェクト研究A.md` |
| 論文読書メモ | `略称_メモ.md` | `TF-Locoformer_メモ.md` |
| 実験ログ | `YYYYMMDD_タイトル.md` | `20260511_音源分離実験用スクリプトの作成.md` |
| 勉強会の各回 | `YYYYMMDD_勉強会NN.md` | `20260423_勉強会02.md` |
| 執筆ドラフト | `YYYYMMDD_タイトル_draft.md` | `20260601_中間報告スライド_draft.md` |
| 週次／月次レビュー | `YYYY-Www_週次レビュー.md` / `YYYY-MM_月次レビュー.md` | `2026-W23_週次レビュー.md` |
| 日次ノート | `YYYY-MM-DD.md` | `2026-05-12.md` |

- **`【MOC】` プレフィックス**：フォルダ内でMOCが先頭に表示され視覚的に区別できる。
- **`_メモ` サフィックス**：PDFファイル名と対応し、どのPDFのメモか一目で分かる。
- **日次・週次・月次**は Periodic Notes プラグインがISO形式で自動生成する（ハイフン区切り）。手動ログ類は `YYYYMMDD_`。
- 迷ったら「半年後に開いたとき、タイトルに日付があってほしいか？」で決める。記録なら Yes、概念なら No。

> 規約の正本は `CLAUDE.md` の「ファイル命名規則」。命名違反は `/vault-review` のリントで検出できる。

tags: [paper]
title: "論文タイトル"
authors: "著者名"
year: 2023
venue: "ICASSP 2023"
status: unread

## 一言要約
## 提案手法
## 気づき・メモ
## 疑問・未解決

実験・作業1セッションにつき1ファイル。`10_Projects/プロジェクト名/` に置く。

```markdown
---
date: 2026-05-12
tags: [experiment]
project: プロジェクト研究A
environment: g12（RTX 2080 Ti）

# ノートタイトル
## 実験概要
## 手順
## 結果
## 次のアクション
```

自分の言葉で書いた概念の説明。

## 他の概念との関係

## MOCとは

**MOC（Map of Content）** はノート同士をつなぐハブファイル。  
フォルダの代わりにリンクで構造を表現する。

### MOC階層

```
Home.md（ダッシュボード）
├── 【MOC】10_Projects
│   └── 【MOC】プロジェクト研究A   ← 実験ノート・論文・タスクが集まる
├── 【MOC】20_Areas
│   ├── 【MOC】ソフトウェア開発
│   ├── 【MOC】機械学習・深層学習
│   │   └── 【MOC】B4勉強会
│   └── 【MOC】永続ノート           ← Zettelkasten の入口
└── 【MOC】30_Resources
```

### MOCを作るタイミング

- 新しいプロジェクトを始めたとき
- 特定トピックのノートが3つ以上になったとき
- 関連ノートを一か所から参照したくなったとき

## Zettelkasten / 永続ノート

**Zettelkasten**（ツェッテルカステン）は、概念を原子的なノートとして蓄積し、リンクでつなぐ知識管理手法。

### 論文メモと永続ノートの違い

| | 論文メモ（`30_Resources/`） | 永続ノート（`20_Areas/永続ノート/`） |
|---|---|---|
| 内容 | その論文に書いてあること | 自分がその概念をどう理解しているか |
| 視点 | 論文の著者視点 | 自分の視点 |
| 用途 | 参照・引用 | 思考の蓄積・新アイデアの接続 |

### 永続ノートの作り方

```
1. 論文・勉強会ノートを読む
2. 「1つの概念」を見つける
3. 99_Templates/永続ノート用.md でファイル作成
   → 保存先: 20_Areas/永続ノート/
   → ファイル名: 概念名.md（日付なし）
4. 自分の言葉で書く（コピーペーストしない）
5. 論文メモ・勉強会ノートにリンクを張る
6. 【MOC】永続ノート.md のリンクリストに追加
```

### ルール

- **1ノート1概念** — 複数の概念が入りそうなら分割する
- **必ずリンクを張る** — 孤立した永続ノートは価値が低い
- **定期的に見直す** — 月次レビューで既存の永続ノートを更新・接続する

## 執筆ワークフロー

研究ノートからレポート・スライドへの変換フロー。

```
実験ノート ──┐
             ├→ 実験結果まとめ ──┐
論文メモ ───┘                    ├→ 執筆ドラフト → 最終成果物
             ┌→ 永続ノート ─────┘
勉強会ノート┘
```

### ステップ

1. **収集** — 実験ノート・論文メモを書く（日常的な作業）
2. **統合** — `実験結果まとめ.md` と `【MOC】永続ノート.md` で素材を整理
3. **構成** — `99_Templates/執筆ドラフト用.md` でアウトラインを作る
4. **執筆** — ドラフトを `10_Projects/プロジェクト研究A/下書き/` に保存して本文を書く
5. **提出** — 最終成果物として完成

### 成果物の種別

`type:` フィールドで管理：
- `report` — 研究レポート・中間報告
- `slide` — 発表スライド
- `paper` — 論文

## メンテナンス・コマンド

Claude Code のスラッシュコマンド（`.claude/commands/`）で、Vaultの健康診断とナレッジ整理を半自動化できる。**どちらも検出・提案までで止まり、削除/リネーム/統合は承認後に実行**する。

| コマンド | 頻度 | 何をするか |
|---|---|---|
| `/vault-review` | 毎週〜隔週 | 軽い健康診断：切れリンク・命名規約リント・VAULT_INDEX整合性・Inbox整理・孤立ノート・MOC更新漏れ・週次/月次レビュー作成 |
| `/vault-gc` | 月次〜四半期 | 重い棚卸し：重複/統合候補・分割候補（1ノート1概念）・陳腐化（stale/被リンク0）を検出し整理を提案 |

```
Claude Code を起動して：
  /vault-review   ← まずこれで日常の崩れを直す
  /vault-gc       ← ノートが増えてきたら知識の重複・肥大を棚卸し
```

**使い分け：** `/vault-review` は「散らかり」を直す日常清掃、`/vault-gc` は「知識構造そのもの」を見直す大掃除。`/vault-gc` の前には作業前コミットを取る（コマンドが提案する）。

> 切れリンクが出たら、まず削除ではなく `git` 履歴からの**復元**を検討する（過去に消えたノートが原因のことがある）。

### Periodic Notes の設定

1. コミュニティプラグインから `Periodic Notes` をインストール
2. 設定 → Periodic Notes を開く
3. **Weekly Notes** を有効化：
   - Format: `GGGG-[W]WW_週次レビュー`
   - Template: `99_Templates/週次レビュー用.md`
   - Folder: `00_Inbox/` またはルート
4. **Monthly Notes** を有効化：
   - Format: `YYYY-MM_月次レビュー`
   - Template: `99_Templates/月次レビュー用.md`
   - Folder: `00_Inbox/` またはルート

### Excalidraw の使い方

1. コミュニティプラグインから `Excalidraw` をインストール
2. コマンドパレット → `Excalidraw: Create new drawing` で新規図を作成
3. 論文メモに図を埋め込む：`![[図のファイル名.excalidraw.md]]`
4. 保存先：論文メモと同じフォルダ（例：`30_Resources/音源分離/TF-Locoformer図.excalidraw.md`）

## 人間がやること / Claudeがやること

### 初期設定（一回だけ）

以下はObsidianのUIからしか設定できないため、手動で行う。

| 作業 | 場所 |
|---|---|
| CSSスニペットを有効化 | 設定 → 外観 → CSSスニペット → `vault-custom` をオン |
| obsidian-git の自動コミット間隔を設定 | 設定 → Git → Auto commit interval（推奨: 30分） |
| Periodic Notes の週次・月次を設定 | 設定 → Periodic Notes（上記参照） |
| グラフビューのカラーグループを設定 | グラフビュー → 表示 → グループ |

### 継続的に人間がやること

「考える・体験する・判断する」はAIに代替できない。

| 作業 | 理由 |
|---|---|
| 実験ノートの本文を書く | 実験結果はあなたしか知らない |
| 論文を読んで気づきを書く | 読解と解釈は知的作業 |
| 永続ノートで「自分の解釈」を書く | コピーではなく自分の言葉が価値になる |
| 週次・月次レビューの振り返り内容 | 主観的な評価・目標設定 |
| Inboxノートの振り分け最終判断 | どのプロジェクト・エリアに属するかの判断 |
| 論文メモの `authors` / `venue` を入力 | 論文を確認して正確な情報を入れる |

### Claudeに頼めること（承認ボタンを押すだけ）

「書く・整理する・繋げる」はClaudeが担当できる。

| 作業 | 依頼例 |
|---|---|
| ノートの新規作成・更新 | 「今日の実験ノートを作って」 |
| MOCへのリンク追加 | 「このノートをプロジェクトMOCに追加して」 |
| VAULT_INDEX.md の更新 | 「インデックスを最新化して」 |
| 論文メモの構造整形 | 「このメモをテンプレート形式に整えて」 |
| 永続ノートの作成支援 | 「TF-Locoformerメモから永続ノートを作って」 |
| 関連ノートの検索 | 「音源分離に関連するノートをすべて教えて」 |
| git commit（作業前スナップショット） | 「コミットして」 |
