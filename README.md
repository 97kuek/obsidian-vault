# Obsidian Vault

研究・学習・開発の知識を管理する個人用ナレッジベース。**PARAメソッド**で構成する。

> Claude Code は [`CLAUDE.md`](CLAUDE.md)、Codex は [`AGENTS.md`](AGENTS.md) を入口にする。詳細規約は [`docs/vault-rules.md`](docs/vault-rules.md)。このREADMEは人間向けの概要。

---

## いちばん大事なこと

**書くときは何も考えず `00_Inbox/` に殴り書きするだけ。** 種別・置き場所・テンプレ・タグ・日付は決めない。

整理（分類・整形・移動・命名・タグ・リンク）は、あとで Claude Code や Codex に任せる。
**悩む前に手が動く**のが正しい状態。

```
1. 思いついたこと・授業メモ・論文メモ・実験ログを、ぜんぶ 00_Inbox/ に新規ノートで書く
2. 区切りがついたら AI エージェントに「Inboxを整理して」と依頼 → 1枚ずつ判定・整形・移動を提案 → 承認で確定
3. 週次「vault-review」・月次「vault-gc」で点検
```

## フォルダ構成（PARA）

```
00_Inbox/        ← 未整理ノートの一時置き場（週1で空にする）
10_Projects/     ← 期限・ゴールのあるアクティブな作業（完了したら 40_Archives へ）
  └ 下書き/      ← 執筆ドラフト
20_Areas/        ← 継続的に管理するトピック
  └ 永続ノート/  ← Zettelkasten（1ノート1概念）
30_Resources/    ← 論文の読書メモ（PDF本体は置かない・Zotero等で管理）
40_Archives/     ← 完了・不要なノート（削除せず保管）
99_Templates/    ← AIエージェントが整形時に当てる「型」（人間は選ばない）
```

**分類に迷ったら「このノートはいつ使うか」で決める。**

| 問い | 分類 |
|---|---|
| 今、特定のゴールに向けて動いている | **P**rojects |
| 継続的に維持・学習しているトピック | **A**reas |
| いつか使えそうな参考資料 | **R**esources |
| 終わった・不要になった | **A**rchives |

## ファイル命名規則

**大原則：時間軸のノートは日付を付け、知識のノートは付けない。**

- **知識ノート**（永続・トピック・科目・解説）= `概念名.md`（日付なし）。`[[概念名]]` でリンクでき、育っても名前が陳腐化しない。
- **時間軸ノート**（実験ログ・勉強会・レビュー・日次）= 日付を付ける。

| 種別 | 例 |
|---|---|
| 知識ノート | `決定木.md` / `人工知能.md` |
| MOC | `【MOC】プロジェクト研究A.md` |
| 論文読書メモ | `TF-Locoformer_メモ.md` |
| 実験ログ | `20260511_音源分離実験.md` |
| 勉強会 | `20260423_勉強会02.md` |
| 執筆ドラフト | `20260601_中間報告スライド_draft.md` |
| 週次／月次／日次 | `2026-W23_週次レビュー.md` / `2026-06_月次レビュー.md` / `2026-05-12.md` |

> 迷ったら「半年後に開いたとき、タイトルに日付があってほしいか？」。記録なら Yes、概念なら No。
> 日次・週次・月次は Periodic Notes プラグインがISO形式で自動生成する。詳細は `docs/vault-rules.md`。

## メンテナンス依頼

Claude Code ではスラッシュコマンド、Codex では同じ名前の自然言語依頼として半自動化する。共通手順は `docs/agent-commands/` に置く。**どれも検出・提案までで止まり、削除/移動/統合は承認後に実行**する。

| 依頼 | 頻度 | 何をするか |
|---|---|---|
| `/inbox` または「Inboxを整理して」 | 日々 | Inboxの殴り書きを整理（種別判定→整形→移動・命名→タグ→リンク） |
| `/vault-review` または「Vaultを点検して」 | 週次 | 軽い健康診断（切れリンク・命名・ノート構造・孤立ノート・索引整合性） |
| `/vault-gc` または「Vaultを棚卸しして」 | 月次 | 重い棚卸し（重複統合・分割・陳腐化の検出） |
| `/publish` または「公開候補を確認して」 | 随時 | 公開価値とリスクを点検し、承認済みノートだけをGitHub Pagesへ公開 |

**使い分け：** `/inbox`＝書いたものを正しい場所へ流す日常作業／`/vault-review`＝散らかりの日常清掃／`/vault-gc`＝知識構造の大掃除。

## 永続ノート（Zettelkasten）

概念を原子的なノートとして蓄積し、リンクでつなぐ。論文メモとは別物。

| | 論文メモ（`30_Resources/`） | 永続ノート（`20_Areas/永続ノート/`） |
|---|---|---|
| 内容 | その論文に書いてあること | 自分がその概念をどう理解しているか |
| 用途 | 参照・引用 | 思考の蓄積・新アイデアの接続 |

**ルール：他分野にも通じる横断概念だけを永続ノートにする（特定分野の手法は科目ノートへ）／1ノート1概念／自分の言葉で書く／必ず出典と関連ノートにリンク／`【MOC】永続ノート.md` に追加。**

> 判断軸は「この概念は学んだ分野の外でも使えるか？」。例：`探索と活用のジレンマ` は意思決定全般に通じる→永続ノート。`Q学習` は手法そのもの→`機械学習・深層学習/` の科目ノート。

## レビューサイクル

- **週次（毎週月曜推奨）** — Inboxを空にする・未完了タスク確認・今週の進捗記録
- **月次（月末推奨）** — 今月の成果と来月の目標・論文読書の進捗・永続ノートの見直し

---

## 公開サイト

- GitHub Pagesでは `20_Areas/` を既定で公開する。
- Inbox・Projects・Resources・Archives・運用文書は既定で公開しない。
- 個別例外はfrontmatterの `publish: true`／`publish: false` で指定する。
- QuartzによりWikiリンク、検索、バックリンク、グラフ表示へ変換する。
- 詳細な運用と注意事項は [`docs/publishing.md`](docs/publishing.md) を参照する。
- AIへ公開候補の選定・追加・停止・監査を依頼する場合は `/publish` を使う。

## セットアップ

### プラグイン

| プラグイン | 重要度 | 用途 |
|---|---|---|
| Dataview | 必須 | Home.mdのクエリ（タスク一覧・更新履歴） |
| Templater | 必須 | テンプレートの変数展開 |
| Local REST API | 推奨 | Claude Code 等との MCP 連携（使う場合は起動） |
| obsidian-git | 必須 | Vaultの自動git管理 |
| Periodic Notes | 推奨 | 週次・月次レビューの自動作成 |
| Excalidraw | 推奨 | アーキテクチャ図・概念図 |
| Homepage | 推奨 | 起動時にHome.mdを開く |
| Natural Language Dates | 任意 | `@today` 等の自然言語で日付入力 |
| Calendar | 任意 | 日付ナビゲーション |
| Iconic | 任意 | フォルダ/ファイルにアイコンを付与 |
| Workspaces Plus | 任意 | 作業レイアウトの保存・切替 |

### 初期設定（一回だけ・ObsidianのUIから手動）

| 作業 | 場所 |
|---|---|
| CSSスニペット有効化 | 設定 → 外観 → CSSスニペット → `vault-custom` をオン |
| git自動コミット間隔 | 設定 → Git → Auto commit interval（推奨30分） |
| 週次・月次の自動生成 | 設定 → Periodic Notes（フォーマット・テンプレ・フォルダを指定） |
| グラフのカラーグループ | グラフビュー → 表示 → グループ |

> Periodic Notes の具体的なフォーマット文字列は `docs/vault-rules.md` を参照。

## AI エージェントとの連携

```bash
cd C:\Users\yamashita\Documents\obsidian-vault
claude
# または
codex
```

**AIエージェントに頼めること（承認するだけ）：** ノート作成・編集・リネーム／Vault検索／MOC整理・リンク追加／実験ログ整形／論文・永続ノートの作成補助。

**人間にしかできないこと：** 実験結果を書く／論文の気づきを書く／永続ノートで自分の解釈を書く／レビューの振り返り／Inbox振り分けの最終判断。

依頼は具体的に。良い例「プロジェクト研究AのMOCに今日の実験ノートを追加して」「Inboxを整理して」／悪い例「適当に整理して」（範囲が広すぎる）。

> 複数ファイルの作成・削除・リネームを頼む前に必ず `git add -A && git commit` で作業前スナップショットを取る。

## タグ体系

- **システム：** `MOC` `inbox` `archive`
- **ノート種別：** `paper` `experiment` `lecture` `permanent` `draft` `dataset`
- **トピック：** `機械学習` `音源分離` など
- **モデル：** `transformer` `bert` など（小文字英語）

**ルール：タグは小文字英語または日本語に統一（`AI` → `ai`）。**
