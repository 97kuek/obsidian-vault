# Obsidian Vault ガイド

このファイルはClaude Codeがこのvaultで作業する際の規約・ルールをまとめたもの。

---

## ノート運用の大方針（最重要）

**人間がノートを書くときの判断をゼロにする。** 種別・置き場所・テンプレ・タグ・日付を、書く前に決めさせない。

- ✍️ **人間がやること**：何を書くときも `00_Inbox/` に新規ノートで殴り書きするだけ（件名ごとに1枚、タイトルは雑でよい）。
- 🤖 **Claudeがやること**：`/inbox` で Inbox を読み、**種別判定 → 整形 → 正しいフォルダへ移動・命名 → タグ → リンク/MOC/索引更新**を提案し、承認後に実行する。
- 🗂 **テンプレートは人間が選ばない**。`99_Templates/` は「Claudeが整形時に当てる型」であって、人間が毎回選ぶ選択肢ではない。

Inbox由来でない直接依頼（「TF-Locoformerの永続ノート作って」等）には従来どおり対応してよい。だが「ノートを整理して／これどこ？」と言われたら、まずこの方針＝`/inbox` の流れに乗せる。

**コマンド分担：** `/inbox`（日々：振り分け）／`/vault-review`（週次：点検）／`/vault-gc`（月次：棚卸し）。

## Quick Start（まずここを読む）

**作業開始前に必ず行うこと：**
1. `VAULT_INDEX.md` を読んでファイル構成を把握する（`list_vault_files` で全探索しない）
2. 目的のファイルが分かったら `get_vault_file` で直接読む
3. 作業前に `git status` でVaultの状態を確認する

**よくある作業パターン：**
| 依頼 | 手順 |
|---|---|
| **ノートを整理して／これどこ?** | **`/inbox`**（Inbox整理が基本の入口） |
| 新規ノート作成（直接依頼） | 命名規則・フロントマター規約を確認 → `create_vault_file` |
| ファイル検索 | `search_vault_simple`（キーワード）or `search_vault_smart`（意味） |
| MOC更新 | `get_vault_file` で読む → `create_vault_file` で上書き |
| 実験ノート追加 | `実験ノート用.md` テンプレートに従い `10_Projects/プロジェクト研究A/` に作成 |
| 論文メモ追加 | `論文ノート用.md` テンプレートに従い `30_Resources/音源分離/` に作成し `VAULT_INDEX.md` を更新 |
| 永続ノート作成 | 論文メモから概念を抽出 → `永続ノート用.md` → `20_Areas/永続ノート/` に作成 → `【MOC】永続ノート.md` にリンク追加 |
| 月次レビュー | `月次レビュー用.md` テンプレートで作成 |
| 執筆ドラフト | `執筆ドラフト用.md` → `10_Projects/プロジェクト研究A/下書き/` に置く |

## ファイル命名規則

**大原則：時間軸のノートは日付を付け、知識のノートは付けない。**
- **知識ノート**（永続ノート・トピックノート・授業の科目ノート・リソース解説）＝ 概念名そのまま、日付なし。`[[概念名]]` で素直にリンクでき、内容が育っても名前が陳腐化しない。
- **時間軸ノート**（実験ログ・勉強会の各回・レビュー・日次）＝ 日付を付ける。「いつのものか」が識別子になる。

日付フォーマットは原則 `YYYYMMDD`（ハイフンなし）。日次ノートのみ Obsidian Daily Note 準拠で `YYYY-MM-DD` を例外的に使う。

| 種別 | 命名規則 | 例 | 日付 |
|---|---|---|---|
| 知識ノート（永続・トピック・科目・解説） | `概念名.md` | `決定木.md` / `K-means法.md` / `人工知能.md` / `ESPnet.md` | なし |
| MOC | `【MOC】タイトル.md` | `【MOC】プロジェクト研究A.md` | なし |
| テンプレート | `〜用.md` | `永続ノート用.md` | なし |
| 論文読書メモ | `略称_メモ.md` | `TF-Locoformer_メモ.md` | なし |
| 実験ログ | `YYYYMMDD_タイトル.md` | `20260511_音源分離実験用スクリプトの作成.md` | あり |
| 勉強会の各回 | `YYYYMMDD_勉強会NN.md` | `20260423_勉強会02.md` | あり |
| 執筆ドラフト | `YYYYMMDD_タイトル_draft.md` | `20260601_中間報告スライド_draft.md` | あり |
| 週次／月次レビュー | `YYYY-Www_週次レビュー.md` / `YYYY-MM_月次レビュー.md` | `2026-W23_週次レビュー.md` / `2026-06_月次レビュー.md` | あり |
| 日次ノート | `YYYY-MM-DD.md` | `2026-05-12.md` | あり |

**日次・週次・月次レビューは Periodic Notes プラグインが上記ISO形式で自動生成する**（プラグインがフォーマットを所有するので `YYYYMMDD_` ではなくハイフン区切り）。手動作成するログ類（実験・勉強会・ドラフト）は `YYYYMMDD_` を使う。

**判断に迷ったら：** 「半年後にこのノートを開いたとき、タイトルに日付があってほしいか？」で決める。記録（ログ・その日の出来事）なら Yes、概念の理解なら No。

## ノートの文体

ノートの作成・整形・添削では次を既定とする。

- **だ・である調（常体）** で書く。です・ます調にしない。
- **箇条書き中心**にする。長い散文より、要点を `-` で分けて並べる。階層は字下げで表す。
- 数式・表・コードブロック・引用（問題文など）はそのまま保持する。
- **元の本文の情報は削らない**。整形・添削は構造化・文体統一であって、要約して捨てることではない。

## Zettelkasten フロー（永続ノートの作り方）

```
論文メモ（30_Resources/）
勉強会ノート（20_Areas/）  ──→  永続ノート（20_Areas/永続ノート/）  ──→  【MOC】永続ノート
```

**ルール：**
- 1ノート1概念（複数の概念は分割する）
- 論文の要約ではなく、自分の言葉で書く
- 必ず出典（論文メモ・勉強会ノート）にリンクする
- 関連する永続ノートにリンクする
- 作成したら `【MOC】永続ノート.md` のリンクリストに追加する

| 用途 | 使うツール | 備考 |
|---|---|---|
| ファイル名・キーワードで探す | `search_vault_simple` | 高速 |

## モデル運用方針（Claude Code）

作業の種類でモデルを使い分ける。メイン本体は `/model` で切り替え、サブエージェントは定義ファイル（`.claude/agents/*.md`）の `model:` で固定される。

| 作業 | モデル | 手段 |
|---|---|---|
| 整形・種別判定・MOC設計・添削など質が要る作業（`/inbox` の整形部分など） | Opus | メインセッション |
| vault横断検索・該当ノート特定（要約だけ返す重い探索） | Sonnet | `vault-explorer` サブエージェント |
| 単純な一覧化・機械的処理 | Haiku | spawn時にモデル指定 |

- **重い探索は `vault-explorer`（読み取り専用・Sonnet）に投げる**。メインのコンテキストを節約でき、結論とファイルパスだけ返ってくる。
- スキル（`/inbox` `/vault-review` 等）は**メインセッションで動く**ためモデル固定不可。安く回したい回は実行前に `/model sonnet` へ切り替える運用。

## Git作業フロー

Claudeに複数ファイルの作成・削除・リネームを依頼する前に必ずcommitすること。

```bash
git add -A && git commit -m "作業前スナップショット"
git diff HEAD    # 作業後の差分確認
```

## VAULT_INDEX.mdの更新ルール

ファイルを追加・削除・リネームしたら `VAULT_INDEX.md` を必ず更新する。
これによりClaude が次回の作業でファイル探索コストを最小化できる。

## センシティブ情報

APIキー・パスワード等が必要になった場合は `99_Private/` フォルダを作成し、`.gitignore` と CLAUDE.md に追記する。
