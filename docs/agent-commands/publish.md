# 公開ノート管理

GitHub Pagesへ公開するノートを選定し、`publish: true` の付与・解除・監査を安全に行う。

## 呼び出し方

- `/publish`、公開候補を確認して
- `/publish add <ノート>`、このノートを公開して
- `/publish remove <ノート>`、このノートの公開を停止して
- `/publish audit`、公開範囲を点検して

Codexでスラッシュコマンドが使えない場合も、同名の自然言語依頼としてこの手順を使う。

## 原則

- `20_Areas/` は公開が既定である。
- `00_Inbox/`、`10_Projects/`、`30_Resources/`、`40_Archives/`、運用文書は非公開が既定である。
- 公開候補とリスクを先に提示し、ユーザーの承認後にフラグを変更する。
- Areas内の非公開例外には `publish: false`、非公開フォルダ内の公開例外には `publish: true` を付ける。
- 公開済みノートから非公開ノートへのリンクは、公開時に表示名だけへ変換される。
- 一度公開した秘密情報は、フラグを外すだけでは無効化できない。認証情報の失効・再発行も必要である。
- GitHubへのcommit・pushとPagesデプロイは、ユーザーが公開実行を依頼または承認した場合だけ行う。

## 公開可否の判断

### 公開しやすい

- 自分の言葉で書いた概念ノート・永続ノート
- 一般的な技術解説・学習ノート
- 出典を明記した論文要約
- 第三者の個人情報や組織内部情報を含まないノート

### 要点検

- 授業ノート：課題解答、試験情報、配布資料の転載がないか確認する。
- 論文メモ：本文・図表・Abstractの長い転載がないか確認する。
- クリップ：投稿本文や画像の転載ではなく、自分の要約になっているか確認する。
- 画像付きノート：画像の権利と、位置情報・個人情報の写り込みを確認する。
- AI・開発運用ノート：ローカルパス、リポジトリ名、内部手順、認証情報を確認する。

### 原則非公開

- `00_Inbox/`
- `10_Projects/` の未公開研究・進行中プロジェクト
- `40_Archives/` の個人的記録
- 課題の解答・採点情報
- APIキー、トークン、パスワード、秘密鍵、接続文字列
- 氏名、学籍番号、メールアドレスなどの個人情報
- 第三者の文章・図表・画像を主体とする資料
- `AGENTS.md`、`CLAUDE.md`、`docs/`、`.obsidian/` などの運用ファイル

これらを公開する場合は、ユーザーの明示承認と `publish: true` の両方を必要とする。

## 候補提示

変更前に次の表を提示する。

```markdown
| ノート | 判定 | 公開価値 | リスク・確認事項 | 処理 |
|---|---|---|---|---|
| 探索と活用のジレンマ | 公開継続 | 横断概念として再利用しやすい | 特になし | Areas既定を使用 |
| 人工知能_課題まとめ | 非公開推奨 | 演習資料 | 課題解答を含む | `publish: false` を追加 |
```

## 公開追加の手順

1. `VAULT_INDEX.md`、`docs/vault-rules.md`、`docs/publishing.md` を読む。
2. `git status --short` で既存変更を確認する。
3. 対象ノートと、参照する画像・埋め込み・外部資料を読む。
4. 次を検索する。
   - 認証情報らしい文字列
   - 個人情報・組織内部情報
   - 未公開研究・課題解答
   - 長い転載・公開権限が不明な画像
5. 候補表を提示し、承認を得る。
6. Areas外の承認済みノートには `publish: true` を追加する。Areas内の公開ノートは原則としてフラグを追加しない。
7. 公開抽出を実行する。

```powershell
node tools/export-public-notes.mjs --output .quartz/content
```

8. `.quartz/content/publication-manifest.json` を確認する。
9. Inbox・Projects・Archives・運用文書が混入していないことを確認する。
10. 明らかな認証情報パターンとDataviewブロックの残存を検索する。
11. `git diff --check` と切れリンク確認を行う。
12. 公開実行まで依頼されている場合は、対象差分だけをcommit・pushし、GitHub Actionsと公開URLを確認する。

## 公開停止の手順

1. Areas内では `publish: false` を追加する。Areas外では `publish: true` を削除する。
2. 公開抽出を再実行し、manifestから消えたことを確認する。
3. 公開実行まで依頼されている場合はcommit・pushし、サイトから消えたことを確認する。
4. 秘密情報が含まれていた場合は、認証情報の失効・再発行、Git履歴・デプロイ履歴への対応をユーザーへ報告する。

## 監査の手順

1. Areas内の全ノート、Areas外の `publish: true`、全フォルダの `publish: false` を列挙する。
2. manifestの公開一覧と一致するか確認する。
3. 公開対象外フォルダ、認証情報、個人情報、課題解答、転載物の混入を点検する。
4. 公開ノートからのリンク切れ、未公開化された画像、Dataviewの置換結果を確認する。
5. 「継続公開／要修正／公開停止」の一覧を報告し、変更は承認後に行う。

## 完了報告

```markdown
## 公開ノート更新

- 追加: N件
- 停止: N件
- 公開総数: N件
- 画像: N件
- 秘密情報パターン: 0件
- 非公開フォルダ混入: 0件
- デプロイ: 実施／未実施
- 公開URL: ...
```
