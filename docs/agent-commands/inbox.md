# Inbox 整理

`00_Inbox/` に溜まった殴り書きノートを整理する。人間は「Inboxに書く」だけ、種別・置き場所・テンプレ・タグ・日付は考えない。

## 原則

- 整理案を先に提示し、ユーザーの承認を得てから実行する。
- 人間が書いた本文は消さない。整理は構造化・移動・命名であり、要約して捨てることではない。
- 破壊的操作（移動・上書き・削除）の前に作業前コミットを提案する。
- 迷ったら勝手に決めず、その1枚について短く確認する。

## 事前準備

`VAULT_INDEX.md`、`CLAUDE.md` または `AGENTS.md`、`99_Templates/` の該当テンプレートを読む。

Inbox の対象は `00_Inbox/` 直下の Markdown で、`はじめに.md` は除外する。Codex では通常のファイル一覧、Claude Code では利用可能な MCP/Bash で確認してよい。

Inbox が空なら「Inboxはクリーンです」と報告して終了する。

## 種別判定

| 種別 | 判定の手がかり | 行き先 |
|---|---|---|
| 授業・勉強会メモ | 講義名・日付・聴講メモ的な箇条書き | 既存の科目ノートへ追記、または `20_Areas/該当/` に新規 |
| 論文メモ | 論文タイトル・著者・手法の記録 | `30_Resources/該当トピック/略称_メモ.md` |
| 実験・作業ログ | 実験条件・コマンド・結果・日付 | `10_Projects/該当プロジェクト/YYYYMMDD_タイトル.md` |
| 永続ノート | 自分の理解を1概念で書いている | `20_Areas/永続ノート/概念名.md` |
| プロジェクトノート | 特定プロジェクトの設計・まとめ | `10_Projects/該当/タイトル.md` |
| 雑メモ・思いつき | 断片・ToDo・未整理 | 育てば昇格、不要なら破棄、保留は Inbox 据え置き |
| MOC / レビュー / ドラフト | それぞれの体裁 | 規約どおりの場所 |

## 整理案

実行前に次の表で提示する。

```markdown
## Inbox 整理案（YYYY-MM-DD）

| # | 元ファイル | 判定 | 整形後 | 行き先 | タグ | リンク/MOC | 処理 |
|---|---|---|---|---|---|---|---|
| 1 | 思いついたAttention.md | 永続ノート | アテンション機構.md | 20_Areas/永続ノート/ | permanent | [[Transformer_メモ]]/【MOC】永続ノート | 新規作成して移動 |
```

各行で「新規作成／既存へ追記／据え置き／破棄」を明示する。命名は vault の命名規則に従う。

## 承認後に実行

1. 必要なら作業前コミットを提案する。
2. 本文を該当テンプレートの型に整形する。
3. 規約どおりの名前で正しいフォルダへ移動または新規作成する。
4. タグを付与する。
5. `20_Areas/` へ移すノートは公開可能か確認し、課題解答・個人情報・秘密情報・未公開内容を含む場合は `publish: false` を付ける。
6. 関連リンク、MOC、`VAULT_INDEX.md` を更新する。
7. `tools/vault-review.ps1 -Check Links` または同等の切れリンク確認を行う。

## サマリー

```markdown
## Inbox 整理 完了（YYYY-MM-DD）
- 処理: N件（新規 a / 追記 b / 据え置き c / 破棄 d）
- 移動先の内訳: ...
- 更新したMOC / VAULT_INDEX: ...
- 残ったInbox: M件（理由）
```
