
- Webアプリ開発でいう「○○層」は、**役割ごとにコードを分ける考え方**
- 画面を担当する部分、処理を担当する部分、データを保存する部分を混ぜずに分けるということ
- ホテル予約システムなら、

```text
ユーザーが予約ボタンを押す
↓
予約内容を受け取る
↓
予約できるか判定する
↓
DBに予約を保存する
↓
結果を画面に表示する
```

## Webアプリは大きく3層で考える

- 最初はこの3つ

```text
画面層
↓
処理層
↓
データ層
```

### 1. 画面層

- ユーザーに見える部分
- ホテル予約システムなら、
	- 部屋一覧画面
	- 予約フォーム    
	- ログイン画面
	- 管理画面    
	- 「予約する」ボタン
    
- Next.jsなら、主にこのあたり

```text
app/reservations/page.tsx
components/ReservationForm.tsx
components/RoomCard.tsx
```

- 画面層に複雑なルールを書きすぎるのはよくない

## 処理層

- アプリの中心的な処理を書く部分
- ホテル予約システムなら、
	- 予約を作成する
	- 予約をキャンセルする
	- 空室を確認する
	- 宿泊料金を計算する
	- ユーザーが予約可能か判定する

### 3. データ層

- DBとのやり取りを担当する部分
- ホテル予約システムなら、
	- 部屋を取得する
	- 予約を保存する
	- ユーザーを取得する
	- 空室状況を検索する
- Prismaを使うなら、たとえばこういう処理

```ts
await prisma.reservation.create({
  data: {
    userId,
    roomId,
    checkIn,
    checkOut,
  },
});
```

## もう少し本格的にすると5層くらいになる

- 実務や大きめのアプリでは、よく次のように分ける


```text
画面層
↓
API受付層
↓
ユースケース層
↓
ドメイン層
↓
インフラ・DB操作層
↓
DB
```

### 1. Presentation層：画面層

ユーザーに見える部分です。

Next.jsなら、

```text
page.tsx
layout.tsx
components
```

などです。

例：

```text
予約フォームを表示する
部屋一覧を表示する
エラーメッセージを表示する
```

ここでは基本的に、

> どう表示するか

を担当します。

たとえば、

```text
予約可能かどうか
料金をどう計算するか
DBにどう保存するか
```

は、画面層に書きすぎない方がよいです。

---

# 2. Controller / API層：リクエスト受付層

外部から来たリクエストを受け取る層です。

Next.jsなら、

```text
app/api/reservations/route.ts
```

のようなAPI Route / Route Handlerがこれに近いです。

たとえば、

```http
POST /api/reservations
```

が来たら、

```text
1. リクエストの中身を読む
2. 入力値を軽くチェックする
3. ユースケース層に処理を依頼する
4. 結果をHTTPレスポンスとして返す
```

という役割を持ちます。

重要なのは、API層は「受付係」だということです。

API層に予約の細かいルールを書きすぎると、後で再利用しにくくなります。

たとえば、

```text
予約作成API
管理画面からの予約作成
バッチ処理での予約作成
```

が全部同じ予約ロジックを使いたい場合、API層にロジックがあると使い回しづらくなります。

---

# 3. Application / UseCase層：アプリの操作を表す層

ここはかなり重要です。

UseCaseとは「ユーザーがアプリでやりたいこと」です。

ホテル予約システムなら、

```text
予約を作成する
予約をキャンセルする
空室を検索する
ログインする
会員登録する
```

などです。

たとえば `createReservation` というUseCaseなら、

```text
予約を作成するための一連の流れ
```

を書きます。

例：

```ts
async function createReservation(input) {
  // 1. 入力を確認する
  // 2. 部屋を取得する
  // 3. 空室か確認する
  // 4. 料金を計算する
  // 5. 予約を保存する
  // 6. 結果を返す
}
```

この層は、

> アプリとして何をするか

を担当します。

---

# 4. Domain層：業務ルールの中心

Domain層は、最初は少し難しいですが、かなり大事です。

Domainとは、そのアプリが扱う問題領域のことです。

ホテル予約システムなら、

```text
部屋
予約
宿泊者
料金
空室
キャンセル
支払い
```

などがドメインです。

Domain層には、そのサービス特有のルールを書きます。

たとえば、

```text
チェックアウト日はチェックイン日より後でなければならない
同じ部屋を同じ期間に二重予約できない
定員を超える人数では予約できない
キャンセル期限を過ぎたらキャンセル料が発生する
```

などです。

これは単なるDB操作ではありません。  
ホテル予約という業務のルールです。

たとえば、

```ts
class Reservation {
  constructor(
    public checkIn: Date,
    public checkOut: Date,
    public guestCount: number
  ) {
    if (checkOut <= checkIn) {
      throw new Error("チェックアウト日はチェックイン日より後にしてください");
    }
  }
}
```

このように、予約そのもののルールをDomain層に置きます。

---

# 5. Infrastructure / Persistence層：外部技術に関わる層

Infrastructure層は、DB、外部API、メール送信、決済サービスなど、外部の具体的な技術を扱う層です。

たとえば、

```text
PrismaでDBに保存する
Stripeで決済する
SendGridでメールを送る
S3に画像を保存する
```

などです。

ホテル予約システムなら、

```text
PrismaReservationRepository
MailService
PaymentService
```

のようなものがここに入ります。

Domain層やUseCase層に、PrismaやStripeのコードがベタ書きされすぎると、技術変更に弱くなります。

たとえば将来、

```text
SQLite → PostgreSQL
Prisma → Drizzle
SendGrid → Resend
```

に変えたくなったとき、影響範囲が大きくなります。

---

# Webアプリの処理の流れ

予約作成を例にすると、こうです。

```text
ユーザー
↓
画面層
予約フォームで入力する
↓
API層
POST /api/reservations が呼ばれる
↓
UseCase層
createReservation が実行される
↓
Domain層
予約可能か、日付が正しいか、定員内かを判定する
↓
Infrastructure層
PrismaでDBに保存する
↓
DB
reservations テーブルに保存される
```

図にするとこうです。

```text
[画面]
  ↓
[API / Controller]
  ↓
[UseCase / Application]
  ↓
[Domain]
  ↓
[Repository / Infrastructure]
  ↓
[Database]
```

# 具体例：ホテル予約システムで考える

たとえば「予約する」という機能を作るとします。

## 画面層

```text
ReservationForm.tsx
```

役割：

```text
チェックイン日を入力する
チェックアウト日を入力する
人数を入力する
予約ボタンを押す
```

---

## API層

```text
app/api/reservations/route.ts
```

役割：

```text
POSTリクエストを受け取る
入力値を取り出す
createReservation を呼ぶ
結果をJSONで返す
```

---

## UseCase層

```text
createReservation.ts
```

役割：

```text
予約作成の流れを管理する
```

具体的には、

```text
部屋を取得する
空室を確認する
予約を作る
DB保存を依頼する
```

---

## Domain層

```text
reservation.ts

```

役割：

```text
予約のルールを書く
部屋の定員ルールを書く
日付の妥当性を書く
```

例：

```text
チェックアウト日はチェックイン日より後
人数は部屋の定員以下
同じ期間に同じ部屋は予約不可
```

---

## Infrastructure層

```text
prismaReservationRepository.ts
```

役割：

```text
Prismaを使ってDBに保存する
Prismaを使って予約を検索する
```

---

## DB層

```text
User
Room
Reservation
Payment
```

役割：

```text
データを永続化する
```

つまり、データを消えないように保存するということです。

# 「ビジネスロジック」はどこに書くのか

かなり大事です。

ビジネスロジックとは、サービス特有のルールです。

ホテル予約システムなら、

```text
部屋が空いていないと予約できない
チェックアウト日はチェックイン日より後
予約人数は部屋の定員以下
キャンセル期限を過ぎるとキャンセル料がかかる
```

などです。

これは基本的に、

```text
UseCase層
Domain層
```

に書きます。

特に、

```text
アプリの流れ → UseCase層
業務上のルール → Domain層
```

と考えるとわかりやすいです。

---

# Application層とDomain層の違い

ここは混乱しやすいです。

## Application / UseCase層

「アプリとして何をするか」です。

例：

```text
予約を作成する
予約をキャンセルする
空室を検索する
```

処理の流れを組み立てる層です。

---

## Domain層

「業務ルールとして何が正しいか」です。

例：

```text
チェックアウト日はチェックイン日より後であるべき
定員を超えてはいけない
同じ部屋を同じ期間に二重予約してはいけない
```

つまり、

```text
UseCase層 = 手順
Domain層 = ルール
```

です。

---

# Repository層とは何か

Repositoryは、DB操作を隠すための層です。

たとえばUseCase層が直接Prismaを呼ぶと、こうなります。

```ts
const room = await prisma.room.findUnique(...)
const reservation = await prisma.reservation.create(...)
```

小さいアプリならこれでも動きます。

ただし、アプリが大きくなると、

```text
UseCase層がPrismaに依存しすぎる
テストしにくい
DB変更に弱い
同じDB処理があちこちに散らばる
```

という問題が出ます。

そこでRepositoryを挟みます。

```text
UseCase層
↓
ReservationRepository
↓
Prisma
↓
DB
```

UseCase層から見ると、

```ts
reservationRepository.save(reservation)
reservationRepository.findByRoomAndDate(roomId, checkIn, checkOut)
```

のように呼べます。

UseCase層は「Prismaで保存しているのか」「SQLで保存しているのか」を知らなくてよくなります。

---

# MVCとの関係

「○○層」と一緒によく出てくるのがMVCです。

MVCは、

```text
Model
View
Controller
```

の略です。

ざっくりいうと、

```text
View       = 画面
Controller = リクエストを受け取る
Model      = データやビジネスロジック
```

です。

ただし、MVCのModelは意味が広いです。  
DBのモデルを指すこともあれば、ビジネスロジック全体を指すこともあります。

なので、実務ではMVCだけで考えるより、

```text
画面層
API層
UseCase層
Domain層
Repository層
DB層
```

のように分けて考えた方が整理しやすいです。

---

# Clean Architectureとの関係

Clean Architectureでは、ざっくり次のように考えます。

```text
外側：UI、DB、フレームワーク
内側：UseCase、Domain
```

重要なのは、

> 重要なルールを、フレームワークやDBに依存させすぎない

という考え方です。

つまり、

```text
Next.jsだからこう
Prismaだからこう
```

ではなく、

```text
予約とは何か
予約できる条件は何か
キャンセルできる条件は何か
```

というアプリの本質を中心に置きます。

Clean Architectureでは、依存の向きが大事です。

```text
UI → UseCase → Domain
DB → Repository実装 → UseCase
```

内側のDomainは、なるべくNext.jsやPrismaを知らないようにします。

# Next.jsでのレイヤー構成例

たとえば、Next.js App Routerでホテル予約システムを作るなら、こういう構成が考えられます。

```text
src/
  app/
    reservations/
      page.tsx
    api/
      reservations/
        route.ts

  features/
    reservations/
      components/
        ReservationForm.tsx
      usecases/
        createReservation.ts
        cancelReservation.ts
      domain/
        reservation.ts
        reservationPolicy.ts
      repositories/
        reservationRepository.ts
      infrastructure/
        prismaReservationRepository.ts

  lib/
    prisma.ts
```

それぞれの役割はこうです。

```text
app/                         画面やAPIの入口
components/                  UI部品
usecases/                    アプリの操作
domain/                      予約に関するルール
repositories/                DB操作の抽象化
infrastructure/              Prismaなど具体的な実装
lib/prisma.ts                Prisma Clientの設定
```

# 小規模開発ではどこまで分けるべきか

最初から厳密に分けすぎると、逆に大変です。

特に少人数開発では、最初はこのくらいで十分です。

```text
画面
API
DB操作
```

ただし、次のような処理が出てきたら、UseCaseやServiceに切り出すとよいです。

```text
処理が長くなってきた
同じ処理を複数画面/APIで使う
条件分岐が増えてきた
テストしたくなってきた
DB操作と業務ルールが混ざってきた
```

たとえば最初は、

```text
app/api/reservations/route.ts
```

に処理を書いてもよいです。

でも、長くなったら、

```text
createReservation.ts
```

に切り出します。

# 悪い例：全部が混ざっているコード

たとえばAPIの中に全部書くとします。

```ts
export async function POST(req: Request) {
  const body = await req.json();

  // 入力チェック
  if (body.checkOut <= body.checkIn) {
    return Response.json({ error: "日付が不正です" });
  }

  // DBから部屋取得
  const room = await prisma.room.findUnique(...);

  // 空室確認
  const existing = await prisma.reservation.findMany(...);

  // 料金計算
  const price = ...

  // DB保存
  const reservation = await prisma.reservation.create(...);

  return Response.json(reservation);
}
```

これは動きます。  
ただし、処理が増えると読みにくくなります。

問題は、

```text
API受付
入力チェック
業務ルール
DB操作
料金計算
レスポンス作成
```

が全部混ざっていることです。

---

# 良い分け方

同じ処理を分けると、こうなります。

```text
route.ts
  ↓
createReservation.ts
  ↓
reservation.ts / reservationPolicy.ts
  ↓
prismaReservationRepository.ts
```

API層は薄くします。

```ts
export async function POST(req: Request) {
  const input = await req.json();

  const result = await createReservation(input);

  return Response.json(result);
}
```

UseCase層に流れを書きます。

```ts
export async function createReservation(input) {
  validateReservationDate(input);

  const room = await roomRepository.findById(input.roomId);

  checkCapacity(room, input.guestCount);

  const isAvailable = await reservationRepository.isAvailable(
    input.roomId,
    input.checkIn,
    input.checkOut
  );

  if (!isAvailable) {
    throw new Error("この部屋は指定期間に予約できません");
  }

  return reservationRepository.create(input);
}
```

Domain層にはルールを書きます。

```ts
function validateReservationDate(input) {
  if (input.checkOut <= input.checkIn) {
    throw new Error("チェックアウト日はチェックイン日より後にしてください");
  }
}
```

こうすると、それぞれの役割が見えやすくなります。

# レイヤーを分けるメリット

## 1. 変更に強くなる

たとえばDBを変えたいとき、DB操作層だけを直せばよくなります。

```text
SQLite → PostgreSQL
Prisma → Drizzle
```

のような変更があっても、画面や業務ルールへの影響を減らせます。

---

## 2. テストしやすくなる

画面を動かさなくても、

```text
予約できる条件
キャンセルできる条件
料金計算
```

だけをテストできます。

これはかなり大きいです。

---

## 3. コードを探しやすくなる

バグが起きたときに、

```text
表示がおかしい → 画面層
APIの返し方がおかしい → API層
予約ルールがおかしい → Domain層
保存がおかしい → Repository / DB層
```

と探しやすくなります。

---

## 4. チーム開発しやすくなる

担当範囲を分けやすくなります。

```text
Aさん：画面
Bさん：API / UseCase
Cさん：DB / Prisma
```

のようにできます。

ただし、完全分業しすぎると危険なので、少人数なら全員が全体像を軽く理解している状態が理想です。

---

# レイヤー設計でよくある失敗

## 失敗1：画面にロジックを書きすぎる

Reactコンポーネントの中に、

```text
料金計算
空室判定
DB保存
権限チェック
```

などを書きすぎるパターンです。

画面が読みにくくなります。

---

## 失敗2：API層が太りすぎる

`route.ts` や Controller に全部書くパターンです。

最初は楽ですが、あとで修正しづらくなります。

API層はできるだけ、

```text
受け取る
呼ぶ
返す
```

くらいにしておくとよいです。

---

## 失敗3：DB設計と業務ルールを混同する

DBはデータを保存する場所です。

一方で、業務ルールはDomain層やUseCase層にあります。

たとえば、

```text
Reservationテーブルがある
```

ことと、

```text
予約とは何か
予約できる条件は何か
```

は別です。

DB設計は重要ですが、DBだけ見てもアプリのルールは完全には表現できません。

---

## 失敗4：最初から分けすぎる

小さいアプリなのに、

```text
Controller
Service
UseCase
Domain
Repository
Factory
DTO
Mapper
Presenter
```

などを全部作ると、逆に開発が重くなります。

最初は、

```text
画面
API
処理
DB
```

くらいから始めて、必要になったら分ければ大丈夫です。

---

# まとめ

Webアプリのレイヤー構造は、ざっくり言うとこうです。

```text
画面層
ユーザーに見せる・入力を受け取る

API層
リクエストを受け取って、処理に渡す

UseCase層
アプリとしての操作の流れを書く

Domain層
業務ルールを書く

Infrastructure / Repository層
DBや外部サービスとのやり取りを書く

DB層
データを保存する
```

一番大事なのは、

> 画面、処理、業務ルール、DB操作を混ぜすぎないこと

です。

最初はこの理解で十分です。

```text
画面は表示に集中する
APIは受付に集中する
UseCaseは処理の流れに集中する
Domainはルールに集中する
RepositoryはDB操作に集中する
```

ホテル予約システムで言えば、

```text
予約フォーム → 画面層
POST /api/reservations → API層
createReservation → UseCase層
予約可能条件・日付ルール → Domain層
Prismaで保存 → Infrastructure層
Reservationテーブル → DB層
```

という感じです。
