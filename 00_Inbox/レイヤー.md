
- Webアプリ開発でいう「○○層」は、**役割ごとにコードを分ける考え方**
- 画面を担当する部分、処理を担当する部分、データを保存する部分を混ぜずに分けるということ
- ホテル予約システムなら、

```text
ユーザーが予約ボタンを押す
↓
予約内容を受け取る
↓
予約できるか判定する
↓
DBに予約を保存する
↓
結果を画面に表示する
```

## Webアプリは大きく3層で考える

- 最初はこの3つ

```text
画面層
↓
処理層
↓
データ層
```

### 1. 画面層

- ユーザーに見える部分
- ホテル予約システムなら、
	- 部屋一覧画面
	- 予約フォーム    
	- ログイン画面
	- 管理画面    
	- 「予約する」ボタン
    
- Next.jsなら、主にこのあたり

```text
app/reservations/page.tsx
components/ReservationForm.tsx
components/RoomCard.tsx
```

- 画面層に複雑なルールを書きすぎるのはよくない

## 処理層

- アプリの中心的な処理を書く部分
- ホテル予約システムなら、
	- 予約を作成する
	- 予約をキャンセルする
	- 空室を確認する
	- 宿泊料金を計算する
	- ユーザーが予約可能か判定する

### 3. データ層

- DBとのやり取りを担当する部分
- ホテル予約システムなら、
	- 部屋を取得する
	- 予約を保存する
	- ユーザーを取得する
	- 空室状況を検索する
- Prismaを使うなら、たとえばこういう処理

```ts
await prisma.reservation.create({
  data: {
    userId,
    roomId,
    checkIn,
    checkOut,
  },
});
```

## もう少し本格的にすると5層くらいになる

- 実務や大きめのアプリでは、よく次のように分ける


```text
画面層
↓
API受付層
↓
ユースケース層
↓
ドメイン層
↓
インフラ・DB操作層
↓
DB
```

### 1. Presentation層：画面層

- ユーザーに見える部分

```text
予約フォームを表示する
部屋一覧を表示する
エラーメッセージを表示する
```

- ここでは基本的に、どう表示するかを担当する

### 2. Controller / API層：リクエスト受付層

- 外部から来たリクエストを受け取る層

```http
POST /api/reservations
```

が来たら、

```text
1. リクエストの中身を読む
2. 入力値を軽くチェックする
3. ユースケース層に処理を依頼する
4. 結果をHTTPレスポンスとして返す
```

という役割を持つ

- 重要なのは、API層は「受付係」だということ
- API層に予約の細かいルールを書きすぎると、後で再利用しにくくなる

### 3. Application / UseCase層：アプリの操作を表す層

- UseCaseとは「ユーザーがアプリでやりたいこと」

たとえば `createReservation` というUseCaseなら、

```text
予約を作成するための一連の流れ
```

を書く

例：

```ts
async function createReservation(input) {
  // 1. 入力を確認する
  // 2. 部屋を取得する
  // 3. 空室か確認する
  // 4. 料金を計算する
  // 5. 予約を保存する
  // 6. 結果を返す
}
```

- アプリとして何をするかを担当する

### 4. Domain層：業務ルールの中心

- Domainとは、そのアプリが扱う問題領域のこと
- ホテル予約システムなら、

```text
部屋
予約
宿泊者
料金
空室
キャンセル
支払い
```

などがドメイン

- Domain層には、そのサービス特有のルールを書く

```text
チェックアウト日はチェックイン日より後でなければならない
同じ部屋を同じ期間に二重予約できない
定員を超える人数では予約できない
キャンセル期限を過ぎたらキャンセル料が発生する
```

### 5. Infrastructure / Persistence層：外部技術に関わる層

- Infrastructure層は、DB、外部API、メール送信、決済サービスなど、外部の具体的な技術を扱う層

```text
PrismaでDBに保存する
Stripeで決済する
SendGridでメールを送る
S3に画像を保存する
```

など

ホテル予約システムなら、

```text
PrismaReservationRepository
MailService
PaymentService
```

のようなものがここに入る

- Domain層やUseCase層に、PrismaやStripeのコードがベタ書きされすぎると、技術変更に弱くなる

## Webアプリの処理の流れ

- 予約作成の例は次の通り

```text
ユーザー
↓
画面層
予約フォームで入力する
↓
API層
POST /api/reservations が呼ばれる
↓
UseCase層
createReservation が実行される
↓
Domain層
予約可能か、日付が正しいか、定員内かを判定する
↓
Infrastructure層
PrismaでDBに保存する
↓
DB
reservations テーブルに保存される
```

## レイヤーを分けるメリット

- 変更に強くなる
- テストしやすくなる
- コードを探しやすくなる
- チーム開発しやすくなる
