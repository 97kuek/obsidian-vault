
## 文字コード技術

### 文字コードとは

#### 符号化(エンコード)という考え方

- 文字コードとは、**コンピュータで文字を処理するために、文字に番号を割り当てる規則・規格**のこと
- 文字コードが表すのは「あ」の形そのもの（図形情報）ではなく、**文字を抽象化した情報**だという点
- 「あ」という文字の"概念"に番号を振っているのであって、フォントの見た目とは別物
- 符号化(coding)とは、文字を {0, 1} のビット列で表現すること
#### 文字集合 vs 符号化文字集合

|用語|英語|意味|
|---|---|---|
|**文字集合**|character set|システムで扱う文字の"集まり"。番号はまだ振っていない|
|**符号化文字集合**|coded character set|文字集合を定義し、**各文字にコード（ビット組合せ）を割り当てた**もの|

- 文字集合の例：ラテン大文字・小文字（52文字）、数字（10文字）、常用漢字（2,136文字）
- 符号化文字集合 ＝ いわゆる「文字コード」。各文字がどのコードポイントに割り当たっているかを視覚的に示した表を**コードチャート**と呼ぶ
- 原則として **1つの文字は一意に符号化される**
- これらは ISO（国際標準化機構）やJIS（日本産業規格）などの標準規格として策定される
- 標準には2種類の立場がある
	- **デジュールスタンダード**：ISO/IEC 10646
	- **デファクトスタンダード**：Unicode
#### 符号化文字集合 vs 文字符号化方式 

|用語|意味|
|---|---|
|**符号化文字集合**|コードを割り当てた文字の集合そのもの|
|**文字符号化方式**（文字エンコーディング）|**複数の符号化文字集合を組み合わせて使う**ための方法|

- 最初は符号化文字集合を作るだけで良かった
- 後から「複数の文字集合を組み合わせたい」「互換性を保ちたい」という要求が出てきて、**符号化方式**が考案された
#### ASCII (ISO/IEC 646)

- 最初に標準化された符号化文字集合
- **ASCII** = American Standard Code for Information Interchange
- ANSI（米国規格協会）が策定。1960年検討開始、**1963年初版**
- **7ビット**の符号化文字集合（128文字）
- これを国際標準化したのが **ISO/IEC 646**。各国版が定義された

**符号位置(code position)** ── コード表での位置を「列/行」で表します。例えば小文字 `m` は6列目13行目で「6/13」と表記し、ビット列としては `1101101`（7ビット）になります。

#### なぜ7ビットだったのか

- 当時は**データリンク層のコストが高く**、同じ情報をなるべく少ないビットで送りたかった
- 1バイト=8ビットのうち、**最上位ビット（8ビット目）をパリティチェック**に使っていた（誤り検出用。第3章の内容とつながります）
- バイナリデータを送るときは uuencode などでいったん ASCII(7ビット) に変換して送信した

### 制御文字と図形文字

- **図形文字** (graphic character)：画面に表示できる普通の文字
- **制御文字** (control character)：改行など、出力を制御する特殊なコード

|制御文字|略号|ASCII|説明|
|---|---|---|---|
|ベル|BEL|0x07|ビープ音|
|水平タブ|HT|0x09|`\t`|
|改行|LF|0x0A|行送り|
|復帰|CR|0x0D|行頭（左端）へ移動|
|エスケープ|ESC|0x1B|後続文字を特別解釈（エスケープシーケンス）|

**改行コードはOSで違う**

- **LF (0x0A)** … UNIX系（Linux, macOS, FreeBSD）
- **CR+LF (0x0D 0x0A)** … MS-DOS, Windows
- **CR (0x0D)** … 旧Mac OS（クラシック）

#### 内部コードと外部コード

- **内部コード**：システム内部の処理に使うコード
- **外部コード**：システムの入出力に使うコード
- 両者は一致している必要はない
- 現在のOSは、内部コードに **Unicode**、符号化方式に **UTF-8** を採用するケースが大半

#### 文字コードの変遷

|段階|時期|内容|
|---|---|---|
|(1) ASCIIと各国対応|1960年代～|アメリカで誕生 → 他国へ広がる|
|(2) 複数バイト文字集合|1970年代～|アジアの文字（漢字）への対応|
|(3) 国際符号化文字集合|1990年代～|世界で統一された文字へ|

- **1バイト文字集合**：ISO/IEC 646・ASCII(1960–)、JIS X 0201(1969–、日本版ISO 646)、ISO/IEC 8859・Latin-1(1987–)
- **複数バイト文字集合**：JIS X 0208(1978–、ISO/IEC 2022準拠)、ISO/IEC 10646・Unicode(1989–)

### 代表的な符号化文字集合と符号化方式

#### (1) ASCII → 各国言語への対応

- ASCIIは英語向けなので、各国が自国の文字を入れた**各国版ISO/IEC 646**を作成

```
              ASCII / ISO-IEC 646
        ┌──────────┼──────────┐
   JIS X 0201    DIN 66003    BS 4730
   (日本:カナ)  (独:アクセント) (英:ポンド記号)
```

- 各国がASCIIの一部を自国文字に置き換えたため、後に互換性問題を起こす
- さらに8ビットに拡張した **ISO/IEC 8859シリーズ**（Latin-1, Latin-2…）が作られ、ラテン用字系191文字で主要な欧米言語に対応
- **JIS X 0201（旧 JIS C 6220）** ── ここが日本の出発点で、後の「文字化け」の遠因になる
- 7ビット部分は ASCII をベースにしつつ、**バックスラッシュ `\` を円マーク `¥`、チルダ `~` をオーバーラインに置換** → この扱いが後で問題化
- 8ビット目が1の領域（ASCIIでは表現できない範囲）に**半角カタカナ**を配置
- 8ビットJIS（和英混在向け）と7ビットJIS（独立向け）がある

#### (2) 複数バイト文字集合の導入と ISO/IEC 2022

- 漢字は文字数が膨大で、1バイト（256通り）では到底足らない
- そこで**2バイト文字**の提案
- 各国版を切り替えて使いたい / 異なる言語の文字を同時表示した
- これらを解決する**符号化方式**が **ISO/IEC 2022** 
- 2バイト（または3バイト）を組み合わせて文字を表現
- インターネット利用を想定し、当初は**7ビット符号**として使われた
- **エスケープシーケンス（ESC = 0x1B）で文字集合を切り替えられる**のが核心

**GL領域とGR領域**（ISO/IEC 2022の重要概念）:

1バイト(8ビット)の範囲を4つの区画に分けます。L=左(Left)、R=右(Right)、C=制御(Control)、G=図形(Graphic)。

```
ビット範囲        区画    用途
0x00–0x1F        CL     制御文字（左）
0x20–0x7F        GL     図形文字（左）← 7ビットで使える範囲
0x80–0x9F        CR     制御文字（右）
0xA0–0xFF        GR     図形文字（右）← 8ビット目が1
```

様々な文字集合を GL または GR に「**呼び出す**(invoke)」ことができます。7ビットしか使わない環境なら GL だけに呼び出せばOKです。

**94×94 の2バイト文字集合**:

- 各バイトとも `0x21`～`0x7E`（7ビットのGL範囲、94通り）を使う
- 2バイトの組合せで **94×94 = 8,836文字** を表現
- （仕様上は94×94×94＝約83万文字の3バイトも可能だが実例はない）

#### JIS X 0208 と区点番号

- 日本語の漢字・ひらがな・カタカナ等を収めた**2バイト符号化文字集合**（ISO/IEC 2022準拠）
- 1978年制定、最新は2012年版
- 漢字辞典に載っている「JISコード」がこれ
- **94行×94列**を **区点番号** で表す：**区＝行番号、点＝列番号**

**区点 → コードの変換ルール**（GLに呼び出した場合）:

```
バイト値 = 0x20 + 区点番号
```

例「森」を考えると（後の例で使います）、JIS X 0208 で **31区25点**:

```
1バイト目 = 0x20 + 31 = 0x3F
2バイト目 = 0x20 + 25 = 0x39
→ 0x3F39  ビット列 00111111 00111001
```

例「愛」= 0x3026 = 16区6点：`0x20+16=0x30`、`0x20+6=0x26` → GL呼び出し時のビット列は `00110000 00100110`。

#### ISO-2022-JP ← メールで日本語を送る方式

- **ASCII + JIS X 0201ラテン文字 + JIS X 0208** を混在できる**7ビットの符号化方式**
- **JIS X 0201（半角カナ）はサポートしない**
- インターネット（電子メール）で日本語を扱うために考案（**RFC 1468**）
- もとは JUNETコード。慶應大学・**村井純**先生らが開発

**エスケープシーケンスで切り替える**のがポイント:

```
ESC $ B   (1b 24 42)  → JIS X 0208 へ切り替え
ESC ( B   (1b 28 42)  → ASCII へ戻す
```

例「森」を ISO-2022-JP で符号化:

```
1b 24 42   ESC $ B   （JIS X 0208 へ）
3f 39      森（31区25点）
1b 28 42   ESC ( B   （ASCII へ戻す）
0a         改行(LF)
```

#### (3) 国際符号化文字集合 ── Unicode と ISO/IEC 10646

- ISO/IEC 2022 のように文字集合を切り替えるのではなく、**1つの大きな表に全文字を入れてしまおう**という発想
- コンピュータの処理能力が向上して現実的に
- 多言語処理や検索処理ではこの方が効率的
- 同じ時期に2つの国際規格が並行して進んだ

||ISO/IEC 10646|Unicode|
|---|---|---|
|立場|国際標準（公的）|民間コンソーシアム（Xerox, MS, Apple, IBM, Sun, HP, ジャストシステム等）|
|初期のコード長|**4バイト**|**2バイト**|
|ISO/IEC 2022互換|あり|なし|

最終的に **ISO/IEC 10646 を Unicode に合わせる**形で妥協し、両者が同一のコード表になるよう改訂されました。このとき日本・中国・韓国・台湾の漢字を統合（**CJK統合漢字**）。これにより JIS X 0208 の資産がそのまま使えない、微妙に字形が異なる、といった問題が生じました。

**現在の Unicode**:

- 講義時点で最新は Unicode 15.0.0（2022年9月）、**約149,186文字**、161の用字・記号・絵文字
- 当初の「2バイト(16ビット)で何とかする」方針は文字数増加で**破綻** → **符号化方式で対応**することに

**BMP（基本多言語面）**: 16ビット（U+0000～U+FFFF）で表現できる範囲を **Basic Multilingual Plane** と呼びます。よく使う文字はここに収まっており、それを超える文字は拡張領域に置かれます。

#### UTF-16 と UTF-8 ── Unicodeの符号化方式

- **Unicode** … コードポイントを定めたもの（文字集合）
- **UTF-16 / UTF-8** … それを実際のバイト列に変換する**方式**

|方式|特徴|
|---|---|
|**UTF-16**|16ビットを符号単位とし、拡張部分の文字はサロゲートで表現。**Java の内部文字表現**はこれ|
|**UTF-8**|**可変長**。コードポイントの位置によってバイト長が変わる|

**UTF-8 の符号化規則**（これは確実に暗記）:

|バイト数|コードポイント範囲|ビットパターン|
|---|---|---|
|1バイト|U+0000 – U+007F|`0xxxxxxx`|
|2バイト|U+0080 – U+07FF|`110xxxxx 10xxxxxx`|
|3バイト|U+0800 – U+FFFF|`1110xxxx 10xxxxxx 10xxxxxx`|
|4バイト|U+10000 – U+10FFFF|`11110xxx 10xxxxxx 10xxxxxx 10xxxxxx`|

ポイント:

- **1バイト文字（U+0000～007F）は ASCII とまったく同じ** → ASCIIとの後方互換性がある
- バイト長によって**先頭バイトのビットパターンが決まる**（2バイトなら先頭`110`、3バイトなら`1110`…）。**2バイト目以降は必ず `10`** で始まる → 途中のバイトと先頭バイトを区別できる

**「森」を UTF-8 でエンコードする**（スライド45の手順を図解）:

```
① コードポイント        U+68EE
② 範囲判定              U+0800–U+FFFF → 3バイト
③ 2進数化              0x68EE = 0110 1000 1110 1110   (16ビット)
④ 3バイト枠に分割       1110[xxxx] 10[xxxxxx] 10[xxxxxx]  ← x は 4+6+6=16個
⑤ 上の16ビットを 4/6/6 に割り当て:
        0110 | 100011 | 101110
⑥ 枠にはめる:
        1110 0110   10 100011   10 101110
        └─┬──┘     └──┬───┘    └──┬───┘
        11100110    10100011    10101110
⑦ 16進数化            E6   A3   AE
```

→ 「森」の UTF-8 は **E6 A3 AE**（検算OK）。

#### 絵文字

- もとは**日本の携帯電話**で使われていたもの
- Unicodeに採用され、国際的に "emoji" として普及
- **BMP外の符号位置**を使う → **UTF-8では4バイト文字**になる
- 絵文字＋修飾コードの組合せ（肌の色など）で見た目を変える仕組みもある
