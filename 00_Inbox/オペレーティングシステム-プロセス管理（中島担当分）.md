
## プロセスとは何か、どう作られるか

### そもそもプロセスとは

- **プログラム**と**プロセス**は別物
	- **プログラム** … ディスク上に置かれた「命令の集まり（静的なファイル）」
	- **プロセス** … そのプログラムが実際にメモリに読み込まれ、CPUで動いている「実行中の姿」

1. OSは複数のプロセスを**同時に**実行できる
2. 複数のプロセスが**同じプログラム**を実行することもできる
3. 各プロセスは**プロセスデータ構造**で管理される

### プロセスの実行環境

- **CPUは物理的に1つ**
- レジスタ、**プログラムカウンタ**、**スタックポインタ**も1組しかない
- つまり、本当の意味で同時に動けるプロセスは1つだけ
- それなのに「複数のプロセスが同時に動いているように見える」のは、OSが高速に切り替えているから

### プログラムが実行される仕組み

- 高級言語（C, C++）はコンパイラで機械語に変換され、リンケージでライブラリが結合され、最終的にこの形で実行される

### システムコール

- **「OSはどこにあるのか？」**
- 答えは「OSもプログラムであり、メモリの中にある」
- ではアプリはどうやってOSの機能を使うのか? 
- それが**システムコール**
- ただし、ただの関数呼び出しとは異なる
- OSの機能は**カーネルモード**という特権状態でしか実行できないから

1. アプリは**システムコール番号**をレジスタに入れ、**トラップ命令**を実行する
2. トラップ命令によって**ユーザモードからカーネルモードに切り替わる**
3. カーネル内の決まった番地（ディスパッチコード）にジャンプする
4. システムコール番号を使って、適切な処理（open, close, fork など）に振り分ける

### プロセスの生成 

#### fork（自分のコピーを作る）

- 現在のプロセスの**完全なコピー**を作る
- コード・データ・スタックの各エリアをコピーする
- レジスタ（MMUやページテーブル含む）をコピーする
- 戻り値が親と子で違う
- **親プロセスには子のPID、子プロセスには0**が返る
- これによって「いま自分は親なのか子なのか」をプログラムが判定できる

#### exec（中身を入れ替える）

- プロセスの**イメージ（中身）を別のプログラムに置き換える**
- 指定ファイルからコード・データをコピーし直す
- 新しいスタックを割り当ててSPを初期化、ヘッダ情報からPCを初期化する

```
shell() {
    プロンプトの出力
    コマンドの入力
    if (fork()) {        // 親プロセス（戻り値=子のPID、つまり真）
        wait();          // 子の終了を待つ
    } else {             // 子プロセス（戻り値=0、つまり偽）
        exec(コマンドのファイル);  // lsなどに変身して実行
    }
}
```

- つまり「**forkで自分を複製し、複製した子の方をexecで目的のプログラムに変身させる**」というのがUNIXのプロセス生成の基本パターン
- `ls > aaa`（出力をファイルにリダイレクト）のような処理も、forkの後・exec前に`open`や`dup2`でファイルディスクリプタを差し替えることで実現される

### プロセスデータ構造

- 各プロセスは以下を持つ
	- **コード・データ・スタック**（メモリ上の実体）
	- **仮想レジスタ**（そのプロセス専用のレジスタの値）
	- **PID**（プロセス識別番号、例：1010, 829）
- これらすべてを**プロセスデータ構造**が一元管理する
- 「プロセスに関するすべての情報を管理する」のがこのデータ構造の役割

## どう切り替え、どう順番を決めるか

### 仮想レジスタと実レジスタ

- 各プロセスは**仮想レジスタ**を持つ（自分専用だと思っている）
- でもCPUが持つ**実レジスタ**は1組だけ
- **OSが「実レジスタ ⇔ 仮想レジスタ」の対応づけを管理する**

- つまり**プロセスAを実行する**とは、実際には**Aの仮想レジスタの値を実レジスタにコピーする**ことを意味する
	- **時間的プロセス管理** … 1つのCPUを時間で分け合う。プロセスごとの仮想CPUが、物理CPUに順番にマッピングされる
	- **空間的プロセス管理** … 各プロセスの仮想メモリを、物理メモリに割り当てる（メモリ管理の話につながる）

### コンテキストスイッチ

- **コンテキストスイッチ**とは「いま実行中のプロセスを別のプロセスに切り替えること」

#### いつ起きるか

- 実行中のプロセスが**ブロック**したとき（入力待ちなどで進めなくなる）
- 実行中のプロセスが**一定時間以上**実行されたとき（時間切れ）

#### 重要な原則

- コンテキストスイッチは**常にカーネル内で発生**
- ブロックはシステムコール中に起こり、時間切れは割り込みによって起こるから

#### Wait/Wakeupの仕組み

1. プロセスAを実行 → AがWaitを実行
2. **実レジスタの内容をAの仮想レジスタにセーブ**
3. **Bの仮想レジスタを実レジスタにコピー** → プロセスBを実行
4. BがAをWakeup（再び実行可能に）し、自分はWait
5. 実レジスタをBの仮想レジスタにセーブして、Aに戻る

- **WaitとWakeup**という2つの操作が切り替えのカギ
- Waitは現在のプロセスを止めて他に実行を移し（→コンテキストスイッチ）
- WakeupはWaitで止まっているプロセスを再び実行可能にする（→スケジューリング）。

**コンテキストスイッチのオーバヘッド（コスト）** も指摘されています。切り替えはタダではありません。

- **パイプライン**が乱れる（CPUの命令先読みが無駄になる）
- **キャッシュメモリ**が効きにくくなる（プロセスが多いと、各プロセスのよく使うデータをキャッシュに保てない）

だから「切り替えすぎ」も問題になる、という現実的な視点です。

## 3. スケジューリング ― 次にどのプロセスを動かすか

切り替えるとして、「**次にどのプロセスを選ぶか**」を決めるのがスケジューリングです。

### プロセスの状態

プロセスは3つの状態を行き来します。

- **実行中（Running）** … いまCPUを使っている
- **実行可能（Ready）** … いつでも動けるが、CPUの順番待ち
- **ブロック（Blocked）** … 入力待ちなどで動けない

遷移は「実行中 →（Wait）→ ブロック →（Wakeup）→ 実行可能 → 実行中」というサイクルです。実行可能状態のプロセスは**Readyキュー**につながれて順番を待ちます。

### Preemptive か Non-Preemptive か

スケジューリングの大きな分かれ目です。

- **Preemptive（プリエンプティブ）** … あるプロセスが実行可能になったら、**すぐに**切り替える。応答性が良い
- **Non-Preemptive（ノンプリエンプティブ）** … いま動いているプロセスが**自分でブロックするか終了するまで**切り替えない。シンプルだが、長く居座られると他が待たされる

### スケジューリングアルゴリズム

スライドでは段階的に紹介されています。

**Non-Preemptive系:**

- **FIFO（First In First Out）** … 到着順。来た順に処理
- **SJF（Shortest Job First）** … 実行時間が短いものを優先

**タイムシェアリング系:**

- すべてのプロセスを**均等に**少しずつ実行する（時分割）。t1, t2, t3…と短い時間ずつ順番に回す

**優先度順:**

- キュー内を優先度順に並べ、高いものから実行
- **UNIXスケジューラ**の巧妙な点：
    - **I/O待ちから復帰したプロセスは優先度が上がる**（対話的な処理をキビキビ動かすため）
    - **長く実行したプロセスは優先度が下がる**（CPUを独占させないため）

**リアルタイム系:**

- プロセスが**周期的**に実行され、それぞれ**デッドライン（締め切り）**を持つ
- **EDF（Earliest Deadline First）** … 締め切りが最も早いものを最優先
- **RM（Rate Monotonic）** … 周期が最も短いものを最優先

## 4. スレッド ― プロセスの中のさらに細かい実行の流れ

最後のテーマがスレッドです。

**スレッドとは何か:** プロセスの中で、**コードとデータは共有しつつ、PC・SP・スタックだけを別々に持つ**実行の流れです。スライドの図がわかりやすく、

- 同じコード、同じデータを共有
- スレッド1：PC=func1, SP=スタックA
- スレッド2：PC=func2, SP=スタックB

というように、1つのプロセスの中に複数の「実行中の地点」を持てます。

**なぜスレッドが必要か:**

- **スループット向上** … マルチスレッドサーバ（多数のリクエストを並行処理）
- **構造化** … 異なるI/O処理に別々のスレッドを割り当てて整理する
- **高速化** … 計算を複数スレッドに分割して並列に処理

**スレッドの操作（プリミティブ）:**

- `thread_create(関数のエントリ, スタックのポインタ)` … 生成
- `thread_exit()` … 終了
- `thread_self()` … 自分のID取得

**実現法:** 同じコード・データを共有し、スタックだけ別（スタックA, スタックB）。各スレッドのPC・SPは**スレッドデータ構造**で管理されます。ユーザ空間とカーネル空間のどちらで管理するかで実装が分かれます。

---

## 2つのスライドの関係を一言で

- **プロセス管理-1** = 「プロセスとは何か」「どう作るか」（fork/exec、システムコール、プロセスデータ構造）= **静的な仕組み・生成**
- **プロセス管理-2** = 「1つのCPUをどう分け合うか」（コンテキストスイッチ、スケジューリング、スレッド）= **動的な実行・切り替え**

根底に一貫して流れているのは、**「物理的には1組しかないCPU（実レジスタ）を、仮想化によって複数プロセスで分け合う」**という発想です。この1点を押さえると、両スライドが1本の線でつながって見えてきます。
