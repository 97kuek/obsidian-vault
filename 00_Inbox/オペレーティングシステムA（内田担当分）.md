
## UNIXの基礎

### OSとは何か

- オペレーティングシステム（OS）とは計算機を制御するプログラムの集合
- ユーザと計算機のあいだのインターフェースとして機能するものと定義される
- UNIX、Windows、Mac OSなどがその例
- ハードウェアの上にOSがあり、その上にシステムインターフェースやライブラリ、さらにアプリケーションが乗る、という層構造でコンピュータは動いている

### UNIXのアーキテクチャとシステムコール

- UNIXは中心に**kernel**があり、その周りをアプリケーションが取り囲む構造
- kernelとのインターフェースとなるソフトウェアのレイヤーを**システムコール**と呼び、プログラムを実行するためのインターフェースを提供する特別なアプリケーションを**シェル**と呼ぶ
- さらにその上の層として、**システムコールの上位にある汎用的な関数群を標準ライブラリ**と呼び、アプリケーションはシステムコールと標準ライブラリの両方を利用することができる
- システムコールも標準ライブラリも、C言語における通常の関数として呼び出せる

### ユーザモードとカーネルモード

- CPUにはユーザモードとカーネルモードがあり、**通常プロセスはユーザモードで動作**するが、**システムコールを呼び出すとカーネルモードに切り替わり**、処理が終わるとシステムコールから戻ってユーザモードに復帰
- 一般のアプリケーション（ユーザモード）には、ハードウェアを直接いじったり他のプロセスの領域を壊したりする権限を与えず、本当に強い権限が必要な操作だけをカーネルモードのカーネルに任せる、という権限の分離を行っている

### プロセスとプロセスID

- 各プロセスは唯一の識別子であるPID（Process ID）を持ち、これは非負の値で与えられる
- プロセスが終了するとその**PIDは再利用される**
- また各プロセスは、自分を生成したプロセスを示す親プロセスID（PPID: Parent PID）を持つ
- これらは `ps –l` コマンドで確認でき、PIDカラムとPPIDカラムに表示される

### UNIXファイルシステム

- ファイルシステムの基礎としてiノードの仕組みが扱われる
- UNIXにおけるディレクトリとは、**iノードとファイルネームの対を並べたファイルのこと**
- iノード番号を指定することでiノードにアクセスし、ファイルサイズ、ユーザ／グループID、タイムスタンプ、パーミッション、リンク数、ディスク上の位置といったファイルの情報を取得できる
- 「ファイル名」と「実体の情報（iノード）」が分離されている、というのがUNIXファイルシステムの特徴

## プロセスの生成と終了、プログラムの実行と割り込み

### fork() 

- プロセス生成の中心はfork()
- fork()を呼び出したプロセス（親プロセス）は新たなプロセス（子プロセス）を生成
- fork()は親プロセスのアドレス空間を子プロセスに複製し、両プロセスはfork文の直後の命令から実行を継続する
- 返り値については、**子プロセスには0**を返し、**親プロセスには子プロセスのPID**を返し、**エラー時には-1**を返す
- fork()が実行されると子プロセスは親プロセスのアドレス空間の複製を受け取るため、fork前に親のPIDを代入しておいた変数は子プロセスにもそのままコピーされ、子側ではその値がgetpid()の返り値（=子自身のPID）と一致しない
- つまり変数の中身までまるごとコピーされるが、コピー後はそれぞれ独立したプロセスとして動くということ

### fork()のユースケースとexec()

- fork()の重要な使い道として、1つのプロセスで異なるプログラムを実行したい場合がある
- これはシェルでよく用いられる
- この場合、子プロセスはfork()の直後にexec()を実行する
- シェルはまずfork()で自分の子プロセスを作り、その子プロセスがexec()で目的のプログラム（たとえば`ls`）に「中身を入れ替わる」。
- 親であるシェル自身はそのまま残り、子の終了を待つ
- `fork→exec→exi`t、親は`wait`で待つ、という流れが標準パターン

#### wait() / waitpid() 

- **親が子を放置するとどうなるか**
- 子プロセスで`sleep`が呼ばれていると、子が終了する前に親が先に終了してしまい、シェルのプロンプトが先に表示されてしまうことがある
- そのため、親プロセスが子プロセスを生成するアプリケーションでは、子がいつどのように終了したかを監視することが有用
- その監視のための仕組みが`wait`
- 親プロセスは`wait()`あるいは`waitpid()`を実行することで、子プロセスが終了するまで自身の実行をブロックできる
- `wait()`は子プロセスのうちの1つが終了するまで親を休止させ、`waitpid()`は指定した特定の子プロセスが終了するまで親を休止させる
- 「どれか1つの子を待つ」か「特定の子を待つ」かの違い

### 割り込み

#### シグナルとは何か

- プロセスに対して「何が起きたのか、何をしてほしいのか」を通知する仕組み
- 各シグナルには名前がついていて、整数値、もしくは`<signal.h>`で定義されたシグナル名`SIGxxxx`で指定される
#### シグナルの種類とデフォルトの動作

- シグナルには種類ごとに「何も設定しなければこうなる」というデフォルトの動作が決まっている

|シグナル|説明|デフォルトの動作|
|---|---|---|
|SIGFPE|浮動小数点例外（例：0による除算）が起きた|コアダンプ終了|
|SIGSEGV|セグメンテーション違反（不正なメモリアクセス）が起きた|コアダンプ終了|
|SIGINT|キーボードからの割り込み（Ctrl-C）が届いた|終了|
|SIGTSTP|キーボードからの一時中断（Ctrl-Z）が届いた|一時中断|
|SIGKILL|プロセスの強制終了|終了|

#### signal() 

- このデフォルト動作は、signal()関数を使って変更できる
- signal()はシグナル受信時の動作を設定する関数で、指定した番号のシグナルに、シグナルハンドラーを設定する

```c
void (*signal(int sig, void (*handler)(int)))(int);
```

ハンドラーには3種類の指定方法があります。

- **SIG_IGN** を指定すると、そのシグナルは無視される
- **SIG_DFL** を指定すると、デフォルトの動作を行う（元に戻す）
- **ユーザが指定した関数** を渡すと、そのシグナルを受け取ったときにその関数が呼び出される

そして動作の流れが重要です。シグナル`sig`を受け取ると関数`handler`が呼び出され、その`handler`の引数には届いたシグナルの番号が渡されます。`handler`が呼び出されるときは、それまで実行していたプログラムの実行はいったん一時中断され、`handler`から戻ると実行を再開します。

ここが「割り込み」という言葉の核心です。**通常の処理を走らせている最中に、シグナルが届くと処理が一時的に脇にそれてハンドラが実行され、ハンドラが終わると中断したところから元の処理に戻る**——これがまさに割り込みの動作です。

#### kill() 

ここまでは「受け取る側」の話でしたが、シグナルを「送る側」のための関数がkill()です。

```c
int kill(pid_t pid, int sig);
```

kill()は、`pid`で指定されたPIDを持つプロセスに、`sig`で指定されたシグナルを送出します。

名前は「kill（殺す）」ですが、必ずしも終了させるとは限らない点に注意してください。送るシグナルを選べるので、たとえばSIGKILLを送れば強制終了になりますが、SIGTSTPを送れば一時中断、ユーザ定義のハンドラが設定されていればその処理が走る、というように、kill()は「任意のシグナルを任意のプロセスに送り込む汎用的な手段」です。第1回で学んだPIDが、ここで「どのプロセスに送るか」の宛先として効いてくるわけです。

## ファイル入出力、リダイレクション、パイプ

### Everything Is a File

- UNIXの設計思想の象徴が**すべてはファイル**という考え方

### ファイル入出力の基本システムコール

- Linuxでのファイル操作は少数のシステムコールに集約されている
- Linuxにおけるファイル入出力は、open（ファイルを開く）、read（ファイルから読む）、write（ファイルに書く）、close（ファイルを閉じる）、lseek（ファイルの位置決め）という5つの関数（システムコール）で実行される
- 基本は「開く→読む／書く→閉じる」という流れ

### ファイル記述子

- これらの操作の鍵が**ファイル記述子**（file descriptor）という整数
- 既存のファイルを開いたり新たなファイルを生成すると、カーネルはプロセスに対してファイル記述子を返す
- ファイルの読み書きを行いたい場合は、`open`から返されたファイル記述子を`read`や`write`の引数として指定しなければならない

### システムコールと標準入出力ライブラリ

- これら低水準のシステムコールの上に、もう一段使いやすい層がある
- 標準入出力ライブラリは、低水準入出力であるシステムコールへの様々な用途のインターフェースを提供する
- 低水準入出力におけるファイル記述子は、標準入出力ではストリームと呼ばれ、`FILE *`という構造体へのポインタとして実装される
- プログラムが実行されると、`stdin`（標準入力、ファイル記述子0に対応）、`stdout`（標準出力、ファイル記述子1）、`stderr`（標準エラー、ファイル記述子2）の3つのファイルストリームが自動的に開かれる
- この「0が入力、1が出力、2がエラー」という対応関係が、次のリダイレクションを理解する前提

### リダイレクション

リダイレクションは、この標準入出力の接続先を付け替える操作です。./a.out < infile で標準入力を切り替え（入力リダイレクト）、./a.out > outfile で標準出力を切り替え（出力リダイレクト）、両方同時に書けば入力と出力の両方を切り替えられます。また ./a.out 2> outfile では標準エラー出力を切り替えられます。

その内部の手順も明確に示されます。リダイレクトの手順は、(1) 読み込み用／書き込み用にファイルを開く、(2) stdin／stdoutのファイル記述子にその開いたファイルを関連付ける、(3) その開いたファイルを閉じる、の3ステップです。

この「関連付け」を実現するのがdup()／dup2()システムコールです。資料の図では、たとえば`close(1)`で標準出力の記述子1を空けてから、開いたファイルをそこに複製する、あるいは`dup2(fd, 1)`で記述子1を一気に目的のファイルに付け替える、という操作が描かれています。`> outfile`の正体は、まさにこの記述子の付け替えなのです。

### パイプ

- パイプはプロセス間で通信（データ交換）するための仕組み
- あるコマンドの標準出力を別のコマンドの標準入力に接続する
- 書式は `cmd1 | cmd2` 
- `ls | wc -l` では、シェルが`ls`と`wc`を実行して2つのプロセスをそれぞれ生成し、これらの2つのプロセスがパイプで接続される
- 重要なのは、**これらのプロセスはパイプの存在に気づいておらず**、ただ**標準入出力のファイル記述子に対して読み書きをするだけ**、という点
- `ls`は「自分は画面に出力しているつもり」で`stdout`に書き、`wc`は「自分はキーボードから読んでいるつもり」で`stdin`から読む
- その間をパイプが橋渡ししている
