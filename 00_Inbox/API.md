## APIとは何か

- **Application Programming Interface** の略  
- 日本語にすると、**アプリケーション同士がやり取りするための窓口・約束ごと** 

### 身近な例

- たとえば、天気アプリを開くと現在の天気が表示される
- 多くの場合、天気情報を提供しているサーバーに対して、「東京の今日の天気を教えて」とリクエストする
- そのとき以下のようなデータを返す

```json
{
  "city": "Tokyo",
  "weather": "sunny",
  "temperature": 28
}
```

- このやり取りを可能にしているのが **Web API** 

## WebアプリにおけるAPIの位置づけ

- Webアプリでは、よく次のような構成になる

```text
ユーザー
  ↓
フロントエンド
  ↓ API
バックエンド
  ↓
データベース
```

## APIでやり取りするもの
### 1. エンドポイント

- エンドポイントは、APIのURLです。  
「どの機能を呼び出すか」を表します。

例：

```text
/api/users
/api/reservations
/api/hotels/1/rooms
```

ホテル予約システムなら、

```text
/api/reservations
```

は「予約に関するAPI」だと考えられます。

---

### 2. HTTPメソッド

HTTPメソッドは、**何をしたいのか** を表します。

代表的なものは次の4つです。

|メソッド|意味|例|
|---|---|---|
|GET|取得する|予約一覧を取得する|
|POST|新しく作る|予約を作成する|
|PUT / PATCH|更新する|予約内容を変更する|
|DELETE|削除する|予約をキャンセルする|

たとえば、

```text
GET /api/reservations
```

なら「予約一覧を取得する」。

```text
POST /api/reservations
```

なら「新しい予約を作る」。

という意味になります。

---

## リクエストとレスポンス

APIの基本は、

```text
リクエスト → レスポンス
```

です。

### リクエスト

リクエストは、クライアントからサーバーへのお願いです。

例：

```http
POST /api/reservations
Content-Type: application/json
```

```json
{
  "userId": 1,
  "roomId": 5,
  "checkIn": "2026-07-01",
  "checkOut": "2026-07-03"
}
```

これは、

> ユーザー1が、部屋5を、7月1日から7月3日まで予約したい

という意味です。

---

### レスポンス

レスポンスは、サーバーからクライアントへの返事です。

例：

```json
{
  "reservationId": 100,
  "status": "confirmed",
  "message": "予約が完了しました"
}
```

つまり、

> 予約ID 100として予約が完了しました

という返事です。

---

## ステータスコード

APIのレスポンスには、処理結果を表す **ステータスコード** が付きます。

代表的なものは次の通りです。

|ステータスコード|意味|
|---|---|
|200|成功|
|201|作成成功|
|400|リクエストが間違っている|
|401|ログインが必要|
|403|権限がない|
|404|データが見つからない|
|500|サーバー側のエラー|

たとえば、存在しない予約IDを取得しようとしたら、

```http
404 Not Found
```

が返ることがあります。

ログインしていないのに予約しようとしたら、

```http
401 Unauthorized
```

が返ることがあります。

---

## JSONとは何か

APIでは、データのやり取りに **JSON** がよく使われます。

JSONは、プログラム同士で扱いやすいデータ形式です。

例：

```json
{
  "id": 1,
  "name": "田中太郎",
  "email": "tanaka@example.com"
}
```

JavaScriptやTypeScriptとの相性がよく、Webアプリでは非常によく使われます。

---

## REST APIとは何か

APIの設計方法の一つに **REST API** があります。

RESTでは、データを「リソース」として考えます。

たとえばホテル予約システムなら、

```text
/users
/hotels
/rooms
/reservations
```

のように、ユーザー、ホテル、部屋、予約をリソースとして扱います。

そして、HTTPメソッドで操作を表します。

例：

```text
GET    /api/reservations        予約一覧を取得
GET    /api/reservations/1      予約ID 1を取得
POST   /api/reservations        新しい予約を作成
PATCH  /api/reservations/1      予約ID 1を変更
DELETE /api/reservations/1      予約ID 1を削除
```

REST APIでは、URLはなるべく「名詞」にして、操作はHTTPメソッドで表すのが基本です。

なので、

```text
/api/getReservations
/api/createReservation
```

よりも、

```text
GET  /api/reservations
POST /api/reservations
```

の方がRESTらしい設計です。

---

## APIはなぜ必要なのか

APIがあると、フロントエンドとバックエンドを分けて開発できます。

たとえば、フロントエンド担当は、

> `/api/reservations` にPOSTすれば予約できる

ということだけ知っていれば、内部でデータベースをどう操作しているかまでは知らなくても開発できます。

一方、バックエンド担当は、

> この形式のリクエストが来たら予約を作成して、この形式で返す

というルールに従って実装できます。

つまりAPIは、**チーム開発における契約書** のような役割もあります。

## Next.jsで考えるAPI

あなたが以前考えていたような Next.js App Router 構成だと、APIは例えば次のように作れます。

```text
app/api/reservations/route.ts
```

このファイルに、

```ts
export async function GET() {
  // 予約一覧を取得する処理
}

export async function POST(request: Request) {
  // 新しい予約を作成する処理
}
```

のように書きます。

フロントエンド側では、

```ts
await fetch("/api/reservations")
```

と書くことでAPIを呼び出せます。

予約を作るなら、

```ts
await fetch("/api/reservations", {
  method: "POST",
  headers: {
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    userId: 1,
    roomId: 5,
    checkIn: "2026-07-01",
    checkOut: "2026-07-03"
  })
});
```

のようになります。

## API設計で大事なこと

APIを作るときは、ただ動けばよいわけではありません。  
次のようなことを考える必要があります。

### 1. 何のデータを扱うか

ホテル予約システムなら、

```text
ユーザー
ホテル
部屋
予約
支払い
```

などが考えられます。

### 2. どんな操作が必要か

予約に対しては、

```text
予約を作る
予約を確認する
予約を変更する
予約をキャンセルする
```

などが必要です。

### 3. 誰が操作できるか

たとえば、

一般ユーザーは自分の予約だけ見られる  
管理者は全ユーザーの予約を見られる

というように、権限の設計も必要です。

### 4. エラー時に何を返すか

満室なのに予約しようとした場合、

```json
{
  "error": "この部屋は指定された期間に予約できません"
}
```

のように、フロントエンドが扱いやすい形で返す必要があります。

## APIとデータベースの違い

ここは混同しやすいです。

APIは、外部から処理を呼び出すための入口です。  
データベースは、データを保存する場所です。

```text
フロントエンド
  ↓
API
  ↓
バックエンドの処理
  ↓
データベース
```

なので、フロントエンドが直接データベースを触るのではなく、基本的にはAPIを通してデータを取得・変更します。

理由は、セキュリティや権限管理、入力チェック、ビジネスルールをサーバー側で管理するためです。

## APIの具体例：予約作成

ホテル予約システムで「予約する」機能を考えると、APIは次のようになります。

### エンドポイント

```text
POST /api/reservations
```

### リクエスト

```json
{
  "userId": 1,
  "roomId": 10,
  "checkIn": "2026-08-01",
  "checkOut": "2026-08-03"
}
```

### サーバー側でやること

```text
1. userId が存在するか確認
2. roomId が存在するか確認
3. checkIn と checkOut が正しい日付か確認
4. その期間に部屋が空いているか確認
5. 問題なければ予約を作成
6. 作成結果を返す
```

### レスポンス

```json
{
  "id": 123,
  "status": "confirmed",
  "roomId": 10,
  "checkIn": "2026-08-01",
  "checkOut": "2026-08-03"
}
```

このように、APIは単なるデータの受け渡しだけでなく、**業務ルールを実行する入口** でもあります。

## APIでよく出てくる用語

|用語|意味|
|---|---|
|クライアント|APIを呼び出す側。ブラウザやスマホアプリなど|
|サーバー|APIを提供する側|
|エンドポイント|APIのURL|
|リクエスト|クライアントからサーバーへの要求|
|レスポンス|サーバーからクライアントへの返事|
|JSON|APIでよく使うデータ形式|
|HTTPメソッド|GET、POST、PATCH、DELETEなど|
|ステータスコード|成功・失敗を表す番号|
|認証|ログインしているか確認すること|
|認可|その操作をしてよい権限があるか確認すること|

## APIを理解するための一番大事なイメージ

APIは、

> 「このURLに、この方法で、このデータを送ったら、こういう結果が返ってくる」

という **約束** です。

たとえば、

```text
POST /api/reservations
```

に、

```json
{
  "roomId": 10,
  "checkIn": "2026-08-01",
  "checkOut": "2026-08-03"
}
```

を送ったら、

```json
{
  "status": "confirmed"
}
```

が返ってくる。

この約束をフロントエンドとバックエンドの間で決めておくことで、Webアプリを作りやすくなります。

---

## まとめ

APIとは、**プログラム同士がやり取りするための窓口** です。  
Webアプリでは、フロントエンドがAPIを呼び出し、バックエンドが処理を行い、必要に応じてデータベースを操作します。

APIを理解するときは、次の5つを押さえるとよいです。

```text
1. どのURLを呼ぶか        → エンドポイント
2. 何をしたいか          → HTTPメソッド
3. 何のデータを送るか    → リクエスト
4. 何が返ってくるか      → レスポンス
5. 成功か失敗か          → ステータスコード
```

最初は難しく感じますが、要するにAPIは **画面とサーバーをつなぐ決められた会話の形式** です。  
Next.jsやPrismaでWebアプリを作るなら、かなり重要な基礎になります。
