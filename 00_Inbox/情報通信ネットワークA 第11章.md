
## P2P技術

### P2Pとは何か

- **P2P (Peer-to-peer)** は、ネットワークに参加する各ピアが相互にタスクやリソースを融通し合うモデル
- Peerとは「同等の・対等な仲間」という意味

||クライアント・サーバ型|P2P型|
|---|---|---|
|リソースの所在|サーバに集中|各ピアに分散|
|役割|提供する側/受ける側が固定|各ノードが提供も受信も両方行う|
|性質|中央集権的|非中央集権的|

### オーバーレイネットワーク

- P2Pはアプリケーション層において独自のオーバーレイ・ネットワーク(overlay network)を構成
- ピア同士で TCP セッションを確立することにより論理的なネットワークを形成
- 下の層に実際の物理的なIPネットワーク（ルータの網）があり、その**上に重ねて** ピア同士のTCP接続による論理的な網を作る、という二層構造

```
[論理層] PC ━━━━━━━━━ PC      ← P2Pが見る世界（隣に見える）
              ╲       ╱           (TCPセッションで直結)
[物理層]  router─router─router   ← 実際のパケットの通り道
```

- 論理層では「隣のピア」でも、物理層では何台ものルータを経由している、というズレがポイント
- なおIP層も実はオーバーレイネットワークの一種

### P2Pの3つのアーキテクチャ

- インデックス情報（誰がどのIPを持ち、どのリソースがどこにあるかの索引）を**誰が管理するか**で3つに分類される

#### ピュアP2P

ノードのIPアドレスやリソースがどのノードにあるかという情報＝インデックス情報を各ノードが分散して保持します。完全に分散しているため、最初にネットワークに参加するにはブートストラップノード（最初の足がかりとなる既知のノード）の情報が必要です。例: Gnutella, Freenet, Winny, Bitcoin。

#### 中央集中型P2P

インデックス情報はすべて中央サーバが管理します。「誰がどのファイルを持っているか」だけを中央サーバが教え、実際のファイル転送はピア同士で直接行う、という折衷型です。例: BitTorrent, Napster, WinMX。

#### ハイブリッド型P2P

インデックス情報はスーパーノード（ピアの中で安定的なリソースを持つもの）が管理します。一部の高性能ピアが索引役を担う中間形態です。例: Skype, Kazaa。

```
ピュア:    全ノードが索引を分散保持   （完全分散）
ハイブリッド: 一部のスーパーノードが索引  （半分散）
中央集中型: 中央サーバが索引を一元管理   （集中寄り）
```

### P2Pの特徴と応用

- 企業が運用するCDNや大規模ウェブサービスと異なり、基本的に一般ユーザの草の根的な活動によって構成される
- 多くの端末の資源を共有することで高効率でスケーラビリティに優れるコンテンツ分配システムを形成可能である
- 一方で匿名性が高いため、悪用されることがある、あるいは監視・検閲を回避する手段にもなる
- 応用例: ファイル共有(BitTorrent, Winny等)、コンテンツ配信、ストリーミング(Spotify等)、暗号通貨システム(Bitcoin)など。

### ビットコインとブロックチェーン

#### ビットコインの全体像

- Satoshi Nakamoto によって2008年に発表された技術
- P2Pシステム＋ブロックチェーン技術によって暗号通貨の取引を可能とする
- 論文タイトルが "Bitcoin: A **Peer-to-Peer** Electronic Cash System" であることからも、P2Pが基盤だと分かる

#### ブロックチェーン = 分散台帳技術

- 一言で言えば「分散台帳技術」の一種
- 台帳」は取引の履歴を記録した情報
- 銀行のように中央集権的に台帳を管理するのではなく、非中央集権的に分散管理し、台帳は参加ノード全体で共有する

#### ブロックチェーンの構造

```
ブロックB(i-1) ── ブロックB(i) ── ブロックB(i+1)
  [TX,TX,TX]      [TX,TX,TX]       [TX,TX,TX]
  H(B(i-2))       H(B(i-1))        H(B(i))
   ↑前ブロックのハッシュ値を各ヘッダが含む
```

- 暗号学的ハッシュ関数を用いることで改ざんを検知でき、各ブロックにはトランザクション(TX)が格納されている
- 各ブロックのヘッダには固有のハッシュ値が含まれ、それぞれのハッシュ値は前のブロックのデータにnonceと呼ばれる値を追加して算出
- 各ブロックが前のブロックのハッシュ値を含むため、1つのブロックを改ざんすると、それ以降の全ブロックのハッシュ値が連鎖的に変わってしまい、すぐ検知できる

#### ブロック内部 — マークル木

- ブロック内のTXはハッシュツリー（マークル木）で管理
- TX1〜TX4を2つずつハッシュでまとめ上げ、最終的に1つの**Merkle root**に
- TXのいずれかを改ざんするとMerkle rootのハッシュ値が変わり、ブロックのハッシュ値も変わって、すでに計算してあったnonceが無効になる → 改ざん検出という構造

#### Proof of Work (PoW) とマイニング

- Nonceを計算するには膨大な計算リソースが必要
- マイナーには成功報酬がBTCで支払われ、計算した証拠 = PoWとなる

## セキュリティ技術

### 暗号の表現と歴史

```
平文 ──[暗号化:暗号鍵]──→ 暗号文 ──[復号:復号鍵]──→ 平文
```

- 近代〜現代の暗号には、バーナム暗号（ワンタイムパッド）、共通鍵暗号方式（DES/3DES, AES, RC4）、公開鍵暗号方式（RSA, 楕円曲線暗号）

#### バーナム暗号の計算例

- 平文の各文字をASCIIコードの二進表現にし、乱数の暗号化鍵とXORを取って暗号文を生成
- 例えば文字 'w' (ASCII 119 = `01110111`) を鍵 `01101011` とXORすると `00011100` になる
- XORは同じ鍵をもう一度かければ元に戻る性質があり、これが復号に使える

### 共通鍵暗号 vs 公開鍵暗号

#### 共通鍵暗号方式

- 暗号化と復号に同じ鍵を利用し、対称鍵暗号方式・秘密鍵暗号方式とも呼ばれる
- 不特定多数が存在する環境では鍵の配布が課題です（**鍵配送問題**）
- 相手に安全に鍵を渡す手段がそもそも難しい、というジレンマ

```
[共通鍵]  平文 →暗号化(鍵K)→ 暗号文 →復号(同じ鍵K)→ 平文
          問題: 鍵Kをどうやって安全に相手へ渡す？
```

#### 公開鍵暗号方式

- 鍵を暗号鍵（公開鍵）と復号鍵（秘密鍵）にペアとして分離する非対称暗号方式
- 公開鍵で暗号化した平文は秘密鍵でないと復号できず、鍵配送問題を解決した点が革新的

```
[公開鍵]  平文 →暗号化(公開鍵)→ 暗号文 →復号(秘密鍵)→ 平文
          公開鍵はばらまいてOK、秘密鍵は本人だけが保持
          → 事前の安全な鍵交換が不要！
```

代表的方式: RSA暗号（素因数分解）、ElGamal暗号（離散対数問題）、楕円曲線暗号（bitcoinはこれを採用）

### RSA暗号

- Rivest, Shamir, Adelmanの3人の頭文字に由来
- 1977年に発明（2002年チューリング賞）
- 巨大な数の素因数分解が困難であることを利用し、その困難さがRSAの安全性です
- ただし処理速度は共通鍵暗号と比較すると遅いため、ハイブリッド暗号化方式が使われます

### 電子署名

- 公開鍵暗号を「逆向き」に使う応用
#### 署名の考え方

- 秘密鍵（署名鍵）で署名でき、署名できるのは秘密鍵を持っている人だけで、検証は誰でもできる（公開鍵＝検証鍵）
- つまり秘密鍵で処理 → 公開鍵で検証、という暗号化とは逆の流れです。これにより「本人しか作れない署名」が実現

#### 仕組み

- メッセージに対するハッシュ値を秘密鍵（署名鍵）で署名することで、本人性の検証と改ざん検知を実現
- 公開鍵暗号方式と組み合わせて使う。ハッシュ関数はSHA-256、公開鍵暗号方式はRSA, DSA, ECDSAなどを使用

```
【署名側】 文書 →ハッシュ関数→ ハッシュ値 →秘密鍵で署名→ 電子署名
【検証側】 文書 →ハッシュ関数→ ハッシュ値A
          電子署名 →公開鍵で復号→ ハッシュ値B
          A == B なら本物 & 改ざんなし
```

- 電子署名はハッシュ値が衝突しないことが前提
- N=128ビットのハッシュ関数に100億個の入力を入れても衝突確率は約1000京に1回という極めて低い確率なので、安全に成り立つ

### 暗号学的ハッシュ関数

- ブロックチェーンでも電子署名でも登場する基礎技術
- 任意の長さのビット列を固定長のビット列に圧縮する関数
- 出力値はハッシュ値・メッセージダイジェスト・フィンガープリントなどと呼ばれる
- 主要な性質は、入力が少しでも変わると出力が全く変わること、出力から入力を逆算できないこと、そして衝突（異なる入力が同じ出力になること）が事実上起きないこと
- この「逆算困難・衝突困難」という性質が、PoWのマイニングにも改ざん検知にも電子署名にも利用されている

### 応用 — TLS（ハイブリッド暗号の実例）

- 共通鍵と公開鍵を組み合わせた実用例
- TLS（サーバ認証モード）では、ハンドシェークでバージョン・暗号アルゴリズム・nonceを交換し、サーバが公開鍵証明書を送付、クライアントが検証、クライアントがサーバの公開鍵でpre-master secretを暗号化して送付し、双方がmaster secretを生成、以降はそれを共通鍵暗号に適用して暗号化通信を行うという流れ
- 公開鍵暗号は安全だが遅い、共通鍵暗号は速いが鍵配送が問題
- そこで「鍵の交換だけ公開鍵暗号で安全に行い、本番のデータ通信は速い共通鍵暗号で行う」という良いとこ取りをする
- これがRSAが遅いという弱点の解決策
