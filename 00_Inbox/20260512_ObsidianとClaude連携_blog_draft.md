# ObsidianをAIアシスタントと完全連携させた話——Claude Codeで「考える環境」を作る

---

## はじめに

ノートを取るのは好きだけど、うまく活用できていない——そんな悩みを持つ人は多いのではないでしょうか。

私もそのひとりでした。Obsidian（オブシディアン）というノートアプリを使っていたものの、「とりあえずメモして終わり」になりがちで、後から見返しても「あのノートどこだっけ？」状態が続いていました。

そこで今回、**Claude Code**（Anthropicが開発したAIアシスタントのCLIツール）とObsidianを本格的に連携させて、ノートを「知識の資産」として機能させる仕組みを作りました。

この記事では、その構築プロセスと工夫したポイントを紹介します。研究者・学生・エンジニアなど、知識を扱う仕事をしている方の参考になれば幸いです。

---

## 使ったツール

- **Obsidian**：ローカルファイルベースのノートアプリ。マークダウンで書けて、リンク機能が強力。
- **Claude Code**：Anthropicが開発したAI（Claude）をターミナルから使えるCLIツール。コードだけでなく、ファイルの作成・編集・検索なども任せられる。
- **obsidian-mcp-tools**：ObsidianとClaude CodeをつなぐMCPプラグイン。これにより、ClaudeがObsidianのVault（ノートの保管庫）を直接読み書きできるようになる。
- **Dataview**：Obsidianプラグイン。タグや日付などのメタ情報をもとにノートをデータベースのように検索・表示できる。
- **Templater**：Obsidianプラグイン。テンプレートに変数を埋め込んで、新規ノートを素早く作れる。

---

## まず「構造」を整える：PARAメソッド

Claudeに頼む前に、まずVault（ノートの保管庫）の整理が必要でした。

採用したのは**PARAメソッド**です。すべてのノートを4つのカテゴリに分類する考え方で、情報管理のフレームワークとしてエンジニア・研究者の間で広く使われています。

| カテゴリ | 英語 | 使い方 |
|---|---|---|
| プロジェクト | Projects | 締め切り・ゴールがある作業 |
| エリア | Areas | 継続的に学んでいるトピック |
| リソース | Resources | いつか使えそうな参考資料・論文 |
| アーカイブ | Archives | 終わったもの・不要になったもの |

```
obsidian-vault/
├── 00_Inbox/       ← とりあえずここに書く（後で分類）
├── 10_Projects/    ← 今進めているプロジェクト
├── 20_Areas/       ← 継続的な学習・関心事
├── 30_Resources/   ← 論文・参考資料
├── 40_Archives/    ← 完了・不要になったもの
└── 99_Templates/   ← テンプレート置き場
```

「分類に迷ったらInboxに入れて後で整理する」という運用にすることで、「どこに保存するか悩む時間」をゼロにできます。

## Claude CodeをObsidianに接続する

ここが今回の肝です。

通常、Claude Codeはターミナル上で動くAIアシスタントで、主にプログラミングの補助に使われます。しかし**MCP（Model Context Protocol）**という仕組みを使うと、外部ツールと連携できます。

`obsidian-mcp-tools`というプラグインを導入すると、Claude CodeがObsidianのVaultを直接操作できるようになります。具体的には：

- ノートを読む（`get_vault_file`）
- ノートを作成・上書きする（`create_vault_file`）
- キーワードで検索する（`search_vault_simple`）
- 意味・文脈で関連ノートを探す（`search_vault_smart`）

これにより、「このプロジェクトのMOCを更新して」「新しい論文メモを作って」といった依頼をClaude Codeにそのまま投げられるようになります。

### CLAUDE.mdで「作業規約」を作る

Claude Codeはプロジェクトのルートに`CLAUDE.md`というファイルを置くと、会話を始めるたびに自動的に読み込みます。ここにVaultのルールを書いておくと、毎回説明しなくても一貫した作業をしてもらえます。

私が書いた主な内容：

```markdown
## ファイル命名規則
- 通常ノート：YYYYMMDD_タイトル.md
- MOCファイル：【MOC】タイトル.md
- 論文読書メモ：論文タイトル略称_メモ.md

## 作業前に必ずやること
1. VAULT_INDEX.md を読んでファイル構成を把握する
2. git status でVaultの状態を確認する
```

これで、Claudeは毎回「どこに何を保存するか」「どういう形式で書くか」を把握した状態で作業を始めてくれます。

---

## トークンを節約する：VAULT_INDEX.md

AI（Claude）は使えばコストがかかります。そして「Vaultのファイル一覧を見せて」という操作を繰り返すと、そのたびにトークン（AIが処理する文字数の単位）を消費します。

この問題を解決するために作ったのが**VAULT_INDEX.md**です。

Vaultのすべてのファイルとその内容を1行で説明した「索引ファイル」で、Claudeはまずこれを読むことで全体像を把握します。

```markdown
# Vault Index

## 10_Projects/
| ファイル | 内容 |
|---|---|
| プロジェクト研究A/【MOC】プロジェクト研究A.md | 研究A全体（目的・タスク）Phase3進行中 |
| プロジェクト研究A/実験結果まとめ.md | 実験結果の横断比較テーブル |

## 30_Resources/
| ファイル | 内容 |
|---|---|
| 音源分離/TF-Locoformer_メモ.md | 主要モデルの論文メモ（status: read） |
```

これにより、「あのファイルどこだっけ？」という探索コストを大幅に削減できました。ファイルを追加・削除・リネームしたときは必ずこのファイルも更新します。

## テンプレートを整備する

Obsidianには**Templater**というプラグインがあり、新しいノートを作るときに「テンプレート」を使って自動的に日付や項目を埋め込めます。

以前のテンプレートには問題がありました。日付が「2026-05-12」のようにハードコード（固定値）になっていたのです。つまり、テンプレートを使っても日付が自動で入らない状態でした。

```yaml
# 修正前（日付が固定）
date: 2026-05-12

# 修正後（Templater変数で自動入力）
date: 2026-05-12
```

また、タイトルも自動で入るように修正しました：

```markdown
# 20260512_ObsidianとClaude連携_blog_draft
```

こうすることで、ファイル名を入力すればそのままノートのタイトルになります。

### 作ったテンプレート一覧

| テンプレート | 用途 |
|---|---|
| 実験ノート用 | 実験・作業ログ（日付・環境・結果・考察） |
| 論文ノート用 | 論文読書メモ（著者・年・会議名・要約） |
| 永続ノート用 | 概念を自分の言葉でまとめたもの（後述） |
| 週次レビュー用 | 毎週月曜の振り返り |
| 月次レビュー用 | 毎月末の振り返り |
| 執筆ドラフト用 | レポート・スライドの下書き |
| 【MOC】用 | MOCファイルの雛形（後述） |

## MOC（マップ・オブ・コンテンツ）とは

Obsidianを使う上で、ぜひ理解してほしいのが**MOC（Map of Content）**という概念です。

フォルダで整理するだけでは、関連するノートがバラバラのフォルダに散らばってしまいます。MOCは「ハブ」となるファイルで、関連するノートへのリンクを1か所に集めます。

```markdown
# 【MOC】プロジェクト研究A

## 参考論文
- [[TF-Locoformer_メモ]]
- [[Conv-TasNet_メモ]]

## 実験ノート
（Dataviewで自動取得）

## タスク一覧
- [x] 環境構築
- [ ] 標準データセットでの再現実験
```

フォルダは「保管場所」を決めるだけですが、MOCは「意味のある繋がり」を作ります。プロジェクトに必要なものが全部ここから辿れる状態が理想です。

また、MOCの末尾には以下を入れることで「最終更新日時」が自動で表示されます：

```
**最終更新:** `= this.file.mtime`
```

## Zettelkasten：知識を「資産」にする

最後に、このVaultの核心となる考え方を紹介します。

**Zettelkasten**（ツェッテルカステン）は、19世紀のドイツの社会学者ニクラス・ルーマンが考案した知識管理手法です。「1ノート1概念」の原則のもと、概念ノートをリンクでつないで知識のネットワークを作ります。

今回のVaultでは、論文を読んだら2種類のノートを作ります：

**論文メモ（`30_Resources/`）**  
「その論文に何が書いてあったか」の記録。著者・年・提案手法・実験結果などを整理する参照用のノート。

**永続ノート（`20_Areas/永続ノート/`）**  
「自分がその概念をどう理解しているか」の記録。論文の要約ではなく、自分の言葉で書いた解釈・気づき。

この2つの違いが重要です。論文メモは「あの論文に書いてあった」という記憶を補助するものですが、永続ノートは「自分の思考の蓄積」です。

```markdown
# Local-Global Attention の利点

長い音声信号を処理するとき、全体を一度に見るTransformerは計算量が爆発する。
TF-Locoformerが採用するLocal-Global Attentionは、
近くは細かく・遠くはざっくりと処理することで、
精度を保ちながら計算量を抑える。

## 他の概念との関係
- 関連: [[Self-Attentionの仕組み]]
- 応用: [[TF-Locoformer_メモ]]
```

永続ノートが増えていくと、新しい論文を読んだときに「あの概念と繋がる！」という接続が生まれやすくなります。

## 研究から執筆へのフロー

研究ノートが溜まっても、アウトプット（レポート・スライド）に結びつかなければ意味がありません。今回、「研究ノート → 執筆」のフローを明確化しました。

```
実験ノート ──┐
             ├→ 実験結果まとめ ──┐
論文メモ ───┘                    ├→ 執筆ドラフト → 最終成果物
             ┌→ 永続ノート ─────┘
勉強会ノート┘
```

1. **収集**：実験ノート・論文メモを日常的に書く
2. **統合**：実験結果まとめ・永続ノートで素材を整理する
3. **構成**：執筆ドラフトテンプレートでアウトラインを作る
4. **執筆**：ドラフトを書く（`10_Projects/下書き/`に保存）
5. **提出**：完成

## 自動化・快適化の工夫

### Dataviewで動的なダッシュボード

**Dataview**プラグインを使うと、ノートのメタ情報（タグ・日付・ステータスなど）をもとに動的なリストやテーブルを作れます。

`Home.md`（ダッシュボード）には以下のクエリを置いています：

```dataview
TABLE file.mtime AS "最終更新", length(filter(file.tasks, (t) => !t.completed)) AS "残タスク"
FROM "10_Projects"
WHERE contains(tags, "MOC") AND file.name != "【MOC】10_Projects"
SORT file.mtime DESC
```

これで「アクティブなプロジェクト一覧と残タスク数」が常に最新の状態で表示されます。

論文の読書状況も一目でわかります：

```dataview
TABLE WITHOUT ID file.link AS "論文", year AS "年", status AS "状態"
FROM "30_Resources"
WHERE contains(tags, "paper")
SORT status ASC, year DESC
```

### CSSスニペットで視認性を上げる

Obsidianはカスタムの**CSSスニペット**で見た目を変えられます。以下のようなスタイルを適用しました：

- **MOCファイル**をファイルエクスプローラーで太字・アクセントカラーで強調
- **タグ**をカラーバッジにする（`#paper`=緑、`#experiment`=紺、`#MOC`=紫）
- **H1に下線**を引いて見出しを目立たせる
- **完了済みタスク**を薄く表示して、残タスクを目立たせる

```css
/* MOCファイルをファイルエクスプローラーで強調 */
.nav-file-title[data-path*="MOC"] {
  font-weight: 700;
  color: var(--color-accent);
}

/* paper タグを緑にする */
a.tag[href="#paper"] {
  background-color: #2d6a4f !important;
  color: white !important;
}
```

### Gitで自動バックアップ

**obsidian-git**プラグインを使うと、Vaultの変更を定期的に自動でGitにコミットしてくれます。「うっかりノートを消した」「昨日の状態に戻したい」という場面で助かります。

`.gitignore`には以下を設定して、機密情報が含まれるファイルをバックアップ対象から外しています：

```
.obsidian/plugins/obsidian-local-rest-api/  ← APIキー（セキュリティ上重要）
.obsidian/workspace.json                    ← 頻繁に変わるため除外
```

## 実際にClaudeに頼んでみる

設定が整ったら、あとはClaude Codeに依頼するだけです。いくつか例を挙げます。

**論文メモの作成**
```
「TF-Locoformerの論文を読みました。タイトルはXXX、著者はYYY、2023年のICASSPです。以下の内容でメモを作ってください：[内容]」
```

**MOCの更新**
```
「今日の実験ノートをプロジェクト研究AのMOCにリンクしてください」
```

**永続ノートの作成**
```
「Conv-TasNetの論文メモから『Temporal Convolutional Network（TCN）の特徴』について永続ノートを作ってください」
```

**週次レビューの作成**
```
「今週の週次レビューノートを作って、未完了タスクをリストアップしてください」
```

CLAUDE.mdにルールが書いてあるので、命名規則・保存先・フロントマター形式を毎回指示する必要がなく、Claudeが自動で適切な形式で作ってくれます。

## .claude/settings.jsonで権限を事前設定

Claude Codeは、ファイルの読み書きなどの操作をするたびに確認を求めます。Vaultの読み取り操作などは毎回確認が煩わしいので、**`.claude/settings.json`**で事前に許可しておきました。

```json
{
  "permissions": {
    "allow": [
      "mcp__obsidian-mcp-tools__get_vault_file",
      "mcp__obsidian-mcp-tools__search_vault_simple",
      "mcp__obsidian-mcp-tools__search_vault_smart",
      "Bash(git status)",
      "Bash(git diff*)",
      "Bash(git log*)"
    ]
  }
}
```

読み取り操作と`git`の確認系コマンドだけを事前許可し、ファイル作成・コミットなどの変更操作は引き続き手動で承認するようにしています。

## 全体の構成まとめ

今回作ったシステムを図でまとめると：

```
Claude Code（ターミナル上のAI）
    ↕ obsidian-mcp-tools（MCP経由で接続）
Obsidian Vault
├── CLAUDE.md        ← Claudeへの作業規約
├── VAULT_INDEX.md   ← ファイル索引（トークン節約）
├── 00_Inbox/        ← とりあえずキャプチャ
├── 10_Projects/     ← 実験ノート・タスク
├── 20_Areas/        ← 永続ノート・学習ノート
├── 30_Resources/    ← 論文メモ
├── 40_Archives/     ← 完了済み
└── 99_Templates/    ← Templaterテンプレート
```

## まとめ：「ノートを書く」から「知識を育てる」へ

今回の取り組みで、ノートアプリの使い方が根本から変わりました。

以前は「とりあえずメモ → 埋もれる」というサイクルでしたが、今は：

- **PARAで整理**することで迷子のノートがなくなった
- **MOC**で関連ノートが繋がり、全体像を把握しやすくなった
- **Zettelkasten**で知識が蓄積され、「あの概念と繋がる」という接続が増えた
- **Claude Code**に頼むことで、ノート作成・整理の手間が大幅に減った
- **CLAUDE.md + VAULT_INDEX.md**の組み合わせで、AIとの連携コストも最小化できた

「Obsidianは難しい」「設定が面倒」と感じている方も多いと思いますが、一度仕組みを作ってしまえば、あとはClaude Codeが大部分の面倒を見てくれます。

研究・勉強・仕事など、知識を扱うあらゆる場面で応用できるシステムです。ぜひ試してみてください。
