

## データリンク層

### データリンク層とは

- TCP/IPモデルの中で**物理層とインターネット層の架け橋**になる層
- 物理層は「ビット（0と1）をケーブルや電波に流す」だけの世界で、ノイズによる誤りや、どこからどこまでが一つの意味あるデータかという区切りもない
- 一方インターネット層は「パケットを送りたい」と考えている
- データリンク層は、この間に立って、上位層から見たときに**隣接するコンピュータどうしが、あたかも誤りのない理想的な専用回線でつながっているように見せる**役割を担う

#### 接続形態：LAN と WAN

- **LAN（Local Area Network）** 
	- 同じ構内などの近距離で、**多数の端末が一つの伝送路を共有**する
	- 複数台が同時に喋ると衝突するので「次に誰が使うか」を決める制御が必要になる
	- これを**多重アクセス**と呼ぶ

- **WAN（Wide Area Network）** 
	- 拠点間など遠距離で、ルーター同士を**1対1**で結ぶ
	- 共有相手がいないので、誰が使うかを決める制御は不要

#### データリンク層のプロトコルスタック（2層構造）

- **LANの場合、データリンク層は2層に分かれる**
- WANはポイントツーポイントで多重アクセスがいらないので、MAC副層は不要
- **「LANでは誰が回線を使うかを決める必要がある → だからMAC副層が追加される」** 

#### データリンク層が提供する4つの機能

|機能|処理層|内容|
|---|---|---|
|サービスインターフェイス|論理リンク層|上位層に対しプロトコル（約束事）を提供|
|フレーム化|論理リンク層|ビットストリームをフレームに分割し、パケットとして上位へ渡す|
|誤り制御|論理リンク層|物理層で生じた伝送誤りに対処し、上位層からできるだけ隠す|
|フロー制御|論理リンク層|速い送信者が遅い受信者をデータであふれさせないよう流量を調整|
|多重アクセス|MAC副層|複数端末がある時、次に誰が使うかを決定|

#### NIC（ネットワークインターフェースカード）

- データリンク層は具体的には **NIC** というハードウェアで実装される
- **MAC controller**（フレーム化・衝突検知などMAC副層の処理）
- **PHY controller**（デジタル変換など物理層の処理）
- 、PCとはPCI express、LANケーブルとはRJ45コネクタでつながる
- 最近のPCではNICは独立した周辺機器ではなく、**マザーボードに内蔵**されているのが普通

#### 上位層に提供する3種類のサービス

- 論理リンクが上位層に提供するサービスは、**物理回線の品質に応じて**3段階ある
- 「誤り確認（Ack）の有無」と「コネクションの有無」の組み合わせ

|サービス|前提|例|トランスポート層での類似物|
|---|---|---|---|
|**Ackなし・コネクションレス**|誤りはあまり起きない|有線イーサネット|UDP|
|**Ackあり・コネクションレス**|誤りは常に起こりうる|無線LAN|（中間）|
|**Ackあり・コネクション指向**|信頼性が極めて低い|衛星通信・長距離電話|TCP|

- 回線が綺麗なら確認応答すら省いて速く送る（イーサネット＝UDP的）
- 回線が悪いほど確認と接続管理で信頼性を担保する（衛星通信＝TCP的）

### フレーム化

#### なぜフレームに区切るのか

- **誤り制御がしやすい** — 小さなフレーム単位で誤りを検出・訂正できる
- **フロー制御がしやすい** — フレーム単位でフィードバックを返せるので、細かい粒度で流量調整できる
- **多重化しやすい** — 複数の送信元・宛先のフレームを混在させられる（これがパケット交換の基本的な考え方）

#### 境目をどう決めるか

> ビットストリームからフレームを切り出すとき、**どこがフレームの境目か**をどう判別すればよいか？

**方法1：バイト数

- フレームの先頭に「このフレームは何バイトです」という数を書いておく方式
- その「バイト数」を表す部分のビット自体が誤って化けると、区切り位置がずれ、**それ以降のフレームがすべて連鎖的にエラーになる**

**方法2：バイト詰めを伴うフラグ・バイト**

- フレームの開始と終了を示す**特別なパターン**を置く方式
- 例：`01111110`（= 16進で `0x7E`）
- これを目印にすれば、区切りはフラグを探すだけで分かる
- 画像や映像などのデータ中に、たまたまフラグと同じビット列が出てきたら、そこを誤って「フレーム終了」と判断してしまう
- **解決策がエスケープバイト**
- データ中にフラグと同じパターンが現れたら、その直前に **ESC（例：`0x7D`）を挿入**して送る
- では、データ中にESC自身が現れたら？ → **ESCの前にもう一つESCを置く**
- 受信側は「先頭のESCは常に削除する」というルールで元に戻す

**方法3：ビット詰めを伴うフラグ・パターン**

- バイト詰めが8ビット単位だったのを、**1ビット単位**で行う方式
- かつての **HDLC** プロトコルのために開発され、PPPでも使われた
- **連続する6個の1（`111111`）が来たら → フレームの開始・終了**
- **送信側はペイロード中に連続する5個の1（`11111`）が現れたら、直後に0を1個挿入する**
- こうすれば、データの中身に偶然「1が6個連続」が現れることが原理的になくなり、`111111` は必ずフレーム境界を意味する
- 受信側は「1が5個連続した直後の0」を見つけたら、その0を取り除けば元に戻る

**方法4：物理層符号化違反**

- 物理層の符号化に**冗長性**がある場合、その使われていない予備パターンに「フレーム開始・終了」の意味を持たせる方式
	- 例：**4B/5B符号**（NRZ伝送路符号でクロック同期しやすくする拡張。古い100BASE-TXイーサネットで使用）
	- 4B/5Bの変換テーブルに**存在しないパターンが来たら「違反」→ フレームの開始か終了**と判断

- **最大の利点** 
	- 元々予備として余っている情報を使うので、バイト詰め・ビット詰めのような**詰め物によるオーバーヘッド（データ量の増加）がない**

> 実際のプロトコルは、これら4つを単独ではなく**組み合わせて**使います。例えばIEEE 802ファミリー（有線イーサネット・無線802.11）は、**プリアンブル**という7バイト（56ビット）のパターンでフレーム到着を示し、終了はフレームヘッダ内の**フレーム長フィールド**（＝バイト数方式）で判断します。

### 誤り制御

#### 2つのアプローチ

- **誤り検出** — 「誤りが起きた」ということ**だけ**を受信側が気づけるだけの冗長性をつけ、気づいたら**送信側に再送を要求する**
- **誤り訂正** — 受信側が**中身を復元できるだけの**十分な冗長情報をつけ、誤りが起きてもその場で直す（再送不要）

- それぞれを実現する技術が、**誤り検出符号**と**誤り訂正符号**

### どちらを使うか：回線の質で決まる

||適した状況|例|
|---|---|---|
|**誤り検出**|誤り率が極めて低い回線。誤りは稀なので、検出して再送する方が効率的|光ファイバー、品質の高い銅線|
|**誤り訂正**|誤り率が高い、または再送が原理的に不可能な場面|無線リンク、CD/DVD/BD、QRコード|

CDやQRコードは「読み取り時にもう一度送ってもらう」ことができないので、**訂正符号が必須**です。この対比はよく問われます。

### 誤り検出

#### 用語の定義

- **データ**：元のデータ … **m** ビット
- **検査ビット**：検査用に付け加えるビット … **r** ビット
- **符号語（code word）**：データ＋検査ビット … **n = m + r** ビット → これを **(n, m)符号** と書く
- **符号化率（code rate）**：`m / n` … 全体のうち本来のデータが占める割合

#### 主要な3つの誤り検出符号

1. **パリティビット**
2. **チェックサム**
3. **巡回冗長検査（CRC）**

**① パリティビット**

- **符号語に含まれる1の個数が偶数（または奇数）になるように**、検査ビットを1ビット付け加える方式
- 受信側は「1の個数が偶数になっているか」を数えるだけで誤りをチェックできます。

**パリティビットの問題点**

- **ランダムエラー** … めったに起きず、1ビットだけ化けることが多い → **エラー数が奇数** → パリティで検出**できる**
- **バーストエラー** … 連続した複数ビットがまとめて化ける → **エラー数が偶数になることがある** → このとき1の個数の偶奇が変わらず、**単一パリティビットでは検出できない**
- つまり**単一パリティはバーストエラーに弱い**のが弱点

**インターリービング**

- **パリティビットを「データが転送される順序とは違う順序」で計算する**手法
- 文字を縦に並べ、ビットを**縦方向**にまとめてパリティを計算してから、横方向に伝送する
- 伝送中に**連続して固まって起きたバーストエラー**が、パリティ計算の単位の上では**バラバラに散らされる**ため、孤立した誤りとして検出できるようになる

**② チェックサム**

- データ全体から**検査対象部分**を抜き出す
- それらの**和（2の剰余系）をとるとゼロになるような数**を検査ビット＝**チェックサム**とする
- チェックサムは通常**データの最後**に置く
- 受信側は全部を足し合わせて、**ゼロになっていれば誤りなし**と判断

**③ CRC**

- 1961年にPeterson・Brownが発表した、**パリティやチェックサムより強力な**誤り検出符号
- 多項式符号（polynomial code）とも呼ばれる

- データの偏りに依存せず安定した性能
- デジタル回路での実装が容易
- 適用例が非常に広い：**Ethernet, HDLC, Bluetooth, USB, gzip/png, MPEG-2** など

**考え方**

- ビット列を**係数が0か1の多項式**として扱う
- 例：`110001`（6ビット）↔ $1x^5 + 1x^4 + 0x^3 + 0x^2 + 0x^1 + 1x^0$
- そして**演算はすべて2の剰余系**で行う
- **桁上げ・借りが無いので、足し算も引き算も XOR（排他的論理和）と同じ**

**CRCの計算手順**

- $G(x)$：**生成多項式**（送受信で共有する除数。r+1 ビット）
- $D(x)$：元のデータ（mビット）
- $R(x)$：チェックサム（rビット）
- $S(x)$：データの後ろに **r個の0** を付けたもの（$x^r D(x)$ に相当）
- $T(x)$：実際に送るデータ＝ $S(x) - R(x)$

**手順：**

1. 元データの後ろに **r個の0** を付けて $S(x)$ を作る
2. $S(x)$ を $G(x)$ で**割り算（mod 2、つまりXORで引き算）**する
3. その**余り $R(x)$**（rビット）がチェックサム
4. 送信データ $T(x) = S(x) - R(x)$（＝0だった部分を $R(x)$ で埋めたもの）を送る
5. 受信側が $T(x)$ を $G(x)$ で割り、**余りが0なら誤りなし**

#### CRCの性質（誤り検出能力）

- **長さ r 以下のすべてのバースト誤りを検出できる**
- ただし、エラーパターン $E(x)$ が**たまたま $G(x)$ を因数に持つ**多項式に相当する場合は検出できない
    - 理由：受信ビットが $T(x) + E(x)$ のとき、$T(x)$ は $G(x)$ で割り切れる（余り0）うえ、$E(x)$ も $G(x)$ の倍数なら、合計も割り切れてしまい、余りが0＝「誤りなし」と誤判定される
- **それ以外のケースはすべて検出可能** → だから生成多項式 $G(x)$ の選び方が重要

#### 代表的な生成多項式

- **CRC-4**：$x^4 + x + 1$
- **CRC-16**、**CRC-32** … ビット長に応じて様々なバリエーションがある（より長く強力な検出能力を持つ）

### 誤り訂正

#### ① 反復符号

- **0 を送りたいときは `000`、1 を送りたいときは `111`** と、同じビットを3回くり返して送る
- 受信側は **多数決** で復号します。たとえば `010` を受け取ったら、0が2票・1が1票なので「元は 0 だった」と判断
- **(3, 1) 符号** … 1ビットのデータを3ビットに符号化する、という意味
- **符号化率 = 1/3** … 送るビットのうち本当の情報は 1/3 だけ。残り 2/3 は誤り対策のための「冗長」
- 1つの符号語（=3ビットのブロック）の中で **誤りが1ビットまでなら必ず訂正できる**（多数決が成立する）
- ただし符号化率 1/3 は非常に低く、実用的な符号としては効率が悪

#### ② 水平垂直パリティ検査符号 ((8,4) 符号)

- 単一パリティ検査符号では「誤りがあるか」は分かっても「どこが誤りか」は分からない
- そこで **データを2次元（縦横）に並べ、縦と横の両方でパリティを計算** 
- **誤った行と誤った列の交点** として、化けたビットの位置を1点に絞り込める

**具体例**: x1=0, x2=1, x3=0, x4=0 のとき

```
        0  1 | 1     p1 = 0⊕1 = 1 ✓
        0  0 | 0     p2 = 0⊕0 = 0 ✓
       -------+
        0  1         p3 = 0⊕0 = 0,  p4 = 1⊕0 = 1
```

- 送信側はこの8ビットを送る
- 受信側で全ての行・列のパリティが合っていれば誤りなしと判断
- たとえば **x2 が 1→0 に化けて** 受信したとする

```
        0  0 | 1     ← この行: 0⊕0 = 0 だが p1=1 → NG
        0  0 | 0     ← この行: 0⊕0 = 0 = p2 → OK
       -------+
        0  1
        ↑  ↑
       OK  NG ← この列: x2⊕x4 = 0⊕0 = 0 だが p4=1 → NG
```

- 1行目 → 合わない (NG)
- 2行目 → 合う (OK)
- 1列目 → 合う (OK)
- 2列目 → 合わない (NG)
- すると **NG だった行と NG だった列の交点 = x2** が誤りビットだと特定できる
- あとは x2 をビット反転（0→1）すれば訂正完了

#### ③ パリティ検査方程式と検査行列

上の手続きを、計算で機械的にやれるよう数式化したものです。符号語を

$$w = (x_1\ x_2\ x_3\ x_4\ p_1\ p_2\ p_3\ p_4)$$

と並べると、検査ビットは次の関係を満たします（資料の Yes/No 表がそのまま式になります）:

||x1|x2|x3|x4|式|
|---|---|---|---|---|---|
|p1|●|●|||p1 = x1 ⊕ x2|
|p2|||●|●|p2 = x3 ⊕ x4|
|p3|●||●||p3 = x1 ⊕ x3|
|p4||●||●|p4 = x2 ⊕ x4|

これを移項して「= 0」の形にしたものが **パリティ検査方程式**（2を法とする計算、つまり和は XOR）:

$$ \begin{aligned} x_1 \oplus x_2 \oplus p_1 &= 0\ x_3 \oplus x_4 \oplus p_2 &= 0\ x_1 \oplus x_3 \oplus p_3 &= 0\ x_2 \oplus x_4 \oplus p_4 &= 0 \end{aligned} $$

この4本の式の係数を行列にまとめたものが **検査行列 H**（4行×8列）です:

```
       x1 x2 x3 x4 p1 p2 p3 p4
H =  [  1  1  0  0  1  0  0  0 ]
     [  0  0  1  1  0  1  0  0 ]
     [  1  0  1  0  0  0  1  0 ]
     [  0  1  0  1  0  0  0  1 ]
```

このとき「w が正しい符号語である」ための **必要十分条件** は

$$w H^{T} = \mathbf{0}$$

と書けます（$H^T$ は H の行と列を入れ替えた転置行列、$\mathbf{0}$ は全成分0のベクトル）。正しい符号語なら、検査方程式がすべて成り立つので結果がゼロになる、という意味です。

#### ④ シンドローム（症候群）

ここが訂正の核心です。受信語 $y$ を検査方程式に入れた結果を **シンドローム $s$** と呼びます:

$$s = y H^{T}$$

いま $i$ ビット目だけが誤った状況を考えます。誤りベクトルを $e_i$（$i$ 番目だけ1、他は0）とすると、受信語は $y = w \oplus e_i$ なので、

$$s = y H^{T} = (w \oplus e_i)H^{T} = \underbrace{w H^{T}}_{=,0} \oplus\ e_i H^{T} = e_i H^{T}$$

これは **検査行列 H の第 $i$ 列**そのものになります。

> **H の各列がすべて異なっていれば、シンドロームの値を見るだけで「何ビット目が誤ったか」が一意に分かる → 訂正できる。**

誤りがない場合は $s = 0$、誤りがある場合は $s$ がその誤り位置に対応する列の値になる、という仕組みです。資料の「シンドロームの計算例」は、誤りなし→$s=0$、誤りあり→該当列の値、という確認をしているスライドです。

実際の計算は具体的に:

- 誤りなしの受信語 → 4本の検査式すべてが成り立つ → s = (0,0,0,0)
- どこか1ビット誤った受信語 → そのビットが関係する検査式だけが成り立たなくなり、s にそのパターンが現れる

#### ⑤ (7,4) ハミング符号

水平垂直パリティ符号 (8,4) は「8ビット送って4ビットの情報」でした。**ハミング符号はもっと効率が良く、(7,4) すなわち7ビットで4ビットの情報**を運びつつ、1ビット誤りを訂正できます。

データ `x1,x2,x3,x4`、パリティ `p1,p2,p3`。検査式は資料の表より:

$$ \begin{aligned} p_1 &= x_1 \oplus x_2 \oplus x_3\ p_2 &= x_2 \oplus x_3 \oplus x_4\ p_3 &= x_1 \oplus x_2 \oplus x_4 \end{aligned} $$

#### シンドロームの計算

符号語 $w=(x_1,x_2,x_3,x_4,p_1,p_2,p_3)$ に対し、シンドローム3ビットを

$$ \begin{aligned} s_1 &= x_1 \oplus x_2 \oplus x_3 \oplus p_1\ s_2 &= x_2 \oplus x_3 \oplus x_4 \oplus p_2\ s_3 &= x_1 \oplus x_2 \oplus x_4 \oplus p_3 \end{aligned} $$

で計算します。どのビットが誤ったか（エラーパターン）と、得られるシンドローム $(s_1,s_2,s_3)$ の対応は次の通り:

|誤った位置|s1|s2|s3|
|---|---|---|---|
|x1 (e1)|1|0|1|
|x2 (e2)|1|1|1|
|x3 (e3)|1|1|0|
|x4 (e4)|0|1|1|
|p1 (e5)|1|0|0|
|p2 (e6)|0|1|0|
|p3 (e7)|0|0|1|
|誤りなし|0|0|0|

**解き方のポイント**: シンドロームは3ビットなので $2^3 = 8$ 通りの値を取れます。そのうち「000」は誤りなしを表し、残りの **7通り**が「7ビットそれぞれが誤った場合」にちょうど1対1で対応しています（だから(7,4)で過不足なく効率的）。受信語からシンドロームを計算し、この表を引くだけで誤り位置が分かり、そのビットを反転すれば訂正できます。

たとえば計算したら $(s_1,s_2,s_3)=(0,1,1)$ になったら → 表より **x4 が誤り** → x4 を反転して訂正、という流れです。

#### ⑥ 主要な誤り訂正符号

- **ハミング符号 (1950〜)**: 高速だが訂正能力は低い。メモリやRAID2で使用
- **畳み込み符号 (1955〜)**: GSM携帯・衛星通信・802.11で広く使用。NASAのボイジャーでも採用
- **リード・ソロモン符号 (1960〜)**: 訂正能力が高く **バーストエラーに強い**。CD/DVD/Blu-ray、DSL、衛星通信などで利用
- **低密度パリティ検査符号 (LDPC) (1995〜)**: 訂正能力が極めて高く、理論限界（**シャノン限界**）に近いレートを達成。10Gイーサ、802.11n/ac、WiMAX、4K/8K放送など

### フロー制御

- 通信では、相手の状況に合わせて送信ペースを調整する必要がある
- これが **フロー制御** 
- プロトコルは3種類あり
	1. **フィードバックなし（理想的ネットワーク）**
	2. **逐次確認 (stop-and-wait)**
	3. **スライディングウィンドウ**

#### ① 理想的なネットワーク

- 次の条件がすべて成り立つ仮想的な状況です
	- 処理時間ゼロ
	- 受信側バッファ無限
	- 通信路に誤りなし
- この場合は **フィードバック不要で、最大転送率で送りっぱなしにできる**

#### ② 逐次確認

- **1フレーム送ったら、相手の確認応答（ACK）が返ってくるまで次を送らない** 方式
- ACK は小さなダミーパケット

#### 雑音があるケース：ARQ

- 通信路に誤りがあると、フレームやACKが失われる
- そこで **自動再送要求 ARQ (Automatic Repeat-reQuest)** を使用
- 送信者はACKが来るまで次を送らず、来なければ **タイムアウト** で再送する
- 問題: 受信者が「初めて来たフレーム」と「再送されてきた重複フレーム」を区別できないと困る
- 解決策: 各フレームのヘッダに **順序番号 (Seq)** を付ける。1フレームずつ送る場合は隣同士の順序（0と1の交互）だけ管理すれば十分

**【問題1】東京 ⇔ ワシントンD.C. の最高伝送速度**

> 逐次確認で通信する。フレームは固定1500バイト、光速で進み、帯域は無制限・雑音なし。東京〜ワシントンD.C.は約11000 km。達成可能な最高伝送速度は?

**解き方**: 逐次確認では「1フレーム送る → ACKが戻る」のに **往復遅延 (RTT, Round-Trip Time)** だけかかり、その間は1フレームしか送れません。だから「1フレームのビット数 ÷ RTT」が上限速度になります。

**ステップ1：RTTを求める** 往復距離は $2 \times 11000,\text{km} = 2.2 \times 10^7$ m、光速 $3.0 \times 10^8$ m/s なので

$$\text{RTT} = \frac{2.2 \times 10^{7}}{3.0 \times 10^{8}} \approx 0.073\ \text{秒}$$

**ステップ2：1フレームのビット数** 1500バイト $\times$ 8 = **12000ビット**

**ステップ3：伝送速度**

$$\frac{12000\ \text{bit}}{0.073\ \text{s}} \approx 163636\ \text{bps} = \mathbf{163.6\ kbps}$$

帯域が無制限なのに 163.6 kbps しか出ません。距離（遅延）が大きいと逐次確認は極端に非効率になる、という現象で **long pipe problem（ロングパイプ問題）** と呼ばれます。

**【問題2】RTTが大きいとき、速度を上げるには?**

**答え**: ACKを待たずに **複数フレームをまとめて先送りする** = **スライディングウィンドウ**を使う。これが次のテーマです。

#### ③ スライディングウィンドウプロトコル

- 送信者に **ウィンドウサイズの範囲内なら、ACKを待たずに複数フレームを同時に送ること** を許可
- 複数フレームを経路上に同時に流しておく技法を **パイプライン**と呼ぶ
- 送信者は「いま送ってよい順序番号の集合（ウィンドウ）」を常に管理し、ACKが来るたびにウィンドウを前へずらします（=スライド）。

**【問題3】パイプライン中にフレームが壊れたら?**

- **一括再送 (Go-back-N ARQ)**
    
    - 受信者は誤りを見つけたら、**それ以降のフレームをすべて捨て、ACKを返さない**
    - 受信者は順序通りのフレームしか受け付けない（バッファ不要 → 受信側ウィンドウ=1、送信側ウィンドウ=N）
    - 実装は簡単だが、1個壊れると後続を全部送り直すので無駄が出る
    - 
- **選択的再送 (Selective Repeat ARQ)**
    
    - 受信者は届いたフレームを **バッファに溜めておく**
    - 送信者は **タイムアウトしたフレームだけ** を再送する
    - 誤りを検出したら **否定的確認応答 (NAK)** を返すと、タイムアウトを待たずに再送でき性能が上がる
    - 送信側・受信側ともウィンドウ=N。効率は良いが受信側にバッファが必要

### メディアアクセス制御 (MAC)・多重アクセス

- 1本の共有チャネルを複数の端末が使うとき、みんなが一斉に喋ると衝突して通信が成立しない
- そこで「**次に誰が送信してよいか**」を決める制御が必要
- これを **MAC（メディアアクセス制御）** と呼ぶ
- データリンク層の **MAC副層** で実現

#### 多重化 vs 多重アクセス

||レイヤー|方式|特徴|
|---|---|---|---|
|**多重化（静的）** TDM/FDM/CDM|物理層|時間・周波数・符号で固定的に分割|回線交換的。**バースト性のあるデータには非効率**|
|**多重アクセス（動的）** CSMA等|データリンク層|**フレーム単位**で動的に割り当て|パケット交換的。**バースト性データに向く（効率的）**|

#### ① ALOHA

- 1970年代初頭、ハワイ大学が離島とホノルルのメインコンピュータを短波無線でつなぐために開発

### ピュアALOHA

ルールは極めて単純: **送りたいデータがあればいつでも送ってよい。衝突して壊れたら、ランダムな時間だけ待って再送する。**

**性能解析（解き方の流れ）**:

- 各フレームの送信に $t$ 秒かかるとする
- $t$ 秒間に発生するフレーム数の平均を $G$ とおく（=印加負荷）
- フレーム発生はランダムなので、フレーム数は平均 $G$ の **ポアソン分布** に従う：$P(k) = \dfrac{G^k e^{-G}}{k!}$
- あるフレームBが成功するには、**衝突しうる区間（前後あわせて $2t$ 秒）** に他のフレームが1個も発生しないことが必要。$2t$ 秒間の平均発生数は $2G$ なので $$P_{\text{成功}} = P(\text{0個}) = e^{-2G}$$
- スループット $S$ は「負荷 × 成功確率」: $$S = G,e^{-2G}$$
- これを $G$ で微分して最大を求めると $G = 1/2$ のとき最大で、 $$S_{\max} = \tfrac{1}{2}e^{-1} \approx 0.184\ \text{フレーム/秒}$$

つまり **最大でもチャネルの18%程度**しか使えず、非効率です。

#### スロット化ALOHA

任意のタイミングではなく、**時間をスロットに区切り、スロットの先頭でしか送信させない**よう同期させます。すると衝突しうる区間が $2t$ 秒から **1スロット($t$秒)** に半減します。

$$P_{\text{成功}} = e^{-G}, \qquad S = G,e^{-G}$$

最大は $G = 1$ のとき $S_{\max} = e^{-1} \approx 0.368$ フレーム/秒。**ピュアALOHAのちょうど2倍**の効率です（送信タイミングを揃えただけで倍増する、というのがポイント）。

### ② CSMA（搬送波検知多重アクセス)

- 各端末が送信前に **搬送波（=チャネルが使用中か）を確認し、空いてから送る**方式
- イーサネットや無線LANの基本
#### CSMAの種類

- **永続CSMA (1-persistent)**
	- チャネルを継続監視し、**空いたら即座に**送信
	- 衝突したらランダム時間待って再送
- **p-永続CSMA**
	- 空いたら **確率 p で** 送信（確率 1−p で見送る）
	- 無線LANはこれを改良したものを利用
- **非永続CSMA**
	- 継続監視はしない
	- 空きを見つけたら送るが、**積極的には空きを探さない**
	
- **p-永続・非永続のほうが高いスループット**を出せる
- 「全員が慎ましく空気を読んで行動すると、全体最適につながる」という直感的な結論

#### CSMAの拡張：CD と CA

- **CSMA/CD（衝突検出, Collision Detection）**
	- 有線クラシックイーサネット (IEEE 802.3) で使用
	- 同軸ケーブルを全員で共有し、送信中に自分の信号と違う電気信号（**ジャム信号**）を検知して衝突を知る
	- **スイッチングハブの登場で使われなくなった**

- **CSMA/CA（衝突回避, Collision Avoidance）**
	- 無線LAN (802.11) で使用
	- 無線では物理的に衝突検出ができないため、衝突を**避ける**方向に
	- チャネルが空いた後、**先にランダムなバックオフを入れてから**送信を開始する
	- 衝突したか否かは **ARQ（ACKが返るか）** で推定

#### 衝突検知 → 指数バックオフ

- CSMA/CDで衝突したら、**指数バックオフ**で再送
- 待ち時間の候補スロット数を衝突のたびに倍々に広げる方式

- 1回目の衝突 → 0〜1 (= $2^1$通り) の中からランダムに選ぶ
- 2回目 → 0〜3 (= $2^2$通り)
- 3回目 → 0〜7 (= $2^3$通り)
- … $n$回目 → 0〜$2^n - 1$
- 再送回数が上限を超えたら諦める
- 衝突が続くほど待ち時間の幅を広げ、混雑時に皆が散らばるようにする工夫

#### CSMA/CAの動作と DCF

- CSMA/CAでは各局が独立に判断して動く
- この分散制御モードを **DCF (Distributed Coordination Function)** 
- 集中制御の **PCF** も規格上あるが、ほとんど使われていない

### データリンクプロトコルの具体例

#### ① IEEE 802 ファミリー

- IEEE 802委員会が定めるLAN関連規格群
- 重要な注意点として、**データリンク層だけでなく物理層も含めて標準化**されている
- 論理リンク制御(LLC)が802.2、有線が802.3、無線が802.11など

#### ② イーサネット (Ethernet)

- 1973-74年にXerox PARCの **ロバート・メトカーフ** が開発
- 語源は、かつて光の伝搬に必要と考えられていた仮想媒質「エーテル(ether)」

**変遷**:

- **クラシックイーサネット**: 同軸ケーブルを共有、半二重通信、衝突するので **CSMA/CD（指数バックオフ）** を使用
- **スイッチ式イーサネット**: スイッチングハブで各端末が専用回路で接続され、原理的に衝突しない。**全二重通信**で CSMA/CD 不要。100Mbps〜100Gbps超へ高速化

**規格表記 `N BASE-X`**: N が速度(1/10/100/1000/10G…)、X が伝送媒体(2,5=同軸、T=ツイストペア、S/L=レーザー波長 等)。

**フレーム構造**: プリアンブル+SoF(8B) → 宛先DA(6B) → 送信元SA(6B) → Type(2B) → データ(0〜1500B) → Pad(0〜46B) → **FCS（チェックサムによる誤り検出）**。

**MACアドレス**: 6バイト。前半3バイトが **OUI（組織識別子、例 00:00:00 = Xerox）**、後半3バイトが各組織がデバイスへ割り当てる番号。スイッチは受信履歴から「どのポートにどのMACがいるか」を **学習** して転送表を作ります。

**ループ対策**: スイッチがループを作ると **ブロードキャストストーム**（フレームが無限増殖）が起きます。これを防ぐため、スイッチの1つを根(ルート)とした木構造 = **スパニングツリー(STP / RSTP, 802.1D)** を作り、余分な（非指定）ポートを閉じてループを断ち切ります。

#### ③ 無線 IEEE 802.11 / Wi-Fi

注意: **802.11(IEEEの技術規格) と Wi-Fi(業界団体Wi-Fi Allianceの相互接続認定) は別物**です。2.4GHz帯・5GHz帯のISMバンドを使います。MAC副層は **半二重・CSMA/CA・Stop-and-Wait ARQ**。

**動作モード**: AP(アクセスポイント)経由の **インフラストラクチャモード** と、端末同士が直接つながる **アドホックモード**。サービスセットは BSS（基本）、それを有線で束ねた ESS（拡張）があり、ESS内のAPは同じ **SSID（ネットワーク名）** を持ちます。

**AP接続手順**: Probe（探索）→ Beacon応答 → Authentication（認証）→ Association（登録）→ Data、の順。フレームには データ/制御/管理 の3種があり、ビーコン・プローブ・認証・アソシエーションは管理フレームです。

#### ④ PPP (Point-to-Point Protocol)

- 1対1接続用のプロトコル
- 主にWAN（アナログ電話、ISDN、ADSL等）で使われた
- 90年代のモデム接続で活躍
- イーサネットと違い **物理層の規定を含まない**
- コネクション型で、確立 → ネゴシエーション（認証・暗号化・圧縮）→ 通信開始、の手順を踏む
- **LCP**: 上位層に依存しない機能（コネクション確立、認証プロトコル設定など）
- **NCP**: 上位層に依存する機能（IPなら IPCP でIPアドレス設定やヘッダ圧縮）
- **認証**: **PAP**（ID+パスワードを **平文** で送る → 弱い）と **CHAP**（チャレンジ＆レスポンスの **ワンタイムパスワード** → 安全）
- **フレーム**: HDLC準拠。フラグバイト 0x7E で囲み、ヘッダ(アドレス/制御/Protocol) → データ → **FCS（チェックサムで誤り検出）**
