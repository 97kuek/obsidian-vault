
## Web技術・基礎編

### Webとは何か

- **WWW（World Wide Web）** は、1989年にCERNの物理学者ティム・バーナーズ＝リーが発明した、ハイパーリンクで相互接続された情報空間
- 通信には **HTTP/HTTPS** プロトコルを使い、これはアプリケーション層で実装される
- こで混乱しやすいのが「Web ≠ インターネット」という点
- インターネットは下層（IP・TCPなど）のネットワーク基盤そのもので、Webはその上で動く**1つのアプリケーション**にすぎない

### HTTPとハイパーテキスト

- **HTTP（HyperText Transfer Protocol）** はWWWで使われる通信プロトコル
- ハイパーテキストとは「参照付きの電子的文章」のこと
- HTTPはWeb以外にも、ヘルプ・辞書・電子書籍・PDFなどでも使われる
- 文書の記述には **HTML（HyperText Markup Language）** を用いる

### Webが革新的だった理由

- ハイパーリンクの導入により、**異なるサイト・異なるメディア・似た興味を持つ人々をつなぐリンク構造**を提供したことが革新的
- インターネット上のホストであればどこでも自由につながる

- ウェブネットワークの登場（アクセスできる世界が広がる）
- サイト間リンクによる「導線（道）」の形成 → 人気サイトに人が集まる
- 広告収入モデル（クリック先＝人々の興味）→ 大きなビジネス機会
- 人間関係を反映したソーシャルメディアの誕生

### HTTPの歴史

|バージョン|時期|ポイント|
|---|---|---|
|HTTP/0.9|1991|最初期|
|HTTP/1.0|1991–1995 (RFC 1945)||
|HTTP/1.1|1996–1997 (RFC 2068)|長く主流|
|HTTP/2|2012–2015 (RFC 7540)|**性能改善**。元はGoogle SPDY|
|HTTP/3|標準化進行中|HTTP/2の拡張、**QUIC（UDPベース）上で動作**（RFC 9000）|

### URL と URI

- **URL（Uniform Resource Locator）** は、上位概念 **URI（Uniform Resource Identifier）** の一例
- URIの構文：`scheme:[//authority]path[?query][#fragment]`
- 例：`https://nsl.cs.waseda.ac.jp/tokyo2020/`
    - `https` = scheme（暗号化されたhttp）
    - `nsl.cs.waseda.ac.jp` = ドメイン（ホスト）
    - `/tokyo2020/` = ファイルパス

- つまり「ホスト名で示されたサーバから、ファイルパスのオブジェクトをhttpsで取得（GET）する」という指定

### HTML

- テキスト・画像・動画・リンクなどを含むWebページを作るための**マークアップ言語**
- マークアップ言語とは「テキスト整形のための明示的なコマンドを含む言語」
- 1997年以降はW3C勧告として標準化された
- HTMLは機能を追加しながらバージョンが進化
- なお**HTML5は2021年1月に廃止**され、後継は **HTML Living Standard**

### HTTPリクエストメソッド（RFC 7231）

- クライアントがサーバに「何をしてほしいか」を伝える動詞

|メソッド|意味|
|---|---|
|**GET**|URLで指定したデータを要求（取得）|
|HEAD|GETと同じだがヘッダのみ取得|
|**POST**|データを送信（name, value の組）|
|PUT|指定ファイルを置き換える|
|DELETE|指定ファイルを削除|
|TRACE|リクエスト内容をそのまま返す（エコー）|
|OPTIONS|オプションの通知|
|CONNECT|HTTPトンネリング（proxy + SSL/TLS時など）|

### HTTPステータスコード（RFC 7231）

- サーバがリクエストに応答する3桁の数字
- **百の位で大分類**されます（2xx=成功, 3xx=リダイレクト, 4xx=クライアント側エラー, 5xx=サーバ側エラー）

| コード | 名称                    | 説明                      |
| --- | --------------------- | ----------------------- |
| 200 | OK                    | 成功                      |
| 206 | Partial Content       | 部分的なコンテンツ（途中から開始）       |
| 302 | Moved Temporarily     | 一時的に移動（リダイレクト）          |
| 401 | Unauthorized          | アクセス権がない（**認証に失敗**）     |
| 403 | Forbidden             | アクセス拒否（**認証によらず許可がない**） |
| 404 | Not Found             | リソースが存在しない              |
| 500 | Internal Server Error | サーバ内部のエラー               |
| 503 | Service Unavailable   | 一時的に利用不可（高負荷など）         |

- **401＝認証に失敗、403＝認証に関係なく許可されていない**。

### WWWのアーキテクチャ

- 基本は**クライアント・サーバモデル**
- サーバは到達可能なIPアドレスさえ割り当てられていればインターネット上のどこにあってもよい
- クライアント・サーバモデルとは、タスクを役割で分ける方式で、
- **サーバ**：リソースやサービスを「提供」する
- **クライアント**：リソースやサービスを「要求」する
- 「お店と顧客」のような**非対称な関係**

## Web技術・発展編

### なぜ発展技術が必要になったか

- Webは規模と質の両面で激変した
- **規模**：1990年代はサーバ数1〜100万・ユーザ1600万〜2.5億人 → 2020年代はサーバ10億・ユーザ46億人
- **質**：静的コンテンツ → 動的コンテンツ、広告収入モデル定着、**一極集中化**
- この変化に対応する技術が、(1)アーキテクチャの進化、(2)分散技術、(3)高速化、の3本柱です。

### (1) アーキテクチャの進化 

- 単純なHTML・画像配信なら **二層（クライアント・サーバ）** で十分
- しかしDB検索や動的コンテンツ生成のような高度処理には **三層アーキテクチャ** を採用する
- 階層化することで**処理を分散**でき、スケーラビリティが上がる

**三層モデル**

|層|担当|実装例|
|---|---|---|
|プレゼンテーション層|データ表示（クライアント）|ブラウザ|
|アプリケーション層（ミドル層）|データ加工|Webサーバ Apache / アプリサーバ JavaEE, .NET, Rails, Node.js|
|データ層|データアクセス|DBサーバ MySQL, Oracle|
- 流れとしては以下の通り
	1. クライアントがリクエスト
	2. Webサーバがクエリ入力
	3. DBサーバが検索結果出力
	4. Webサーバがデータ加工してHTML出力

#### 動的ウェブコンテンツ

- プログラムでブラウザ表示を動的に生成
- プログラムはサーバ側（アプリサーバ）とクライアント側（**JavaScript**）の両方で動作
- 特に **AJAX** はクライアント側JavaScriptで**サーバとの非同期通信**を実現

### (2) 分散技術

**ローカルな分散（サーバファーム）** 複数サーバからなるPCクラスタが単一サーバのように振る舞う。数千台規模になることもあり、データセンターで19インチラックに一式（サーバ・スイッチ・電源・空調）が運用される。負荷分散には**ロードバランサ**や**DNSラウンドロビン**を使う。

- **リバースプロキシ**：アクセス頻度の高いリクエストをフロントエンドでキャッシュ → バックエンドの処理負荷を軽減

**グローバルな分散（CDN：Content Delivery Network）** コンテンツ配信を広域ネットワークで負荷分散し、**物理的に近くて空いているサーバ**から配信。配信の仕組みは2つ：

- **ミラーリング**：事前にCDNノードへ配信、リクエスト発生時にコピーを複製（キャッシュ）
- **DNSリダイレクション**：DNS名前解決時にクライアントのIPをもとに**最も近いCDNノードのIPを返す**（CNAMEトリック）

### (3) 高速化

- **Amazonは遅延が100ミリ秒延びると売上が1%落ちる**

#### ① キャッシュの利用（データの再利用）*

- アクセス数分布は偏っている
- **Zipf則**：少数のオブジェクトに多数のアクセスが集中するため、キャッシュが効く
- ブラウザやProxyサーバがテキスト・画像をストレージに記憶し、条件が合えば再利用

キャッシュ制御は**HTTPの Cache-control ヘッダ**（サーバで設定）：

|設定|意味|
|---|---|
|no-store|一切キャッシュさせない|
|no-cache|キャッシュが使えるか毎回確認させる|
|public|他のブラウザもキャッシュ利用可|
|private|特定ブラウザのみキャッシュ利用可|
|max-age=xxxx|xxxx秒以内ならキャッシュ有効|

- 静的コンテンツ（更新されない画像）はキャッシュさせたい
- 動的コンテンツ（カスタム広告）はさせたくない、という使い分けで

#### ② プロトコルの改良（HOLブロッキングの解消）

- **HOL（Head-of-Line）ブロッキング**
	- あるリソースをリクエストすると、その応答が戻るまで次のリクエストを送れない問題
	- **3章後半のストップアンドウェイト（逐次確認）と本質的に同じ**で、各コネクションでRTT時間待つのが無駄になる
- **HTTP/1.1のパイプライン**：応答受信前に並行リクエスト送信可。ただし実装が難しく普及せず
- **複数HTTP接続**：1ドメインあたり6コネクション程度。サブドメインを変えてコネクション数を稼ぐ実装も
- **HTTP/2**（前身はGoogleの**SPDY**）：上記制約を緩和、主流ブラウザがサポート

**SPDYが導入した概念**：多重化ストリーム（単一TCP接続でリクエスト/レスポンスを処理しTCP確立のオーバーヘッドを回避、各パケットを個別ストリームに分類）、リクエストの優先度制御、HTTPヘッダー圧縮（Cookie・User-Agent等の繰り返し送信を解消）。

**HTTP/2が導入した概念**

1. テキスト形式 → **バイナリ形式**へ（バイナリフレーム化、HPACKエンコード RFC 7541）
2. **多重化ストリーム**
3. **フロー制御**
4. ストリームの**優先度制御**
5. **ヘッダー圧縮**
6. **サーバープッシュ**
