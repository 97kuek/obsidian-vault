# 【保存版】AIエージェントのためのObsidian活用術

『自分の代わりに仕事をしてくれるAI社員を作りたい』

こういう声は多いですが、実際のところあなたがマイクロマネジメントをして結局あなた自身の手離れは進まないという結末が大半です。

原因は様々ありますが、そもそもAI社員を仕事が任せられる状態に仕上げてないことがほとんどです。Work ReadyなAI社員にするのは簡単なようで意外と難しいです。

特に前提情報や必要に応じてコンテキストをうまく渡す記憶設計などで詰まる人が多いです。

逆にここを解決できればあとは自社のそれぞれの仕事に合わせてチューニングしていくことであなたのチーム専属のAI社員を作ることができます。

半年前に決めた方針と、その後どう判断が積み上がったかを、1本の流れにまとめたい。手元の Obsidian には、その会議のメモも、途中の議事録も、関連する数字も、確かに書いてあるはずです。

Claude Code に Vault を渡して、こう頼みます。

「この案件について、過去に決めたことを時系列で出して」。返ってくるのは、関係しそうなノートを浅くまとめた文章だけ。肝心の「いつ何を決めたか」は拾えていません。自分で検索しても、似た言葉のノートが何十件も並ぶだけで、目的の1件にたどり着けない。数百ページ書いたのに、必要な瞬間に何ひとつ引き出せない。このまま増やし続けても、引き出せないノートは、書いていないのと同じです。

ここで多くの人が「AIがまだ賢くないから」と考えます。でも、原因は別のところにあります。ノートを取り出せないのは、Vault が「しまうため」に作られていて、「AIが引き出すため」に設計されていないからです。

今回はAIエージェントを第一の読み手に置いて Obsidian を設計し直す方法を一緒に見ていきます。AIエージェントネイティブにObsidianを設計するためのヒントが下記の記事に詰まっていました：

[https://x.com/cyrilXBT/status/2058373087330959829](https://x.com/cyrilXBT/status/2058373087330959829)

今回の記事では、この内容を日本の業務現場でAIに引かせる視点から組み直して解説します。読み終えるころには、「あの案件の過去の判断を出して」と聞けば、AIが関連メモごと数十秒で返してくる状態の作り方が分かります。

## ① AIに渡しても要約しか返ってこないObsidian

[](/ObsidianOtaku/article/2060308984221868163/media/2060305341104152577)

先ほどの場面を、もう少し具体的にしてみましょう。

新規事業の検討を半年続けてきたとします。市場調査のメモ、競合の価格分析、役員からのフィードバック、ピボットを決めた日の議事録。Obsidian の中に、全部あります。来週の経営会議で「これまでの意思決定の流れ」を1枚にまとめる必要があり、AIに手伝ってもらおうとします。

ところが、Claude に Vault を読ませて「新規事業の意思決定を時系列で出して」と頼むと、返ってくるのは「新規事業に関するいくつかのノートがあり、市場調査や競合分析が含まれています」という、目次のような要約です。あなたが欲しいのは「3月にこの方針を決め、4月の競合分析を受けて5月に修正した」という筋なのに、その筋は出てきません。

これは、AIの能力の問題ではありません。同じ Claude でも、引き出せる Vault と引き出せない Vault があります。その差は、ノートの中身ではなく、ノートの「外側」にあります。AIが「いつ・何について・どの状態のノートか」を判断する手がかりが、Vault に用意されていないのです。

以前のシリーズで、私は Vault に知識を蓄積する方法を扱ってきました。フォルダの組み方、CLAUDE.md でAIに前提を渡す方法、夜のあいだにノートを自動で取り込む仕組み。蓄積する装置は、もう持っているはずです。けれど、蓄積と引き出しは別の設計です。今日は、その引き出しの側を作ります。

## ② しまう設計と引き出す設計は別物

[](/ObsidianOtaku/article/2060308984221868163/media/2060305659707699200)

なぜ蓄積はあるのに引き出せないのか。理由は、ほとんどの Vault が「保存する瞬間」に合わせて作られているからです。

ノートを書くとき、人は「とりあえずどこかに置く」ことを考えます。

フォルダを開いて、それらしい場所に入れる。この瞬間は、それで困りません。困るのは、半年後に取り出すときです。保存した瞬間に「ここでいい」と思った場所は、取り出すときには何のヒントにもなりません。

「アイデア」というフォルダに入れたビジネスの着想を、半年後に探そうとして、それが「アイデア」なのか「案件」なのか「議事録」なのか思い出せない、という経験は誰にでもあります。

引き出すための設計は、評価の軸が違います。フォルダをきれいに分けることが目的ではありません。「将来この情報が必要になったとき、自分とAIは何を手がかりに探すか」を先に決めて、その手がかりをノートに持たせておくことが目的です。

フォルダの組み方そのものは、以前のシリーズで扱いました。5個から8個のトップレベルのフォルダにまとめる、という基本は、すでに作っている前提で進めます。今日の主役はフォルダではなく、ノート1枚ずつが持つ「AIが読む手がかり」です。

まず、自分の Vault が「しまう設計」で止まっていないか、診断してみましょう。

当てはまる項目が多いほど、蓄積した知識がAIに届いていないことになります。ここから先で、1つずつ埋めていきます。

## ③ AIを第一の読み手に置く

[](/ObsidianOtaku/article/2060308984221868163/media/2060305860791070720)

引き出せる Vault を作る出発点は、設計の主語を変えることです。

これまで、ノートを整える基準は「人間が探しやすいか」でした。これからは「AIエージェントが引き出せるか」を基準にします。これがこの記事のタイトル「AIエージェントのためのObsidian」の意味です。Vault を、人間が目で探す場所ではなく、AIが第一の読み手として読む場所として設計します。

なぜ人間ではなくAIのために設計するのか。AIが正確に引き出せるようになると、人間が「探す」という行為そのものがなくなるからです。

フォルダを開いて記憶を頼りにたどる代わりに、AIに「あの案件の過去の判断を出して」と話しかければ、AIが代わりに引いてくる。探す時間がゼロに近づきます。人間のために整える設計は、人間が探し続けることを前提にしています。AIのために設計すると、その前提ごと消えます。

これは奇抜な発想ではありません。元記事も、よく設計された構造がAIの検索精度を直接上げる、と書いています。

整える作業とAIに引かせる作業は、別々のものではなく、同じ1つの設計の表と裏です。ノートにAIが読める手がかりをつけることは、人間にとっての見やすさにもなり、同時にAIにとっての引きやすさになります。

ここで大事なのは、AIが読むための手がかりは、人間が貼る整理ラベルとは別物だという点です。人間は色や場所で何となく覚えますが、AIはノートに書かれた構造化された情報を読みます。

だから、AIのために設計するときは、「AIが何を手がかりにノートを引くのか」を具体的に知っておく必要があります。それが次の話です。

## ④ AIが必ず使う4つの手がかり

[](/ObsidianOtaku/article/2060308984221868163/media/2060305997529530368)

将来ノートを引き出すとき、自分とAIが必ず手にしている手がかりは4つしかありません。

実際の業務でAIに何かを頼むとき、私たちは無意識にこの4つを組み合わせています。「進行中の(status)、新規事業案件で(topic)、先月以降の(when)、意思決定(type)を出して」。

この1文は、4つの手がかりの組み合わせです。

逆に言えば、ノートにこの4つが記録されていればAIはこの問いにそのまま答えられます。記録されていなければ、AIはノートの本文を一字ずつ読んで推測するしかなく結果として「浅い要約」しか返せません。第1章で要約しか返ってこなかったのは、この4つがノートになかったからです。

ここから先の章は、この4つの手がかりをノートに持たせる具体的な方法です。順番に、ファイル名(命名)、ノート冒頭の情報欄(プロパティ)とタグ、と進みます。どれも、新しいツールを入れる話ではありません。今ある Obsidian のノートに、AIが読める手がかりを足していくだけです。

## ⑤ ファイル名だけで中身が分かる命名の型

[](/ObsidianOtaku/article/2060308984221868163/media/2060306223854174208)

4つの手がかりのうち、いちばん手軽に持たせられるのがファイル名です。

ファイル名に型を決めておくと、AIはノートを開く前に、名前だけで「いつの・何の・何についての」ノートかを判断できます。おすすめの型は、日付から始めるものです。

日付を先頭に置くと、3つのことが同時に起きます。ファイルが自動で時系列に並び、最近のノートが上に来ます。正確な名前を思い出せなくても、だいたいの時期から絞れます。同じ主題のノートが別の日に作られても、名前がぶつかりません。

種類と主題をファイル名に入れておくと、AIは本文を読まなくても「これは新規事業の会議録だ」と判別できます。第4章の4つの手がかりのうち、when(日付)、type(種類)、topic(主題)の3つが、ファイル名だけで埋まります。

この命名には、もう1つの強みがあります。入口で使うツールを変えても、AIのモデルが新しくなっても、ファイル名そのものは残ります。Vault の中身は、ツールが移り変わっても引き継げる資産になります。

ファイル名で埋まらない残りの1つが、status(状態)です。状態は日々変わるので、ファイル名には向きません。これは、次の情報欄で持たせます。

## ⑥ AIが読むメタデータを設計する

[](/ObsidianOtaku/article/2060308984221868163/media/2060306377873223680)

ファイル名の外側で、AIにさらに細かく引かせるための情報を、ノート自身に持たせます。Obsidian には、ノートの冒頭に情報欄を置く仕組みがあります。

これは、ノートのいちばん上に --- で挟んで書く、項目と値の一覧です。人間向けの見出しや装飾ではなく、AIが構造化された情報として読む部分です。

この5項目が、第4章の4つの手がかりをそのまま埋めます。type、date(when)、topic はファイル名と重なりますが、情報欄にも持たせておくと、AIが本文を開いたときに構造として読めます。そして、ファイル名に向かなかった status を、ここで持たせます。

引き出しでいちばん効くのが、この status です。「進行中の案件だけ」「完了したものを除く」といった絞り込みが、status ひとつでできます。AIに「進行中の新規事業ノートを出して」と頼めば、status が「進行中」のノートだけを返します。

種類によっては、追加の項目を足すと、もっと細かく引けます。

タグも、AIが意味で絞るための手がかりです。Obsidian のタグは3つの種類に分けて、頭に印をつけると、AIも人間も「何で絞っているか」がすぐ分かります。

tags: はまとめて並べて書きます(1つでも複数形で書くのが今の標準です)。タグを増やしすぎると、かえって絞れなくなります。5個以上のノートで使う見込みのあるものだけ作る、という線引きが効きます。

ここまでで、ファイル名と情報欄とタグに、AIが読む手がかりが揃いました。あとは、これをAIにどう引かせるかです。

## ⑦ 探す行為そのものを消す

[](/ObsidianOtaku/article/2060308984221868163/media/2060306725799116800)

ここまでで、Vault はAIが読める手がかりを持ちました。最後に、これをAIに引かせます。

Obsidian の Vault は、中身を見るとただのテキストファイルが集まったフォルダです。だから、Claude Code をその Vault のフォルダで開くだけで、Claude はノートを全部読めます。特別なプラグインを入れる必要はありません。Vault のフォルダを Claude Code に読ませる、それだけです。

あとは、自然な言葉で頼みます。検索のやり方を覚える必要はありません。

これを頼むと、AIはファイル名の日付と種類を見て時期と種類を絞り情報欄の status で状態を絞り、タグで主題を絞り、本文を読んで関連を判断します。

第1章で「浅い要約」しか返さなかったAIが、同じ Vault でも、手がかりを持たせた後は「3月にこの方針を決め、4月の競合分析を受けて5月に修正した」という筋を、関連ノートごと返してきます。違いは、AIの賢さではなく、ノートが手がかりを持っているかどうかです。

このとき、人間がやっていた「どのフォルダだったか思い出す」「検索語を変えて何度も試す」という作業は、まるごと消えます。「探す」のではなく「聞けば出てくる」。これが、AIエージェントを第一の読み手に置いて設計した Vault の到達点です。

第1章の経営会議の準備に戻れば、「この案件の意思決定を時系列で、関連する数字も添えて出して」の一言で、論点ペーパーの下地が返ってきます。あなたは、その筋が正しいかを判断することに時間を使えます。引き出す作業はAIが、判断は人間が受け持ちます。

## ⑧ 次の1ノートから始める

[](/ObsidianOtaku/article/2060308984221868163/media/2060308063513694209)

ここまで読むと、数百ページある既存のノートを全部作り直さないといけないように感じるかもしれません。その必要はありません。

作り直しは、いちばん挫折しやすいやり方です。代わりに、次に作る1ノートから始めます。

この型で作ったノートは、その日からAIが引けます。1週間後には、今週作ったノートはどれも「あの件のメモを出して」で返ってくる状態になります。過去のノートは、必要になって開いたときに、ついでに名前と情報欄を直していけば十分です。完璧に整え終わるのを待つ必要はありません。

蓄積した知識は、構造を持たせてはじめて、引き出せる資産になります。手がかりのないノートは、書いてあってもAIには届きません。次の1ノートに10分かけるだけで、あなたの Vault は、AIが読み手として引ける場所に変わり始めます。

## まとめ

[](/ObsidianOtaku/article/2060308984221868163/media/2060308235094237184)

- ノートを引き出せないのは、AIの能力ではなく、Vault が「しまう設計」で止まっているから。引き出す設計は「将来何を手がかりに探すか」を先に決める
- 設計の主語を人間からAIに移すと、人間が「探す」行為そのものが消える
- AIが引くときに使う手がかりは4つ。種類 / 時期 / 主題 / 状態
- ファイル名を「日付-種類-主題」の型にすると、AIは開く前に中身を判別できる
- ノート冒頭の情報欄(プロパティ)とタグが、AIが読む手がかりになる。状態(status)が引き出しでいちばん効く
- Claude Code を Vault のフォルダで開けば、自然な言葉で「あの判断を出して」と聞くだけで関連メモごと返ってくる
- 全部作り直さない。次の1ノートに10分、命名と情報欄をつけるところから始める
