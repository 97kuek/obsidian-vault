
## 仮想メモリ

### なぜ仮想化するのか

- ひとつは**仮想化**＝各プロセスに「大きなアドレス空間を持っているように見せる」こと
- もうひとつは、その**実現方法**として、MMUで仮想記憶を物理メモリへマッピングし、さらにオンデマンドページングによって「物理メモリより多くのメモリ」をアプリに使わせること

### 仮想アドレス空間と物理アドレス空間

- **仮想アドレス空間（Virtual Address Space）**：各プロセスが**それぞれ独自に**持つ。どのプロセスも「0番地〜0xffffffff」という同じ見かけのアドレスを使える。
- **物理アドレス空間（Physical Address Space）**：コンピュータに**1つだけ**存在する、実際のRAM
- **MMU**：仮想アドレス（VA）を物理アドレス（PA）へ変換する

- App A も App B も App C も、それぞれの「0番地から始まる仮想空間」を持ち、MMU がそれを物理メモリ上のバラバラの場所へ振り分ける（`PA = f(Pi, VA)`、Pi はプロセス番号）
- これによって**プロセス同士が互いのメモリを壊せない**という重要な性質が生まれる
- MMU は CPU が提供する機能で、**カーネル空間からのみアクセス可能**
- OS は MMU のレジスタを書き換えることで「今どのプロセスの仮想空間を使うか」を切り替える

### フラグメンテーション

- プロセスを**連続した領域**にそのまま置くと、Process A/B/C が並んだ後に B が終了して隙間が空いても、その隙間が新しいプロセスより小さければ**合計の空きは足りているのに連続領域が取れず入らない」という事態が起きる
- これを**外部フラグメンテーション**という
- 解決策の1つが**コンパクション**（プロセスを詰め直して隙間を埋める）だが、大量のメモリコピーが発生して非現実的
### ページとページテーブル

- **メモリを固定長のページに区切る**（512B / 1K / 4K / 8K など）
- 仮想空間のページを、物理メモリの**好きな場所のページ**へ自由にマッピングする
- 連続して置く必要がなくなるので、外部フラグメンテーションが原理的に消える
- このマッピング表を**ページテーブル**という
- 各プロセスが自分のページテーブルを持ち、MMU レジスタが「今のプロセスのページテーブルの場所」を指す

### ページテーブルエントリ（PTE）の中身

- ページテーブルの1行（エントリ）には、物理ページ番号のほかに次のビットが入る
- これが後の最適化すべての土台になる

- **Valid bit (V)**：そのページが今、物理メモリに有効に存在するか
- **Dirty bit (D)**：そのページが書き換えられたか（追い出すとき書き戻しが必要か判断）
- **Reference bit (R)**：最近アクセスされたか（追い出し候補の判断に使う）
- **Protection bit**：Read / Write / Execute の許可

### ページテーブルをどこに置くか／TLB

ページテーブル自体の置き場所には一長一短があります。**CPUレジスタ**は速いが数が限られ、プロセスが増えると退避が必要。**物理メモリ**に置くのが基本だが、テーブルが大きいと場所を食う。**仮想メモリ**に置くと「ページテーブルをマップするためのページテーブル」が要る、という入れ子問題が出ます。

そして大事なのが速度。毎回ページテーブルを引くと、メモリアクセスが2回（テーブル参照＋本体）になり遅い。そこで **TLB（Translation Lookaside Buffer）** という変換結果のキャッシュを CPU 内に置きます（`CPU → TLB → ページテーブル`）。

ただし TLB には注意点があり、**ページテーブルを書き換えても TLB には自動反映されない**ため、OS が明示的に **TLBフラッシュ**する必要があります。フラッシュが要る場面は、I/Oデバイスのマップ/アンマップ（Memory Mapped I/O）、仮想領域のマップ/アンマップ、そして**プロセスのコンテキストスイッチ**時（仮想アドレスはプロセスごとに違うため）です。

### ページテーブルが大きすぎる問題と改良

- 仮想記憶が大きくなるとページテーブルも巨大化する
- ページサイズ1Kのとき、1プロセス分のテーブルは：

- 32ビット：2^22 × エントリサイズ ＝ 4Mバイト × pes
- 64ビット：2^54 × エントリサイズ ＝ 約1.8×10^16 × pes ＝ **天文学的サイズ**

- このままでは破綻するので、3つの改良が示される

#### (a) 階層化ページテーブル

- 仮想アドレスを「上位ビット／中位ビット／オフセット」に分け、ページテーブルを**ツリー状の段**にする
- 利点は、**実際に使われていない部分の下位テーブルを作らなくてよい**こと
- プログラムが実際に使うのはコード・データ・スタックの周辺だけなので、テーブル全体をフルに用意する必要がなくなり、メモリを大幅に節約できる

#### (b) セグメントアーキテクチャ

- 空間を「使用中／未使用」の**セグメント**単位で管理し
- セグメントテーブル経由でページテーブルを引く

#### (c) インバーテッドページテーブル

- 発想を逆転させ、「仮想ページごと」ではなく**物理ページごと**に1エントリだけ持つ
- エントリには（プロセスID, 仮想ページ番号, 物理ページ番号）を入れ、`エントリ番号 = hash(プロセスID, 仮想ページ番号)` というハッシュで引く
- テーブルサイズが**物理メモリ量に比例**して済むので、仮想空間がいくら大きくてもテーブルは肥大しない

### 内部フラグメンテーション

- ページは固定長なので、Code 100B / Data 20B / Stack 10B（合計130B）しか使わなくても、それぞれ別ページに置くと **1024×3 = 3072B** を占有する
- ページ内の使い残しがムダにな
- これが**内部フラグメンテーション**
- 外部フラグメンテーションと引き換えに、こちらが発生する

### ページフォールト

- **ページフォールト**とは、アプリがアクセスしたページの PTE の **Valid ビットが Off** だったときに発生する例外
- 発生は**ハードウェア**が行い、処理は**OS**が担当する
- OS は適切なページを割り当てて Valid ビットを On にし、アプリは何事もなかったように続行できる
- 保護ポリシ違反でも発生し、これは**プロテクションフォールト**と呼ばれる

**ハンドラの呼ばれ方**：CPU はページフォールト用の特別なレジスタを持ち、OS はそこに「フォールト時に呼ぶ関数のアドレス」を登録しておきます。フォールトが起きると CPU は実行中の処理を中断し、その関数へジャンプします。


この「**アクセスされるまで本当の処理を遅らせる（遅延評価／Lazy Evaluation）**」のが、以下すべての最適化の共通原理です。

### ページフォールトを使った3つの最適化

#### (a) On Demand Page Allocation（オンデマンド割り当て）

- `malloc` やスタック拡張でメモリを「確保」しても、**実際にアクセスされるまで物理ページを割り当てない**
- PTE を `V=0` にしておき、初めて触られたときのフォールトで初めて物理ページを与える
-プログラムが本当に使うページだけが割り当てられるので、**大きなスタックを宣言してもメモリのムダにならない**

#### (b) Copy On Write（COW）

- `fork()` でプロセスを複製するとき、コード・データを丸ごとコピーするのは非常に高コスト
- そこで、最初は**親子で同じ物理ページを共有**し、PTE を **Read Only** にしておく
- どちらかが**書き込もうとした瞬間**にフォールトが起き、そのとき初めて**そのページだけを実コピー**して、書き込む側に書き込み可能（RW）な独立ページを与える
- `fork`直後は両者RWを共有 → いや、正確には共有ページをRead Onlyにマークし、変更が入ったページのみコピーして RW にする。**書き換えられないページは最後まで共有されたまま**なので、コピーコストを実際に変更が起きた分だけに抑えられます。

#### (c) Demand Paging（デマンドページング）

- 物理メモリより大きなプログラムを動かすための仕組み
- ページを最初は**ハードディスク**に置き、必要になったページだけを物理メモリへ運び込む
- ここで**3つのポリシ**が登場する
	- **ページフェッチポリシ**：いつディスクから物理メモリへコピーするか
	- **ページリプレースメントポリシ**：物理メモリが満杯のとき、どのページをスワップへ追い出すか
	- **ページアロケーションポリシ**：物理メモリ上のどのページを割り当てるか

### ページフェッチポリシとフリーページ管理

**いつ運ぶか**には2方式：

- **オンデマンド（On Demand）**：アクセスされてから運ぶ
- **プリフェッチ（Prefetch）**：先読みして前もって運ぶ

ページの種類は、コードエリア／データエリア（Read Only か Read/Write）／スタックや動的に割り当てた領域、に分類されます。

**フリーページの管理**：未使用ページは**リスト**として繋いで管理し、新しいページが必要になったらリストの先頭から割り当てます（ゼロクリア済みページが必要なとき、ファイル/ディスクからページを読み込むとき、など）。

### ページリプレースメント

- 物理メモリが満杯のとき、
	1. 追い出す物理ページを**選択**
	2. 必要なら**スワップディスクへコピー**（**Dirty なページだけ書き戻せばよい**。書き換えていないページは元がディスクにあるので破棄でOK）
	3. PTE の Valid を 1→0 に**変更**

- 問題は「**どのページを選ぶか**」。
	- **FIFO（First-In First-Out）**：最初にフェッチしたページから追い出す。単純だが、よく使うページも順番が来れば追い出してしまう。
	- **LRU（Least Recently Used）**：最近もっとも使われていないページを追い出す。**理想に近い**が、実装が重い ── アクセスのたびにキュー（アクセス時刻順のリスト）を並べ替える必要があり、その**オーバヘッドが現実的でない**。
	- **理想的なポリシ**：「将来もっとも使われないページ」を追い出す。しかし未来は読めないので**実装困難**。

### クロックアルゴリズム（LRUの近似）

- LRU を安く近似する実用解が**クロックアルゴリズム**
- ページを円環状に並べ、針が回りながら**参照ビット**を見ていく
	1. 針は一定時間ごとにページをスキャンする
	2. スキャン時、**R が ON ならば R を OFF にする**（猶予を1回与える）
	3. **R が OFF のページを追い出し候補**とする
	4. よく参照されるページは、2回目のスキャンが来る前にまた R が ON になっているので、生き残る

- つまり「最近触られていないページ＝Rがずっとoffのまま」を追い出す

**問題点**：メモリ量が非常に多くなると、針が一周する間にほぼ全ページが再アクセスされて**全部の R が ON**になってしまい、近似が機能しなくなります。改良版では「R=0 のとき追い出し候補にする針」と「R=1 のとき R=0 に落とす針」の**2本の針**を使い、間隔を調整して機能させます。


## デバイス管理

### なぜデバイス管理が必要なのか

#### I/Oアーキテクチャ

- CPU・メモリ・グラフィックメモリが**バス**でつながり、その先に**バスコントローラ**を介してデバイスがぶら下がっている
- CPUとデバイスのやり取りには、
	- **I/Oポート**（データのやり取り口）
	- **割り込み**（デバイス → CPU への「イベント発生したよ」という通知）の2つの経路がある、というのがこの図の要点

#### デバイスドライバ


```
アプリケーション
   ↓（標準インタフェース）
ファイルシステム／ネットワークシステム（カーネル内）
   ↓
デバイスドライバ
   ↓（デバイス独自プロトコル）
デバイス
```

- ポイントは、**アプリ側は標準インタフェースで統一されていて、デバイス側だけがデバイス独自のプロトコル**になっていること
- つまりデバイスドライバが「翻訳係」となり、デバイスごとの違いを吸収してくれる
- 新しいデバイスを使いたければ、対応するドライバを差し込めばよい、という設計

#### I/Oインタフェース

- Unixでは、デバイスも**ファイルのように扱う**
	- `/dev/device` のような名前でデバイスを表す
	- 各デバイスは**メジャー番号**と**マイナー番号**を持つ
	- カーネル内部ではこの番号でドライバを識別する（メジャー番号＝どの種類のドライバか、マイナー番号＝その中のどの個体か、というイメージ）

#### デバイスアクセスの方法

- デバイスとやり取りするには、2つの観点がある

#### (1) I/Oイベントの通知（デバイスの状態をどう知るか）

- エラー通知・データ利用可能通知・状態通知などを受け取る方法として、**ポーリング**と**割り込み**の2種類がある

#### (2) データの転送（実際にデータをどう運ぶか）

- **Read/Write**：通常の読み書き
- **プログラマブルI/O**：CPUが1バイトずつ自分で運ぶ方式
- **DMA（Direct Memory Access）**：CPUを介さず、デバイスが直接メモリへデータを書き込む方式。大量データを運ぶときCPUの負担を大きく減らせます

#### デバイスのアクセス：2つの方式

- デバイスのレジスタにアクセスする方法も2通り

|方式|しくみ|アクセス手段|
|---|---|---|
|**I/Oポート方式**|メモリ空間とは別に「I/O空間」があり、各デバイスにポートを割り当てる|専用命令（x86なら `in` / `out`）が必要|
|**Memory Mapped I/O**|デバイスの一部を「メモリの一部」として見せる|普通のメモリアクセス命令でOK（ただしアクセスコストは異なる）|

### ポーリング（スライド12・13）

**ポーリング**は、「準備できた？まだ？」と**CPUが繰り返し確認し続ける**方式です。

```c
while(flagPort != READY);   // 準備フラグが立つまでひたすら待つ
mem = dataPort;             // 準備できたらデータを読む
```

この方法はシンプルですが、**待っている間ずっとCPUを使い続けるので無駄が大きい**のが欠点です。スライドでは「他の処理の合間にチェックする」工夫も紹介されていますが、それをやるとプログラムの構造が複雑で見通しが悪くなる、と指摘されています。

### 割り込み（スライド14）

ポーリングの無駄を解決するのが**割り込み**です。考え方はこうです：

1. デバイスドライバはI/Oを設定したあと、`wait(&device)` で**眠って（CPUを手放して）**他の処理に譲る
2. デバイスの処理が終わると、デバイスがCPUに**割り込み**をかける
3. **割り込みハンドラ**が起動し、`wakeup(&device)` で眠っていた処理を起こす

```
デバイスドライバ側          割り込みハンドラ側
I/Oポートの設定
wait(&device);   ←眠る    interrupt_handler() {
                              wakeup(&device);   ←起こす
                           }
```

こうすれば、**待っている間CPUは他の仕事ができる**ので、ポーリングのような無駄使いが起きません。これが現代のOSで割り込みが広く使われる理由です。

## ファイルシステム

### ファイルシステムが解決する2つの問題

- ファイルシステムの仕事は、突き詰めると**2つの名前解決**
	1. **ファイル名の変換**：人間が使う文字列の名前を、内部のIDに変換する 例：`/a/b/c` → `101020`
	2. **ファイル内ポインタの変換**：ファイル内の「論理アドレス（先頭から何バイト目）」を、ストレージ上の「物理アドレス」に変換する
- ストレージは、メモリと同じように `0` から `0xffffff` まで続く**リニアなアドレス空間**として捉える

#### 論理アドレス→物理アドレスのマッピング方式

- ファイルの中身がディスク上のどこにあるかを管理する方式が3つ
	- **連続割り当て**：各ファイルの「先頭アドレス」と「ファイルサイズ」を記録する。シンプルだが、空き領域の断片化に弱い
	- **リンク方式**：「先頭アドレス」＋各ブロックに「次のブロックへのポインタ」を持たせ、数珠つなぎにする。途中のブロックへ飛びにくい
	- **ディレクトリ（インデックス）方式**：ブロックの位置を一覧表で管理する

#### ディレクトリの構造

- ファイルは**木構造**で整理される

```
/
├── bin
├── usr ── local ── bin ── ls, netscape
└── home ── tatsuo, reikona
```

#### UnixファイルシステムのAPI

デバイスのときと同じAPIが、ファイルにもそのまま使えます。

```c
fd = open(file, mode);
read/write(fd, buf, size);
lseek(fd, offset, mode);    // ファイルポインタの位置を移動
fstat(fd, statbuf);         // ファイルの情報を取得
close(fd);
```

`lseek` は「ファイルポインタ（次に読み書きする位置）」を動かす命令です。

### Unixファイルシステムの全体像（スライド26）

ここがファイルシステムの核心で、登場人物が多いので整理します。

```
プロセスのデータ構造
   ↓
ファイルのデータ構造（オフセットを持つ）
   ↓
i-nodeテーブル ──→ i-node ──→ ファイルブロック（実データ）

ストレージ上には：
スーパブロック ／ i-nodeリスト ／ ファイルブロック
```

それぞれの役割を順に見ていきましょう。

#### i-node（アイノード）とは

- **i-node** は、1つのファイルにつき1つ存在する「管理情報の塊」
- ファイルの中身そのものではなく、ファイルに関する情報を持っている
	- 所有者
	- アクセス権（誰が読み書きできるか）
	- アクセスタイム・変更時間
	- そして**実データの場所を指す10個のエントリ**（ファイルブロックへのポインタ）
- つまり、「ファイル名」ではなく「i-node番号」がファイルの本当の正体

#### ファイルデスクリプタとファイルデータ構造

- プロセスがファイルを開くと、`fd`（ファイルデスクリプタ＝0, 1, 4…のような番号）が返される
- この番号と実データの間に、**オフセット（今ファイルのどこを読んでいるか）を保持するファイルデータ構造**が挟まる
- この仕組みのおかげで、**プロセスAとプロセスBが同じファイルを開いても、それぞれ別々のオフセット**を持てる
- 逆に、共有させることもできる、という柔軟性がここで生まれる

#### スーパブロックとフリーブロック管理

- **スーパブロック**：そのファイルシステム全体の属性（ブロックサイズなど）を記録した、いわば「目次・台帳」
- **フリーブロック管理**：空きブロックを**ビットマップ**（0/1の並び）で管理し、「どこが空いていてどこが使用中か」を高速に判定する

#### バッファキャッシュ

- ディスクアクセスはメモリに比べて圧倒的に遅いので、**一度読んだブロックはメモリ上にキャッシュとして残しておく**
	- よく使うブロックは、2回目以降ディスクへアクセスせずに済む → 高速化
	- キャッシュがいっぱいになったら、**LRU**アルゴリズムで入れ替える

#### ディレクトリ管理

- ディレクトリの実体も、実は**名前とi-node番号の対応表が書かれた特殊なファイル**
- 例えばルートディレクトリの中身は：

```
名前        i-node番号
.       →   2     （自分自身）
..      →   2     （親、ルートなので自分）
usr     →   3
bin     →   10
```

- `/usr/local` をたどるときは、「ルートのi-node → usr（i-node 3）の中身を見る → local を探す…」と、**名前とi-node番号を1段ずつたどっていく**ことでファイルにたどり着く
- `.`（自分）と `..`（親）も必ず登録されているのがポイント

#### ファイルシステムのマウント

- 複数のディスクを1つの木構造にまとめる仕組みが**マウント**
- 実装としては、**マウントテーブル**で対応関係を管理し、マウントポイントになっているディレクトリの**i-nodeに「マウントフラグ」が立っている**ので、そこに来たら別のディスクへ切り替える、という仕組み
- これによりユーザーは複数のディスクを意識せず、ひとつながりのディレクトリツリーとして扱える
