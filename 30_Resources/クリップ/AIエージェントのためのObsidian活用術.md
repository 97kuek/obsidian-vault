# AIエージェントのためのObsidian活用術

Xクリップを出典として保管したノート。要点は自分の言葉で、原文はそのまま下部に残す。

## 要点（自分の学び）

- 記事の主張は「Vaultを"しまう設計"から"AIが引き出す設計"へ作り直す」。
- ノートを引き出せないのはAIの能力ではなく、Vaultが**保存した瞬間**に最適化され「いつ・何の・どの状態のノートか」という手がかりを持たないから。
- AIがノートを引くとき必ず使う手がかりは4つ。**種類(type)・時期(when)・主題(topic)・状態(status)**。
- ファイル名を「**日付-種類-主題**」の型にすると、AIは開く前に type / when / topic の3つを判別できる。ツールやモデルが変わってもファイル名は資産として残る。
- ファイル名に向かない **status** はフロントマター（プロパティ）で持たせる。引き出しで最も効くのが status（「進行中だけ」「完了を除く」の絞り込み）。
- タグは意味で絞る補助。種類別に接頭辞を付け、複数形で書き、5ノート以上で使う見込みのものだけ作る。
- 全部を作り直さない。**次の1ノートから**命名とプロパティを付け始めるのが現実解。

### このVaultとの接続

## 原文（@ObsidianOtaku, 2026-05-29）

『自分の代わりに仕事をしてくれるAI社員を作りたい』——こういう声は多いが、実際にはマイクロマネジメントをして手離れが進まない結末が大半。原因の多くは、そもそもAI社員を仕事を任せられる状態に仕上げていないこと。特に前提情報やコンテキストを渡す記憶設計で詰まる人が多い。逆にここを解決できれば、自社の仕事に合わせてチューニングし、専属のAI社員を作れる。

半年前に決めた方針と、その後どう判断が積み上がったかを1本の流れにまとめたい。手元の Obsidian には会議メモも議事録も数字も書いてあるはず。Claude Code に Vault を渡して「この案件について、過去に決めたことを時系列で出して」と頼んでも、返るのは関係しそうなノートを浅くまとめた文章だけ。肝心の「いつ何を決めたか」は拾えない。検索しても似た言葉のノートが何十件も並ぶだけ。引き出せないノートは、書いていないのと同じ。原因は「AIがまだ賢くないから」ではなく、Vault が「しまうため」に作られ「AIが引き出すため」に設計されていないこと。

### ① AIに渡しても要約しか返ってこないObsidian
新規事業の検討を半年続け、市場調査・競合価格分析・役員フィードバック・ピボットの議事録が全部 Obsidian にあるとする。「新規事業の意思決定を時系列で出して」と頼んでも、返るのは「市場調査や競合分析が含まれています」という目次のような要約。欲しいのは「3月にこの方針を決め、4月の競合分析を受けて5月に修正した」という筋。同じ Claude でも引き出せる Vault と引き出せない Vault がある。差はノートの中身ではなく「外側」——AIが「いつ・何について・どの状態か」を判断する手がかりが用意されているか。蓄積する装置（フォルダ・CLAUDE.md・夜間自動取込）は持っていても、蓄積と引き出しは別の設計。

### ② しまう設計と引き出す設計は別物
ほとんどの Vault は「保存する瞬間」に合わせて作られている。保存時に「ここでいい」と思った場所は、半年後に取り出すときのヒントにならない。「アイデア」フォルダに入れた着想が、後で「アイデア」か「案件」か「議事録」か思い出せない。引き出すための設計は評価軸が違う。フォルダをきれいに分けることが目的ではなく、「将来この情報が必要になったとき、自分とAIは何を手がかりに探すか」を先に決め、その手がかりをノートに持たせること。トップレベルのフォルダは5〜8個にまとめる基本は前提。主役はフォルダではなく、ノート1枚ずつが持つ「AIが読む手がかり」。

### ③ AIを第一の読み手に置く
設計の主語を「人間が探しやすいか」から「AIエージェントが引き出せるか」へ変える。AIが正確に引き出せると、人間が「探す」行為そのものがなくなる。AIが読む手がかりは、人間が貼る整理ラベル（色・場所）とは別物で、ノートに書かれた構造化情報。だから「AIが何を手がかりに引くのか」を具体的に知る必要がある。

### ④ AIが必ず使う4つの手がかり
将来ノートを引き出すとき、必ず手にしている手がかりは4つ：**種類(type)・時期(when)・主題(topic)・状態(status)**。「進行中の(status)、新規事業案件で(topic)、先月以降の(when)、意思決定(type)を出して」は、この4つの組み合わせ。ノートにこの4つが記録されていればAIはそのまま答えられる。なければ本文を一字ずつ読んで推測するしかなく、浅い要約しか返せない。

### ⑤ ファイル名だけで中身が分かる命名の型
4つのうち最も手軽なのがファイル名。**日付から始める型**を推奨。日付を先頭に置くと、自動で時系列に並び、最近のものが上に来て、正確な名前を思い出せなくても時期から絞れ、同主題の別日ノートでも名前がぶつからない。種類と主題も入れると、本文を読まずに「新規事業の会議録だ」と判別できる。これで when / type / topic の3つがファイル名だけで埋まる。ツールやモデルが変わってもファイル名は残る資産。残る1つ status は日々変わるのでファイル名に向かない。

### ⑥ AIが読むメタデータを設計する
ファイル名の外側で、ノート冒頭の情報欄（`---` で挟むプロパティ）にAIが読む構造化情報を持たせる。type / date(when) / topic はファイル名と重なるが、情報欄にも持たせると本文を開いたとき構造として読める。そしてファイル名に向かなかった **status** をここで持たせる。引き出しで最も効くのが status（「進行中だけ」「完了を除く」）。タグも意味で絞る手がかり。3種類に分けて接頭辞を付けると人もAIも「何で絞っているか」が分かる。`tags:` はまとめて並べ、1つでも複数形で書くのが標準。増やしすぎると絞れない。5ノート以上で使う見込みのものだけ作る。

### ⑦ 探す行為そのものを消す
Obsidian の Vault は中身を見ればテキストファイルの集まり。Claude Code をその Vault のフォルダで開くだけで全ノートを読める。特別なプラグインは不要。あとは自然な言葉で頼む。AIはファイル名の日付・種類で時期と種類を絞り、情報欄の status で状態を絞り、タグで主題を絞り、本文で関連を判断する。①で浅い要約しか返さなかったAIが、手がかりを持たせた後は「3月に方針を決め、4月の競合分析を受けて5月に修正した」という筋を関連ノートごと返す。違いはAIの賢さではなく、ノートが手がかりを持っているかどうか。人間がやっていた「どのフォルダか思い出す」「検索語を変えて試す」がまるごと消え、「探す」のではなく「聞けば出てくる」になる。

### ⑧ 次の1ノートから始める
数百ページを全部作り直す必要はない。作り直しは最も挫折しやすい。次に作る1ノートから始める。この型で作ったノートはその日からAIが引ける。1週間後には今週分が「あの件のメモを出して」で返る。過去ノートは必要になって開いたときに、ついでに名前と情報欄を直せば十分。

### まとめ
- 引き出せないのはAIの能力ではなく、Vault が「しまう設計」で止まっているから。引き出す設計は「将来何を手がかりに探すか」を先に決める。
- 設計の主語を人間からAIに移すと、人間が「探す」行為そのものが消える。
- AIが引くときの手がかりは4つ：種類 / 時期 / 主題 / 状態。
- ファイル名を「日付-種類-主題」にすると、AIは開く前に中身を判別できる。
- 情報欄（プロパティ）とタグが手がかりになる。引き出しで最も効くのは status。
- Claude Code を Vault のフォルダで開けば、自然な言葉で「あの判断を出して」と聞くだけで関連メモごと返ってくる。
- 全部作り直さない。次の1ノートに10分、命名と情報欄を付けるところから始める。
