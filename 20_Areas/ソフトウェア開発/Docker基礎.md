# Docker基礎

## Dockerとは

- Dockerとは、アプリケーションとその実行に必要なものを**コンテナイメージ**としてまとめ、異なる環境でも動かしやすくするための仕組みである。
  - アプリケーション本体
  - ランタイム
  - ライブラリ
  - OS上のコマンドや設定ファイル
- Dockerを使う主な目的は、アプリケーションを**構築・共有・実行しやすくすること**である。
- 「自分のPCでは動くが、別の人のPCでは動かない」という環境差の問題を減らせる。
- 開発環境、テスト環境、本番環境で同じイメージを使うことで、実行環境の再現性を高められる。
- Webアプリ、データベース、キャッシュ、開発ツールなどを、ホストOSへ直接インストールせずに試しやすくなる。

## まず理解する全体像

- Dockerの基本的な流れは次のとおりである。
  1. `Dockerfile` に、実行環境の作り方を書く。
  2. `docker build` で、Dockerfileから**イメージ**を作る。
  3. `docker run` で、イメージから**コンテナ**を作成して起動する。
  4. 必要に応じて、ポート、環境変数、ネットワーク、ボリュームを接続する。
  5. 完成したイメージをレジストリへ保存し、他の環境で取得して実行する。
- 最も重要な関係は、**Dockerfile → イメージ → コンテナ**である。
  - Dockerfileは設計書である。
  - イメージは設計書から作った実行環境のひな型である。
  - コンテナはイメージをもとに実際に動いているプロセスである。
- 1つのイメージから、複数の独立したコンテナを作成できる。
- コンテナを削除しても、元になったイメージは削除されない。

## 仮想マシンとの違い

- 仮想マシンは、仮想化されたハードウェア上でゲストOS全体を動かす仕組みである。
- コンテナは、ホスト側のOSカーネルを利用しながら、プロセス、ネットワーク、ファイルシステムなどを隔離して動かす仕組みである。
- 一般にコンテナは仮想マシンより軽量で、起動が速い。
- ただし、コンテナは仮想マシンと同じものではなく、完全に独立した別OSが動いているわけではない。
- WindowsやmacOSのDocker Desktopでは、Linuxコンテナを動かすために内部で軽量なLinux仮想マシンを利用する。

## 基本要素

### Docker Engine

- Docker Engineは、イメージやコンテナを管理する中心部分である。
- 主に次の要素から構成される。
  - **Docker CLI**：`docker` コマンドを受け付けるクライアントである。
  - **Docker daemon**：イメージのビルドやコンテナの起動などを実際に処理する常駐プログラムである。
  - **Docker API**：CLIなどからdaemonへ処理を依頼するための窓口である。
- `docker run` を入力すると、CLIがdaemonへ依頼し、daemonが必要な処理を実行する。

### コンテナ

- **Container**とは、隔離された環境で動くアプリケーションの実体である。
- より正確には、イメージをもとに作られた、隔離されたプロセスである。
- コンテナごとに次のようなものを分けて扱える。
  - プロセス
  - ファイルシステム
  - ネットワーク
  - 環境変数
  - ユーザーや権限
- コンテナには主に次の状態がある。
  - **created**：作成済みだが、まだ起動していない状態である。
  - **running**：起動中の状態である。
  - **exited**：停止した状態である。
- コンテナの主プロセスが終了すると、原則としてコンテナも停止する。
- `docker stop` はコンテナを停止するだけであり、コンテナ自体は残る。
- `docker rm` は停止中のコンテナを削除する。
- コンテナを削除すると、永続化していないコンテナ内部の変更も失われる。

### コンテナイメージ

- **Container Image**とは、コンテナを作る元になる読み取り専用のテンプレートである。
- イメージには、アプリケーションの実行に必要なファイル、バイナリ、ライブラリ、設定などが含まれる。
- イメージは複数の**レイヤー**を重ねた構造になっている。
  - `FROM` でベースとなるレイヤーを用意する。
  - `COPY` や `RUN` などの命令によってレイヤーを追加する。
  - 変更のないレイヤーはキャッシュとして再利用できるため、ビルドを高速化できる。
- 作成済みのイメージ自体は変更せず、内容を変えたい場合は新しいイメージをビルドする。
- コンテナを起動すると、読み取り専用イメージの上に、そのコンテナ専用の書き込み可能な層が追加される。
- イメージ名は一般に `リポジトリ名:タグ` の形で表す。
  - 例：`python:3.13-slim`
  - 例：`mysql:8.4`
- タグを省略すると通常は `latest` が使われるが、`latest` は「必ず最新版である」ことを保証する名前ではない。
- 再現性を高めるには、`latest` ではなく、使用するバージョンを明示するほうがよい。

### イメージレジストリ

- **Image Registry**とは、コンテナイメージを保存・共有する場所である。
- Docker Hubは代表的な公開レジストリである。
- 組織内だけで使うプライベートレジストリを利用することもできる。
- 主な操作は次のとおりである。
  - `docker pull イメージ名:タグ`：レジストリからイメージを取得する。
  - `docker push イメージ名:タグ`：レジストリへイメージを送信する。
  - `docker login`：認証が必要なレジストリへログインする。

### Dockerfile

- **Dockerfile**とは、コンテナイメージを作るための設計書である。
- テキストファイルとして管理できるため、誰がビルドしても同じ環境を再現しやすい。
- 代表的な命令は次のとおりである。
  - `FROM`：ベースイメージを指定する。
  - `ARG`：ビルド時だけ使用する変数を定義する。
  - `ENV`：イメージやコンテナ内で使う環境変数を設定する。
  - `WORKDIR`：以降の命令や起動時に使う作業ディレクトリを指定する。
  - `COPY`：ビルド元のファイルをイメージ内へコピーする。
  - `RUN`：イメージのビルド中にコマンドを実行する。
  - `USER`：以降の命令やコンテナ起動時に使うユーザーを指定する。
  - `EXPOSE`：アプリケーションが待ち受けるポートを記録する。
  - `CMD`：コンテナ起動時のデフォルトコマンドを指定する。
  - `ENTRYPOINT`：コンテナを特定の実行プログラムとして扱う場合に、その本体を指定する。
- `EXPOSE` はポートの用途を示すメタデータであり、ホストへ実際に公開する命令ではない。
- ホストへポートを公開するには、起動時に `docker run -p` などを指定する必要がある。

```dockerfile
FROM python:3.13-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000
CMD ["python", "app.py"]
```

- このDockerfileは次の順番で環境を作る。
  1. Pythonが入った既存イメージを土台にする。
  2. `/app` を作業場所にする。
  3. 依存関係の一覧をコピーしてインストールする。
  4. アプリケーションのソースコードをコピーする。
  5. アプリケーションが8000番ポートを使うことを示す。
  6. コンテナ起動時に `python app.py` を実行する。
- 依存関係のファイルをソースコードより先にコピーすると、依存関係が変わっていない場合にビルドキャッシュを利用しやすい。

### ネットワーク

- **Docker Network**とは、コンテナ同士やコンテナと外部を通信させる仕組みである。
- コンテナ内のポートと、ホスト側へ公開するポートは別物である。
- `-p ホスト側ポート:コンテナ側ポート` の形で対応付ける。
  - 例：`-p 33066:3306`
  - ホストの33066番ポートへ接続すると、コンテナ内のMySQLが待ち受ける3306番ポートへ転送される。
- 同じDockerネットワークに参加するコンテナ同士は、通常、コンテナ名やComposeのサービス名で通信できる。
- 例えば、`app` コンテナから `db` コンテナへ接続する場合、接続先を `localhost` ではなく `db` とする。
- コンテナ内の `localhost` は、そのコンテナ自身を指す。
- `docker network create ネットワーク名` でユーザー定義ネットワークを作成できる。

### ファイルシステムとマウント

- コンテナは独立した書き込み可能なファイル層を持つ。
- コンテナを停止しただけであれば、その層のデータは残る。
- コンテナを削除すると、その層にしか存在しないデータは失われる。
- データベースのデータや、コンテナを作り直しても残したいファイルは、マウントによってコンテナの外へ保存する必要がある。
- 主なマウント方法は次の3種類である。
  - **volume**：Dockerが管理する保存領域を使う。
  - **bind mount**：ホスト側の任意のファイルやディレクトリを使う。
  - **tmpfs mount**：ホストのメモリ上に一時保存し、永続化しない。

#### volume

- Dockerが保存場所を管理する。
- データベースなど、コンテナが生成するデータの永続化に適している。
- ホスト側の具体的なディレクトリ構成に依存しにくい。

```powershell
docker volume create mysql-data
docker run -d --name db -v mysql-data:/var/lib/mysql mysql:8.4
```

#### bind mount

- ホスト側のファイルやディレクトリを、そのままコンテナ内へ見せる。
- ソースコードを編集しながらコンテナ内で実行する開発用途に適している。
- ホスト側で行った変更が、コンテナ側にも即座に反映される。
- コンテナ側からの書き込みがホスト側にも影響するため、マウント先を誤らないよう注意が必要である。

```powershell
docker run --rm -it --mount type=bind,src="${PWD}",dst=/work python:3.13-slim
```

#### tmpfs mount

- データをメモリ上に保存する。
- コンテナ停止後に残す必要がない一時データに向いている。
- 機密情報をディスクへ残したくない場合にも候補となるが、メモリ容量を消費する。

## よく使うコマンド

### イメージを操作する

- `docker pull nginx:1.27`：イメージを取得する。
- `docker image ls`：ローカルにあるイメージを一覧表示する。
- `docker build -t my-app:1.0 .`：現在のディレクトリをビルドコンテキストとしてイメージを作る。
- `docker image rm my-app:1.0`：不要なイメージを削除する。

### コンテナを作成・起動する

- `docker run イメージ名` は、必要に応じたイメージ取得、コンテナ作成、コンテナ起動をまとめて行う。
- よく使うオプションは次のとおりである。
  - `-d`：バックグラウンドで起動する。
  - `--name 名前`：コンテナ名を指定する。
  - `-e キー=値`：環境変数を渡す。
  - `-p ホスト:コンテナ`：ポートを公開する。
  - `-v`：volumeまたはbind mountを接続する。
  - `--mount`：種類や接続元、接続先を明示してマウントする。
  - `--rm`：停止時にコンテナを自動削除する。
  - `-it`：対話的に端末を操作する。

```powershell
docker run -d --name web -p 8080:80 nginx:1.27
```

- この例では、次の処理が行われる。
  - `nginx:1.27` イメージがなければ取得する。
  - `web` という名前のコンテナを作る。
  - バックグラウンドでNginxを起動する。
  - ホストの `8080` 番ポートをコンテナの `80` 番ポートへ接続する。

### 状態を確認する

- `docker ps`：起動中のコンテナを表示する。
- `docker ps -a`：停止中を含むすべてのコンテナを表示する。
- `docker logs コンテナ名`：標準出力と標準エラーのログを表示する。
- `docker logs -f コンテナ名`：ログを継続して表示する。
- `docker inspect コンテナ名`：コンテナの詳細設定をJSON形式で表示する。
- `docker stats`：CPUやメモリなどの使用量を表示する。

### 起動中のコンテナを操作する

- `docker exec コンテナ名 コマンド`：起動中のコンテナ内で追加のコマンドを実行する。
- `docker exec -it コンテナ名 sh`：コンテナ内のシェルへ入る。
- `bash` が入っていない軽量イメージもあるため、その場合は `sh` を使う。
- `docker cp コンテナ名:/コンテナ内のパス ホスト側のパス`：コンテナからホストへファイルをコピーする。
- `docker cp ホスト側のパス コンテナ名:/コンテナ内のパス`：ホストからコンテナへファイルをコピーする。

### 停止・削除する

- `docker stop コンテナ名`：コンテナを正常停止させる。
- `docker start コンテナ名`：停止中の既存コンテナを再起動する。
- `docker restart コンテナ名`：コンテナを再起動する。
- `docker rm コンテナ名`：停止中のコンテナを削除する。
- `docker rm -f コンテナ名`：起動中のコンテナを強制停止して削除するため、通常は先に `docker stop` を使う。

## Docker Compose

- **Docker Compose**とは、複数コンテナからなるアプリケーションの構成を `compose.yaml` にまとめて管理する仕組みである。
- Webアプリ、データベース、キャッシュなどを、それぞれ別のコンテナとして定義できる。
- Composeでは次の項目を宣言的に管理できる。
  - サービス
  - 使用するイメージまたはDockerfile
  - 環境変数
  - ポート
  - ボリューム
  - ネットワーク
  - 起動時の依存関係
- DockerfileとComposeファイルの役割は異なる。
  - **Dockerfile**：1つのコンテナイメージをどのように作るかを定義する。
  - **compose.yaml**：作成済みまたはビルドするイメージを、どのような設定で組み合わせて動かすかを定義する。

```yaml
services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      DB_HOST: db
    depends_on:
      - db

  db:
    image: mysql:8.4
    environment:
      MYSQL_DATABASE: app
      MYSQL_USER: app
      MYSQL_PASSWORD: change-me
      MYSQL_ROOT_PASSWORD: change-root-password
    volumes:
      - db-data:/var/lib/mysql

volumes:
  db-data:
```

- この構成では、次の2サービスを定義している。
  - `app`：現在のディレクトリにあるDockerfileからビルドするWebアプリである。
  - `db`：MySQLイメージを使うデータベースである。
- `app` からデータベースへ接続するときのホスト名は、Composeのサービス名である `db` を使う。
- `depends_on` は起動順序を制御するが、データベースが接続可能になるまでの準備完了を必ず保証するものではない。
- 確実に待つ必要がある場合は、`healthcheck` やアプリ側の再試行処理を組み合わせる。
- 実際のパスワードをこの例のようにComposeファイルへ直接書かず、開発環境では `.env` などから渡し、本番環境ではシークレット管理機能を使う。

### Composeでよく使うコマンド

- `docker compose up`：サービスを作成して起動し、ログを画面に表示する。
- `docker compose up -d`：サービスをバックグラウンドで起動する。
- `docker compose up -d --build`：必要なイメージを再ビルドしてから起動する。
- `docker compose ps`：サービスの状態を確認する。
- `docker compose logs app`：`app` サービスのログを確認する。
- `docker compose logs -f`：すべてのサービスのログを継続表示する。
- `docker compose exec app sh`：`app` サービスの起動中コンテナ内でシェルを開く。
- `docker compose stop`：コンテナを削除せずに停止する。
- `docker compose down`：Compose管理下のコンテナとネットワークを停止・削除する。
- `docker compose down -v`：コンテナとネットワークに加えて名前付きボリュームも削除する。
- `down -v` を使うとデータベースの保存データなども消えるため、実行前に対象を確認する必要がある。

## MySQLコンテナを使うときの要点

- MySQLのデータは、通常 `/var/lib/mysql` に保存される。
- コンテナを作り直してもデータを残すには、この場所へvolumeをマウントする。
- MySQL公式イメージでは、`/docker-entrypoint-initdb.d` にSQLファイルや対応するスクリプトをマウントすると、データディレクトリが空である**初回初期化時**に実行される。
- すでに初期化済みのvolumeを使っている場合、初期化用SQLを変更してコンテナを再起動しても自動では再実行されない。
- 初期化処理の失敗を調べる場合は、`docker logs` でMySQLのログを確認する。

## 初心者向けの実践手順

- 最初は、既存イメージをそのまま起動して、イメージとコンテナの関係を確認するとよい。

```powershell
docker run -d --name hello-nginx -p 8080:80 nginx:1.27
docker ps
docker logs hello-nginx
docker stop hello-nginx
docker rm hello-nginx
```

- 次に、自分のアプリケーション用Dockerfileを作り、ビルドと実行を試す。

```powershell
docker build -t my-app:dev .
docker run --rm -p 8000:8000 my-app:dev
```

- データベースなどを追加するときは、複数の長い `docker run` コマンドを手作業で管理するのではなく、Composeへ移行すると理解しやすい。
- 問題が起きた場合は、次の順番で確認する。
  1. `docker ps -a` でコンテナの状態を確認する。
  2. `docker logs コンテナ名` で起動失敗の理由を確認する。
  3. ポート番号、環境変数、マウント先を確認する。
  4. コンテナ同士の接続先に `localhost` を誤って使っていないか確認する。
  5. `docker inspect` で実際に適用された設定を確認する。

## よくある誤解

- **「イメージ」と「コンテナ」は同じではない。**
  - イメージはひな型であり、コンテナはその実行インスタンスである。
- **コンテナを停止しただけでは削除されない。**
  - `docker stop` の後でも `docker start` で再起動できる。
- **コンテナを削除すると、内部だけに保存したデータは消える。**
  - 残すべきデータにはvolumeやbind mountを使う。
- **`EXPOSE` だけではホストから接続できない。**
  - `docker run -p` やComposeの `ports` で公開する必要がある。
- **コンテナ内の `localhost` はホストPCや別コンテナではない。**
  - 同じコンテナ自身を指す。
- **`latest` は最新版を保証しない。**
  - 再現性が必要な場合はバージョンタグを固定する。
- **コンテナ内で手作業した変更は、環境構築手順にはならない。**
  - 再現したい変更はDockerfileやComposeファイルへ記述する。
- **Dockerを使うだけで安全になるわけではない。**
  - コンテナの隔離は有用だが、設定や権限、イメージに問題があれば危険である。

## セキュリティと運用上の注意

- 業務でDocker DesktopやDocker Hubを使う場合は、組織規模や用途に応じた最新の利用条件を公式情報で確認する。
- 信頼できる提供元のイメージを使い、不要に古いイメージを放置しない。
- イメージタグだけでなく、必要に応じてdigestで内容を固定する。
- 不要なポートを公開せず、外部へ公開する範囲を最小限にする。
- DBパスワード、APIキー、秘密鍵などの**秘密情報をイメージへ含めない**。
- Dockerfileの `ARG` や `ENV` に秘密情報を書いてビルドすると、イメージの履歴や設定に残る可能性がある。
- `.dockerignore` を使い、`.git`、`.env`、秘密鍵、ビルドに不要な大容量ファイルなどをビルドコンテキストから除外する。
- コンテナは可能な限りroot以外のユーザーで動かす。
- `--privileged` や過剰なLinux capabilityを安易に与えない。
- ホストの重要なディレクトリやDockerソケットを、安易にコンテナへマウントしない。
- 必要に応じてRootless modeや最小権限での実行を検討する。
- CPUアーキテクチャが異なると、同じイメージをそのまま実行できない場合がある。
- x86-64とarm64など複数環境へ対応する場合は、Docker Buildxによるマルチアーキテクチャイメージを検討する。

## 要点

- Dockerfileはイメージの作り方を記述する設計書である。
- イメージはコンテナを作るための読み取り専用のひな型である。
- コンテナはイメージから作られた、実際に動く隔離プロセスである。
- 永続データはコンテナ内部だけに置かず、volumeなどへ保存する。
- ホストから接続するにはポート公開が必要である。
- コンテナ同士は同じネットワーク上で、名前を使って通信できる。
- 複数コンテナの構成はDocker Composeで宣言的に管理する。
- 環境を再現するための変更は、手作業ではなくDockerfileやComposeファイルへ残す。
