# AI駆動開発

「人間が書く」開発から、「**人間が意図を伝え → AIが生成 → 人間がレビューする**」開発へ。仕事の重心が *コードを書く* から *指示してレビューする* に移る。

## 3つのレベル

| レベル | 内容 | 例 |
|---|---|---|
| L1 補完 | 行・関数単位で候補を出す | GitHub Copilot |
| L2 対話 | チャットで相談・生成 | ChatGPT, Gemini |
| L3 エージェント | タスク単位で自律実行 | Claude Code, Cursor |

## LLM とエージェントの違い

- **LLM**：入力→出力の1往復の生成エンジン。
- **エージェント**：そこに **Plan → Act → Observe** ループが加わる。「テストを書いて」に対し、コードベース探索→テスト作成→実行→失敗修正まで自走する。

## ツールとワークフロー

- CLI：Claude Code（多ファイル編集・計画力）／Codex CLI（高速・サンドボックス）／Gemini CLI（巨大コンテキスト）
- IDE：Cursor、Kiro（仕様駆動）など
- どのツールも「Webサーチ・MCP・ファイル/ターミナル操作・並列実行」は標準。**差がつくのは上に乗せるワークフロー設計**（Plan-Execute-Verify、`CLAUDE.md` / `AGENTS.md` での文脈永続化、仕様書駆動 など）。

## 実践ノウハウ

- **`CLAUDE.md` / `AGENTS.md`**：プロジェクト規約の入口。**薄く**保ち、大きな仕様は `docs/` や `spec/` に外出ししてポインタだけ張る。
- **仕様駆動開発**：Vibe Coding（ふんわり指示）の対極。**What（要件）→ How（設計）→ Steps（タスク分解）** を先に書く。Vibeは500行を超えると破綻しがち。
- 研究・実験コードでは、実装前に「データ仕様・評価指標・期待出力」を `spec.md` に書いてからAIに渡すと、目的とずれた実装が減る。
- 落とし穴：未検証のままマージ／機密をプロンプトに入れる はNG。こまめにcommit・TDD・重要ロジックは理解してから採用。
- スライド作成は **Marp**、図は **Mermaid** のように出力形式を明示すると、AIの生成結果が安定しやすい。
- 専門作業を繰り返すなら `SKILL.md` のような再利用可能な手順化が効く。単発プロンプトではなく、文脈から自動で呼び出せる作業単位にする。
- 価値が上がるスキル：**問題を正しく定義する力・設計力・レビュー力・ドメイン知識**。

## ツール・サービス候補

クリップ等で見つけた未検証ツールの台帳（1ツール1行）。試して定着したら本文へ昇格する。

- **MarkItDown**（Microsoft, OSS）— Word/PDF/Excel/PowerPoint を表・見出し構造を保ったまま Markdown 化。画像・音声の文字起こしにも対応。AIに資料を渡す前処理に。 出典: [@opensourcelab9](https://x.com/opensourcelab9/status/2066289580194791487)
- **サイト→DESIGN.md/SKILL.md 生成 Chrome拡張** — 任意サイトのスタイルを抽出して DESIGN.md / SKILL.md を自動生成（詳細未確認）。 出典: [@ClaudeCode_UT](https://x.com/ClaudeCode_UT/status/2066438059428819377)
