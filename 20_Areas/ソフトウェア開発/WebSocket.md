## そもそもなぜWebSocketが必要なのか

- HTTPは基本的に「クライアントが要求 → サーバーが応答」という一往復のモデル
- この構造では**サーバーから自発的にクライアントへデータを送れない**
- チャット、通知、株価のライブ更新、オンラインゲーム、共同編集のような「サーバー側で起きた変化を即座にクライアントへ届けたい」用途では、この制約が致命的になる
- WebSocket以前は、これをHTTPの枠内で無理やり実現していた
	- **ポーリング**: クライアントが定期的に「新しいデータある?」と聞き続ける。実装は楽だが、レイテンシがポーリング間隔に依存し、毎回HTTPヘッダのオーバーヘッドが乗るので無駄が多い。
	- **ロングポーリング**: クライアントがリクエストを送り、サーバーはデータが発生するまで応答を保留する。発生した瞬間に返し、クライアントはすぐ再接続する。レイテンシは改善するが、接続を握り続けるしオーバーヘッドも残る。
	- **Server-Sent Events (SSE)**: サーバー→クライアントの一方向プッシュをHTTP上で実現。プッシュには良いが双方向ではない。
- これらの「HTTPで頑張る」アプローチの限界を超えるために設計されたのが WebSocket
- **1本のTCPコネクション上で全二重(full-duplex)・持続的(persistent)な通信**を実現
- 最初にHTTPでハンドシェイクした後、そのコネクションをWebSocketプロトコルへ昇格させ、以降は両側がいつでも最小限のオーバーヘッドでメッセージを送り合えるシステム

## 接続のライフサイクル全体像

- WebSocketの通信は大きく3つのフェーズに分かれる
- HTTPで始まり、プロトコルを切り替え、その後は同じ1本のTCP接続上で双方向にやり取りし、最後にきれいに閉じる

### フェーズ1: ハンドシェイク(HTTP Upgrade)

- WebSocket接続は、必ず**通常のHTTP/1.1リクエスト**から始まる
- いきなり独自プロトコルで繋ぎにいくのではなく、既存のHTTPインフラを素通りできるように、まずHTTPの体裁で始める
- クライアントが送るのはこのようなリクエスト

```
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
```

- 各ヘッダの意味を分解すると、`Upgrade: websocket` と `Connection: Upgrade` が「このコネクションをWebSocketに切り替えたい」という意思表示
- `Sec-WebSocket-Version: 13` が現行のプロトコルバージョン指す
- `Sec-WebSocket-Key` は、クライアントが生成した16バイトのランダムな値をBase64エンコードしたもの
- サーバーが対応していれば、ステータスコード `101 Switching Protocols` を返す

```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

ここで `Sec-WebSocket-Accept` の計算が地味に大事です。サーバーは、クライアントから受け取った `Sec-WebSocket-Key` の文字列に、固定のマジックGUID `258EAFA5-E914-47DA-95CA-C5AB0DC85B11` を連結し、SHA-1ハッシュを取り、それをBase64エンコードして返します。クライアントは自分が送ったKeyから同じ計算をして一致を確認します。

これは暗号的なセキュリティ機構ではなく、「相手が本当にWebSocketを理解しているサーバーか」を確かめるサニティチェックです。たとえばキャッシュプロキシや、WebSocketを知らない古いサーバーが偶然 `101` を返してしまっても、この値を正しく計算できなければ接続は成立しません。誤接続を防ぐための仕掛けだと捉えてください。

`101` が返った時点で、**そのTCPコネクションはもうHTTPではなくWebSocketになります**。新しい接続を張り直すわけではなく、同じソケットをそのまま使い続けます。URLスキームは平文が `ws://`、TLS上が `wss://` です。本番では実質 `wss://` 一択になります。

## フェーズ2: フレーム単位のデータ交換

- ハンドシェイク後は、**フレーム**という単位で区切ってやり取りする
- TCPはバイトストリームでメッセージの境界を持たないので、`WebSocket`が自前で境界とメタ情報を持つフレーム形式を定義しているわけ
- フレームのヘッダの構造を見てみる

**1バイト目** — 最上位ビットが `FIN`(1ビット)で、これが「このフレームでメッセージが完結するか」を表します。1つのメッセージを複数フレームに分割(フラグメンテーション)でき、その途中のフレームは `FIN=0`、最後のフレームが `FIN=1` になります。続く `RSV1〜RSV3`(3ビット)は拡張用の予約領域で、`permessage-deflate`(メッセージ単位の圧縮)などの拡張で使われます。残りの4ビットが `opcode` で、フレームの種別を示します。

opcodeの主な値は次の通りです。`0x0` が継続フレーム(分割されたメッセージの続き)、`0x1` がテキスト(UTF-8)、`0x2` がバイナリ、`0x8` がクローズ、`0x9` がPing、`0xA` がPongです。`0x1`〜`0x2` がデータフレーム、`0x8`〜`0xA` が制御フレームと呼ばれます。

**2バイト目** — 最上位ビットが `MASK`(1ビット)、残り7ビットが `Payload len` です。ペイロード長の表現が少し変わっていて、7ビットの値が0〜125ならそれがそのまま長さ、値が126なら「続く2バイト(16ビット)が実際の長さ」、値が127なら「続く8バイト(64ビット)が実際の長さ」という3段構えになっています。小さなメッセージは2バイトのヘッダで済み、大きなメッセージだけ長さフィールドを拡張する、という省サイズ設計です。

その後に、`MASK=1` のときだけ**4バイトのマスキングキー**が入り、最後に**ペイロード本体**が続きます。

#### マスキングの意味

ここが初見で引っかかりやすい部分です。仕様上、**クライアント→サーバーのフレームは必ずマスクしなければならず、サーバー→クライアントのフレームはマスクしてはいけません**。マスクとは、4バイトのランダムな鍵をペイロードに対してXORする操作です。鍵はフレーム内に平文で同梱されるので、暗号化ではありません。受け取った側は同じ鍵でもう一度XORすれば元に戻せます。

ではなぜわざわざやるのか。理由はセキュリティで、**キャッシュ汚染攻撃(cache poisoning)の防止**です。マスクが無いと、ブラウザ上の悪意あるJavaScriptが、HTTPリクエストそっくりのバイト列をペイロードに仕込めてしまいます。途中のプロキシがそれをHTTPと誤解釈してキャッシュを汚染する、という攻撃が成立しうる。鍵ごとにバイト列がランダム化されると、攻撃者は「ワイヤー上に出る実際のバイト」を予測できなくなるため、この攻撃が封じられます。つまりマスクは中身を秘匿するためではなく、**バイト列を予測不能にする**ためのものです。

#### 制御フレーム — Ping/Pong とクローズ

`Ping`(0x9)と `Pong`(0xA)は、接続が生きているかの確認とキープアライブに使います。一方がPingを送ると、相手はPongを返す決まりです。TCP接続はNATのタイムアウトや経路の切断で**音もなく死ぬ**ことがあり、アプリ層でPing/Pongを回しておくと、死んだ接続を検知でき、また中間装置にアイドル接続を切られるのも防げます。

### フェーズ3: クローズハンドシェイク

接続を閉じるときも、いきなりTCPを切るのではなく作法があります。どちらか一方が `Close` フレーム(0x8)を送り、相手も `Close` フレームを返し、その後にTCPコネクションを閉じます。`Close` フレームにはステータスコードと理由を載せられます。代表的なコードは、正常終了の `1000`、サーバー停止などの `1001`、メッセージが大きすぎる `1009` などです。この往復によって「途中でぶつ切りにされたのか、合意の上で閉じたのか」を区別できます。

### 実装から見たWebSocket

- ブラウザ側のAPIは非常にシンプルで、ハンドシェイク・フレーミング・マスクはすべて内部で自動的に処理される

```javascript
const ws = new WebSocket('wss://example.com/chat');

ws.onopen  = () => ws.send('hello');           // 接続確立後に送信
ws.onmessage = (e) => console.log(e.data);     // 受信(テキストor Blob/ArrayBuffer)
ws.onclose = (e) => console.log(e.code, e.reason);
ws.onerror = (e) => console.error(e);
```

- サーバー側(Node.js)では `ws` ライブラリが定番

```javascript
import { WebSocketServer } from 'ws';
const wss = new WebSocketServer({ port: 8080 });

wss.on('connection', (socket) => {
  socket.on('message', (data) => socket.send(`echo: ${data}`));
  socket.on('close', () => { /* 後処理 */ });
});
```

ひとつ注意したいのが **Socket.IO** との混同です。Socket.IOはWebSocketの「上に乗っている」独自ライブラリで、自動再接続・ルーム・フォールバック(WebSocketが使えない環境でのロングポーリング)などを足したものです。**純粋なWebSocketとは互換性がない**ので、相手がSocket.IOサーバーなら素の `WebSocket` クライアントでは繋がりません。ここは面接でも突っ込まれやすい区別です。

### 実務で効いてくるポイント

スケーリングが最初の壁になります。WebSocket接続は**長時間生きてステートフル**なので、HTTPのようにリクエストごとに別サーバーへ振り分ける、という単純なロードバランシングができません。複数サーバーへ水平スケールするなら、スティッキーセッション(同じクライアントを同じサーバーへ固定)か、Redisのpub/subのような共有バックプレーンを挟んで、どのサーバーに繋がったクライアントにも配信できるようにします。

認証も癖があります。ブラウザの `WebSocket` APIはカスタムHTTPヘッダを自由に付けられないため、`Authorization` ヘッダでトークンを渡す定石が使えません。クエリ文字列にトークンを載せる、Cookieベースで認証する、あるいは接続後の最初のメッセージで認証する、といった回避策を取ります。クエリ文字列はログに残りやすい点に注意です。

再接続ロジックも自前で必要です。接続は普通に落ちるので、クライアント側で指数バックオフ付きの再接続を実装するのが定番です。サーバー側では各接続がソケット(ファイルディスクリプタ)を占有するので、大量同時接続を捌くならOS側のチューニングも要ります。

最後に位置づけだけ整理すると、**SSE**(Server-Sent Events)はサーバー→クライアントの一方向プッシュで済む用途(通知・進捗配信など)ならWebSocketより手軽で、HTTPの仕組みに素直に乗ります。一方、チャットやゲーム、共同編集のように**双方向で低レイテンシ**が要るならWebSocketが本命です。HTTP/2やHTTP/3は多重化が強力ですが本質はリクエスト/レスポンスのままなので、サーバー主導の任意タイミング送信という点ではWebSocketの代替にはなりません(WebSocket over HTTP/2 を定めたRFC 8441も存在します)。

backendをやっているなら、まず `ws` ライブラリで素のエコーサーバーを立てて、ブラウザのDevToolsのNetworkタブでハンドシェイクの `101` と、その後のフレームのやり取りを実際に覗いてみるのが一番腹落ちします。どこか深掘りしたい部分があれば言ってください。
