## DBとは何か

- DBは **Database** の略
- データを安全に保存し、あとから検索・追加・更新・削除できる仕組み
## DBとDBMSの違い

|用語|意味|
|---|---|
|DB|保存されているデータの集まり|
|DBMS|DBを操作・管理するソフトウェア|

| DBMS            | よく使われる場面          |
| --------------- | ----------------- |
| MySQL           | Webアプリでよく使う       |
| PostgreSQL      | 本格的なWebアプリ、業務システム |
| SQLite          | 小規模アプリ、ローカル開発、学習用 |
| Oracle Database | 大企業の基幹システム        |
| SQL Server      | Microsoft系の業務システム |

## なぜDBが必要なのか

|DBが得意なこと|説明|
|---|---|
|検索|条件に合うデータを高速に探せる|
|追加|新しいデータを安全に登録できる|
|更新|既存のデータを書き換えられる|
|削除|不要なデータを消せる|
|整合性管理|おかしなデータを防げる|
|同時アクセス|複数人が同時に使っても壊れにくい|
|権限管理|誰が何を見られるか制御できる|

## DBの基本操作：CRUD

|操作|意味|SQLの命令|
|---|---|---|
|Create|データを追加する|`INSERT`|
|Read|データを読む|`SELECT`|
|Update|データを更新する|`UPDATE`|
|Delete|データを削除する|`DELETE`|

- 「顧客を追加する」はCreate

```sql
INSERT INTO customers (name, email)
VALUES ('山田太郎', 'yamada@example.com');
```

- 「顧客一覧を見る」はRead

```sql
SELECT * FROM customers;
```

- 「メールアドレスを変更する」はUpdate

```sql
UPDATE customers
SET email = 'new-yamada@example.com'
WHERE id = 1;
```

- 「顧客を削除する」はDelete

```sql
DELETE FROM customers
WHERE id = 1;
```

## リレーショナルデータベース(RDB)

- RDBでは、データを **表** の形で管理する
### RDBの例

|  id | name | email                                           |
| --: | ---- | ----------------------------------------------- |
|   1 | 山田太郎 | [yamada@example.com](mailto:yamada@example.com) |
|   2 | 佐藤花子 | [sato@example.com](mailto:sato@example.com)     |

### テーブル・行・列

|用語|意味|
|---|---|
|テーブル|データを入れる表|
|行|1件分のデータ|
|列|データの項目|
|レコード|行とほぼ同じ意味|
|カラム|列とほぼ同じ意味|

### 主キーとは

- そのデータを一意に識別するための値のこと
- たとえば顧客テーブルなら `id` が主キーになる

|id|name|
|--:|---|
|1|山田太郎|
|2|山田太郎|

- 同じ名前の人が複数いる可能性はあるが、`id` があれば、「idが1の山田太郎さん」「idが2の山田太郎さん」として区別できる

### 外部キーとは

-  他のテーブルのデータを参照するためのキー
- たとえば予約テーブルを考えたとき、

|id|customer_id|room_id|check_in|check_out|
|--:|--:|--:|---|---|
|1|3|101|2026-06-25|2026-06-27|

- ここで `customer_id` は、顧客テーブルの `id` を指している

つまり、

```txt
customer_id = 3
```

というのは、

```txt
customersテーブルのid=3の顧客が予約した
```

- 部屋も同じで、`room_id` は rooms テーブルの `id` を参照します。

### なぜテーブルを分けるのか

- たとえば予約テーブルをこのように作ることもできる

|予約ID|顧客名|メール|部屋番号|部屋タイプ|チェックイン|
|--:|---|---|---|---|---|
|1|山田太郎|[yamada@example.com](mailto:yamada@example.com)|101|シングル|6/25|
|2|山田太郎|[yamada@example.com](mailto:yamada@example.com)|102|ダブル|7/1|

- 顧客のメールアドレスが変わったら、山田太郎さんが出てくるすべての予約行を更新しないといけない
#### customers

|id|name|email|
|--:|---|---|
|1|山田太郎|[yamada@example.com](mailto:yamada@example.com)|
#### reservations

|id|customer_id|room_id|check_in|
|--:|--:|--:|---|
|1|1|101|2026-06-25|
|2|1|102|2026-07-01|

- このように、データの重複を減らして整理することを **正規化** と言う

### テーブル同士の関係

- RDBでは、テーブル同士の関係がとても重要
- 主に1対1、1対多、多対多がある

## SQLとは

- SQLは、DBを操作するための言語
### データを取得する

```sql
SELECT *
FROM customers;
```

### 条件をつけて取得する

```sql
SELECT *
FROM customers
WHERE name = '山田太郎';
```

### 並び替える

```sql
SELECT *
FROM reservations
ORDER BY check_in ASC;
```

### 件数を数える

```sql
SELECT COUNT(*)
FROM reservations;
```

### 複数テーブルを結合する

```sql
SELECT
  reservations.id,
  customers.name,
  rooms.room_number,
  reservations.check_in,
  reservations.check_out
FROM reservations
JOIN customers
  ON reservations.customer_id = customers.id
JOIN rooms
  ON reservations.room_id = rooms.id;
```

これで、

```txt
予約ID
顧客名
部屋番号
チェックイン日
チェックアウト日
```

をまとめて取得できる
## JOINとは

JOINは、複数のテーブルをつなげて見るための仕組みです。

たとえば reservations テーブルには `customer_id` しか入っていません。

|id|customer_id|room_id|
|--:|--:|--:|
|1|3|101|

これだけだと、顧客名はわかりません。

そこで customers テーブルとJOINします。

```sql
SELECT
  reservations.id,
  customers.name
FROM reservations
JOIN customers
  ON reservations.customer_id = customers.id;
```

これにより、

|reservation_id|name|
|--:|---|
|1|山田太郎|

のように、予約と顧客情報を組み合わせて見られます。

RDBを理解するうえで、JOINはかなり重要です。

## 制約とは

- DBには **おかしなデータを入れないためのルール** を設定できる
- これを制約と言う

|制約|意味|
|---|---|
|PRIMARY KEY|主キー。一意に識別する|
|FOREIGN KEY|外部キー。他テーブルを参照する|
|NOT NULL|空を許さない|
|UNIQUE|重複を許さない|
|CHECK|条件を満たす値だけ許す|
|DEFAULT|初期値を設定する|

たとえば、

```sql
email TEXT NOT NULL UNIQUE
```

とすると、

```txt
メールアドレスは空にできない
同じメールアドレスを重複登録できない
```

という意味になる

- DB設計では、アプリ側だけでチェックするのではなく、DB側にも最低限の制約を置くのが重要

## トランザクションとは

- たとえば銀行振込を考える

```txt
Aさんの残高を1000円減らす
Bさんの残高を1000円増やす
```

- この2つは必ずセットで成功しないといけない

もし、

```txt
Aさんの残高だけ減った
Bさんの残高は増えていない
```

- そこでトランザクションを使う
- 複数の処理を「全部成功」か「全部失敗」のどちらかにする仕組み

```txt
予約を作成する
空室数を減らす
支払い情報を作る
```

- この途中で失敗したら、全部なかったことに戻す必要がある
- これを **ロールバック** と言う

---

## ACID特性

トランザクションには **ACID** という性質があります。

|性質|意味|
|---|---|
|Atomicity|原子性。全部成功か全部失敗|
|Consistency|一貫性。DBのルールを壊さない|
|Isolation|独立性。同時実行しても干渉しにくい|
|Durability|永続性。確定したデータは消えない|

初心者のうちは、まずはこれだけ覚えれば大丈夫です。

> トランザクションは、複数のDB操作を安全にまとめる仕組み。

## インデックスとは

インデックスは、検索を速くするための仕組みです。

本の索引をイメージするとわかりやすいです。

本で「トランザクション」という単語を探すとき、最初から全ページを読むのは大変です。  
でも索引があれば、何ページに載っているかすぐわかります。

DBのインデックスも同じです。

たとえば、

```sql
SELECT *
FROM customers
WHERE email = 'yamada@example.com';
```

という検索をよくするなら、`email` にインデックスを貼ると速くなります。

```sql
CREATE INDEX idx_customers_email
ON customers(email);
```

ただし、インデックスを貼れば貼るほどよいわけではありません。

インデックスが多すぎると、

```txt
データ追加が遅くなる
更新が遅くなる
DB容量を使う
```

というデメリットもあります。

基本は、

> よく検索条件に使う列にインデックスを貼る

と考えるとよいです。

## 正規化とは

正規化は、

> データの重複や矛盾を減らすために、テーブルを適切に分けること

です。

たとえば、予約テーブルに毎回顧客名とメールアドレスを入れると、同じ情報が何度も出てきます。

これはよくない設計です。

```txt
予約テーブル
- 予約ID
- 顧客名
- 顧客メール
- 部屋番号
- 部屋タイプ
```

これだと重複が多いです。

そこで、

```txt
customers
- id
- name
- email

rooms
- id
- room_number
- room_type

reservations
- id
- customer_id
- room_id
- check_in
- check_out
```

のように分けます。

これが正規化です。

ただし、何でも分ければよいわけではありません。  
分けすぎるとJOINが増えて、扱いにくくなることもあります。

まずは、

> 同じ情報を何度も保存しない  
> 1つの事実は1か所に保存する

という感覚を持つとよいです。

## DB設計の流れ

DB設計は、いきなりテーブルを作るのではなく、次の流れで考えるとよいです。

## ステップ1：何を管理したいか考える

ホテル予約なら、

```txt
顧客
ホテル
部屋
予約
支払い
宿泊プラン
```

などです。

これを **エンティティ** と呼びます。

---

## ステップ2：それぞれが持つ情報を考える

たとえば顧客なら、

```txt
顧客ID
氏名
メールアドレス
電話番号
住所
```

部屋なら、

```txt
部屋ID
部屋番号
部屋タイプ
宿泊料金
定員
```

予約なら、

```txt
予約ID
顧客ID
部屋ID
チェックイン日
チェックアウト日
予約状態
```

です。

---

## ステップ3：関係を考える

たとえば、

```txt
1人の顧客は複数の予約を持つ
1つの部屋は複数の予約履歴を持つ
1つのホテルは複数の部屋を持つ
1つの予約には1つの支払いが対応する
```

という関係を考えます。

---

## ステップ4：テーブルにする

例として、かなり簡略化するとこうなります。

```sql
CREATE TABLE customers (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE
);

CREATE TABLE rooms (
  id INTEGER PRIMARY KEY,
  room_number TEXT NOT NULL,
  room_type TEXT NOT NULL,
  price_per_night INTEGER NOT NULL
);

CREATE TABLE reservations (
  id INTEGER PRIMARY KEY,
  customer_id INTEGER NOT NULL,
  room_id INTEGER NOT NULL,
  check_in DATE NOT NULL,
  check_out DATE NOT NULL,
  status TEXT NOT NULL,
  FOREIGN KEY (customer_id) REFERENCES customers(id),
  FOREIGN KEY (room_id) REFERENCES rooms(id)
);
```

## アプリとDBの関係

Webアプリでは、だいたい次のような構造になります。

```txt
ユーザー
  ↓
画面・フロントエンド
  ↓
バックエンド
  ↓
DB
```

たとえば予約するときは、

```txt
ユーザーが予約フォームを入力
↓
フロントエンドがバックエンドに送る
↓
バックエンドが入力内容を確認する
↓
バックエンドがDBに予約データを保存する
↓
結果をユーザーに返す
```

という流れです。

DBを直接ユーザーが操作するのではなく、普通はバックエンドを通して操作します。

## ORMとは

最近のWeb開発では、SQLを直接書かずに **ORM** を使うことも多いです。

ORMは、

> プログラムのコードからDBを扱いやすくする道具

です。

たとえば TypeScript / Next.js なら Prisma がよく使われます。

SQLで書くと、

```sql
SELECT * FROM customers WHERE id = 1;
```

ですが、Prismaだとイメージとしては、

```ts
const customer = await prisma.customer.findUnique({
  where: { id: 1 },
});
```

のように書けます。

ORMのメリットは、

```txt
型安全に書きやすい
SQLを直接書く量が減る
テーブル定義をコードで管理しやすい
マイグレーションを管理しやすい
```

です。

ただし、ORMを使っていても、裏ではSQLが動いています。  
なので、RDBやSQLの基本は理解しておいたほうがよいです。

## マイグレーションとは

マイグレーションは、

> DBの構造変更を履歴として管理する仕組み

です。

たとえば最初は customers テーブルがこうだったとします。

```txt
customers
- id
- name
- email
```

あとから電話番号を追加したくなりました。

```txt
customers
- id
- name
- email
- phone_number
```

このような変更を記録して、チーム全員や本番環境に反映できるようにするのがマイグレーションです。

Prismaなら、

```bash
npx prisma migrate dev
```

のようなコマンドでマイグレーションを作成・適用します。

## 「DBに状態を持たせる」とはどういうことか

ここは少し誤解しやすいです。

まず、DBは本来 **状態を保存する場所** です。

たとえば、

```txt
予約が存在する
支払い済みである
部屋が利用可能である
ユーザーが登録済みである
```

こういう永続的な状態はDBに保存します。

なので、

> DBに状態を持たせてはいけない

と単純に覚えるのは危険です。

正しくは、文脈によります。

たとえば、よくないのは以下のような設計です。

```txt
一時的な画面の入力途中状態を何でもDBに保存する
アプリの処理手順をDBの値だけで無理やり管理する
ビジネスロジックをDBのストアドプロシージャに寄せすぎる
不要なフラグを大量に作って状態管理が破綻する
```

一方で、保存すべき状態もあります。

```txt
予約ステータス：予約済み / キャンセル済み / 宿泊完了
支払いステータス：未払い / 支払い済み / 返金済み
ユーザーの登録状態：有効 / 停止中
```

大事なのは、

> 永続化すべき業務上の事実はDBに保存する。  
> 一時的な画面状態や処理途中の都合まで何でもDBに押し込まない。

ということです。

## RDBとNoSQLの違い

DBにはRDB以外にも **NoSQL** があります。

|種類|例|特徴|
|---|---|---|
|RDB|PostgreSQL, MySQL, SQLite|表形式、SQL、整合性が強い|
|ドキュメントDB|MongoDB|JSONのような形で保存|
|キーバリューDB|Redis|キーと値で高速に保存|
|グラフDB|Neo4j|関係性の表現が得意|

初心者が最初に学ぶなら、まずはRDBでよいです。

理由は、

```txt
業務システムでよく使われる
SQLが標準的
データ設計の基礎が身につく
整合性を学びやすい
```

からです。

ホテル予約システムのようなものも、基本はRDBが向いています。

## DB設計でよくあるミス

## 1. 何でも1つのテーブルに入れる

これは最初にやりがちです。

```txt
予約テーブルに顧客名、部屋情報、支払い情報、ホテル情報を全部入れる
```

のようにすると、重複や更新漏れが起きやすくなります。

---

## 2. 逆に細かく分けすぎる

正規化を意識しすぎて、何でも分けすぎるのもよくありません。

テーブルが多すぎると、

```txt
JOINが増える
理解しにくい
実装が複雑になる
```

という問題が起きます。

---

## 3. 主キーがない

各テーブルには、基本的に `id` を持たせます。

```txt
customers.id
rooms.id
reservations.id
```

のように、各データを一意に識別できるようにします。

---

## 4. 外部キーを考えていない

たとえば予約テーブルに `customer_name` を直接入れるより、`customer_id` を入れるほうがよいです。

```txt
reservations.customer_id → customers.id
```

という関係にします。

---

## 5. ステータス設計が雑

たとえば予約状態を管理するとき、

```txt
is_cancelled
is_paid
is_checked_in
is_completed
```

のようなフラグが増えすぎると、矛盾が起きやすくなります。

たとえば、

```txt
is_cancelled = true
is_completed = true
```

のようになると意味がわかりません。

この場合は、

```txt
status = 'reserved'
status = 'cancelled'
status = 'completed'
```

のように、状態を1つの列で管理したほうがよいことがあります。

## まず覚えるべき要点

DBの最初の理解としては、以下を押さえればかなり十分です。

|要点|説明|
|---|---|
|DB|データを整理して保存する仕組み|
|RDB|表形式でデータを管理するDB|
|テーブル|データを入れる表|
|レコード|1件分のデータ|
|カラム|データの項目|
|主キー|1件を一意に識別する値|
|外部キー|他のテーブルを参照する値|
|SQL|DBを操作する言語|
|JOIN|複数テーブルをつなげて取得する仕組み|
|正規化|重複や矛盾を減らす設計|
|トランザクション|複数処理を安全にまとめる仕組み|
|インデックス|検索を速くする仕組み|
- クラス図はDBのテーブルと一致
