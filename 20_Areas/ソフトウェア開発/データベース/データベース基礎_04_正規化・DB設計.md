# 正規化とは

正規化は、

> データの重複や矛盾を減らすために、テーブルを適切に分けること

です。

たとえば、予約テーブルに毎回顧客名とメールアドレスを入れると、同じ情報が何度も出てきます。

これはよくない設計です。

```txt
予約テーブル
- 予約ID
- 顧客名
- 顧客メール
- 部屋番号
- 部屋タイプ
```

これだと重複が多いです。

そこで、

```txt
customers
- id
- name
- email

rooms
- id
- room_number
- room_type

reservations
- id
- customer_id
- room_id
- check_in
- check_out
```

のように分けます。

これが正規化です。

ただし、何でも分ければよいわけではありません。  
分けすぎるとJOINが増えて、扱いにくくなることもあります。

まずは、

> 同じ情報を何度も保存しない  
> 1つの事実は1か所に保存する

という感覚を持つとよいです。

# DB設計の流れ

DB設計は、いきなりテーブルを作るのではなく、次の流れで考えるとよいです。

# ステップ1：何を管理したいか考える

ホテル予約なら、

```txt
顧客
ホテル
部屋
予約
支払い
宿泊プラン
```

などです。

これを **エンティティ** と呼びます。

---

# ステップ2：それぞれが持つ情報を考える

たとえば顧客なら、

```txt
顧客ID
氏名
メールアドレス
電話番号
住所
```

部屋なら、

```txt
部屋ID
部屋番号
部屋タイプ
宿泊料金
定員
```

予約なら、

```txt
予約ID
顧客ID
部屋ID
チェックイン日
チェックアウト日
予約状態
```

です。

---

# ステップ3：関係を考える

たとえば、

```txt
1人の顧客は複数の予約を持つ
1つの部屋は複数の予約履歴を持つ
1つのホテルは複数の部屋を持つ
1つの予約には1つの支払いが対応する
```

という関係を考えます。

---

# ステップ4：テーブルにする

例として、かなり簡略化するとこうなります。

```sql
CREATE TABLE customers (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE
);

CREATE TABLE rooms (
  id INTEGER PRIMARY KEY,
  room_number TEXT NOT NULL,
  room_type TEXT NOT NULL,
  price_per_night INTEGER NOT NULL
);

CREATE TABLE reservations (
  id INTEGER PRIMARY KEY,
  customer_id INTEGER NOT NULL,
  room_id INTEGER NOT NULL,
  check_in DATE NOT NULL,
  check_out DATE NOT NULL,
  status TEXT NOT NULL,
  FOREIGN KEY (customer_id) REFERENCES customers(id),
  FOREIGN KEY (room_id) REFERENCES rooms(id)
);
```
