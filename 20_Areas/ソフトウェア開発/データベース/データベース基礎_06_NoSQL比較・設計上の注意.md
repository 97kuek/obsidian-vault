# RDBとNoSQLの違い

DBにはRDB以外にも **NoSQL** があります。

|種類|例|特徴|
|---|---|---|
|RDB|PostgreSQL, MySQL, SQLite|表形式、SQL、整合性が強い|
|ドキュメントDB|MongoDB|JSONのような形で保存|
|キーバリューDB|Redis|キーと値で高速に保存|
|グラフDB|Neo4j|関係性の表現が得意|

初心者が最初に学ぶなら、まずはRDBでよいです。

理由は、

```txt
業務システムでよく使われる
SQLが標準的
データ設計の基礎が身につく
整合性を学びやすい
```

からです。

ホテル予約システムのようなものも、基本はRDBが向いています。

---

# DB設計でよくあるミス

# 1. 何でも1つのテーブルに入れる

これは最初にやりがちです。

```txt
予約テーブルに顧客名、部屋情報、支払い情報、ホテル情報を全部入れる
```

のようにすると、重複や更新漏れが起きやすくなります。

---

# 2. 逆に細かく分けすぎる

正規化を意識しすぎて、何でも分けすぎるのもよくありません。

テーブルが多すぎると、

```txt
JOINが増える
理解しにくい
実装が複雑になる
```

という問題が起きます。

---

# 3. 主キーがない

各テーブルには、基本的に `id` を持たせます。

```txt
customers.id
rooms.id
reservations.id
```

のように、各データを一意に識別できるようにします。

---

# 4. 外部キーを考えていない

たとえば予約テーブルに `customer_name` を直接入れるより、`customer_id` を入れるほうがよいです。

```txt
reservations.customer_id → customers.id
```

という関係にします。

---

# 5. ステータス設計が雑

たとえば予約状態を管理するとき、

```txt
is_cancelled
is_paid
is_checked_in
is_completed
```

のようなフラグが増えすぎると、矛盾が起きやすくなります。

たとえば、

```txt
is_cancelled = true
is_completed = true
```

のようになると意味がわかりません。

この場合は、

```txt
status = 'reserved'
status = 'cancelled'
status = 'completed'
```

のように、状態を1つの列で管理したほうがよいことがあります。

# まず覚えるべき要点

DBの最初の理解としては、以下を押さえればかなり十分です。

|要点|説明|
|---|---|
|DB|データを整理して保存する仕組み|
|RDB|表形式でデータを管理するDB|
|テーブル|データを入れる表|
|レコード|1件分のデータ|
|カラム|データの項目|
|主キー|1件を一意に識別する値|
|外部キー|他のテーブルを参照する値|
|SQL|DBを操作する言語|
|JOIN|複数テーブルをつなげて取得する仕組み|
|正規化|重複や矛盾を減らす設計|
|トランザクション|複数処理を安全にまとめる仕組み|
|インデックス|検索を速くする仕組み|
- クラス図はDBのテーブルと一致
