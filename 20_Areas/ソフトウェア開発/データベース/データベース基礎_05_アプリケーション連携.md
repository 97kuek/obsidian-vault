# アプリとDBの関係

Webアプリでは、だいたい次のような構造になります。

```txt
ユーザー
  ↓
画面・フロントエンド
  ↓
バックエンド
  ↓
DB
```

たとえば予約するときは、

```txt
ユーザーが予約フォームを入力
↓
フロントエンドがバックエンドに送る
↓
バックエンドが入力内容を確認する
↓
バックエンドがDBに予約データを保存する
↓
結果をユーザーに返す
```

という流れです。

DBを直接ユーザーが操作するのではなく、普通はバックエンドを通して操作します。

# ORMとは

最近のWeb開発では、SQLを直接書かずに **ORM** を使うことも多いです。

ORMは、

> プログラムのコードからDBを扱いやすくする道具

です。

たとえば TypeScript / Next.js なら Prisma がよく使われます。

SQLで書くと、

```sql
SELECT * FROM customers WHERE id = 1;
```

ですが、Prismaだとイメージとしては、

```ts
const customer = await prisma.customer.findUnique({
  where: { id: 1 },
});
```

のように書けます。

ORMのメリットは、

```txt
型安全に書きやすい
SQLを直接書く量が減る
テーブル定義をコードで管理しやすい
マイグレーションを管理しやすい
```

です。

ただし、ORMを使っていても、裏ではSQLが動いています。  
なので、RDBやSQLの基本は理解しておいたほうがよいです。

# マイグレーションとは

マイグレーションは、

> DBの構造変更を履歴として管理する仕組み

です。

たとえば最初は customers テーブルがこうだったとします。

```txt
customers
- id
- name
- email
```

あとから電話番号を追加したくなりました。

```txt
customers
- id
- name
- email
- phone_number
```

このような変更を記録して、チーム全員や本番環境に反映できるようにするのがマイグレーションです。

Prismaなら、

```bash
npx prisma migrate dev
```

のようなコマンドでマイグレーションを作成・適用します。

# 「DBに状態を持たせる」とはどういうことか

ここは少し誤解しやすいです。

まず、DBは本来 **状態を保存する場所** です。

たとえば、

```txt
予約が存在する
支払い済みである
部屋が利用可能である
ユーザーが登録済みである
```

こういう永続的な状態はDBに保存します。

なので、

> DBに状態を持たせてはいけない

と単純に覚えるのは危険です。

正しくは、文脈によります。

たとえば、よくないのは以下のような設計です。

```txt
一時的な画面の入力途中状態を何でもDBに保存する
アプリの処理手順をDBの値だけで無理やり管理する
ビジネスロジックをDBのストアドプロシージャに寄せすぎる
不要なフラグを大量に作って状態管理が破綻する
```

一方で、保存すべき状態もあります。

```txt
予約ステータス：予約済み / キャンセル済み / 宿泊完了
支払いステータス：未払い / 支払い済み / 返金済み
ユーザーの登録状態：有効 / 停止中
```

大事なのは、

> 永続化すべき業務上の事実はDBに保存する。  
> 一時的な画面状態や処理途中の都合まで何でもDBに押し込まない。

ということです。
