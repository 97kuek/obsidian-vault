## 詳細設計・実装の手順

### 設計の全体フロー

**基本設計（1）**

- 全体アーキテクチャを決める
- 分析レベルのクラス名を実装向けに変換（例：「日程」→ `Date`）
- ユースケースに対応する処理をオブジェクト間相互作用として設計

**詳細設計（2）**

- どこから設計するか順序を決める
- 各部分を実装方法に即して詳細化する
### 中心から外側へ（Inside-Out Principle）

- **「変更されにくい安定した部分から設計する」**という原則
- 良いソフトウェアは「変更に強く（EoC）」「テストしやすい（EoT）」必要があるから

|順序|層|役割|安定性|
|---|---|---|---|
|1|エンティティ|データや核となるロジック|安定（変わりにくい）|
|2|コントロール|エンティティを操作する制御ロジック|中間|
|3|バウンダリ|画面GUIなど|不安定（変わりやすい）|

## クラス図からJavaコードへの落とし方

### 基本型

- 属性は `private` フィールド
- 操作（メソッド）は `public` 

```java
class Window {
    private int xPos;
    public void moveHz(int offset) {...}
}
```

### 関連（1対1）

- 相手のオブジェクトをフィールドとして持つ

```java
class Order {
    private Product theProduct;  // 1対1の関連
}
```

### 集約（1対多）

- 配列やコレクションで持つ

```java
class Window {
    private Button[] theButton;  // 1対*の集約
}
```

- 実装の選択肢：配列、`ArrayList`、`HashMap` など、要件に応じて選択

### 継承

- `extends` キーワードで実現

```java
class ErrorWindow extends Window {
    String message;
}
```

### シーケンス図とコードの対応

- シーケンス図の各メッセージは、メソッド呼び出しに対応
- `<<create>>` は `new` に対応

## 航空便座席予約システムの具体例

- ドメイン層を中心から順に①〜⑧の番号で設計していく
### ① 日程クラス → `java.util.Date` を再利用

- 新しく作らず、既存のJava標準ライブラリを使用
- **車輪の再発明をしない**という大事な考え方

### ② 空港クラス → Singleton的な設計

- 「成田空港」というオブジェクトが複数生成されると困る。そこで、
- コンストラクタを `private` にして外部からの生成を禁止
- `static` な `HashMap` で「空港名 → Airportオブジェクト」を管理
- `getInstance(name)` で取得 → なければ生成、あればキャッシュから返す

```java
public static Airport getInstance(String locate) {
    Airport result = (Airport) airportMap.get(locate);
    if (result == null) {
        result = new Airport(locate);
        airportMap.put(locate, result);
    }
    return result;
}
```

- これは **Singletonパターン**（の応用、正確にはMultitonと呼ばれることもある）の実例

### ③〜⑤ 座席クラス → 継承で多値分類を表現

- 「座席種別」には「エコノミークラス」「ビジネスクラス」がある。これを次のように設計
- `SeatClass` を `abstract` クラスにして直接インスタンス化を禁止
- `EconomyClass` と `BusinessClass` が継承
- 共通属性（`price`, `vacancy`）は親クラスに、固有の振る舞い（`getClassInfo()`）は子で実装

### ステートマシン図で状態遷移を設計

座席は「空席数」という状態を持つ**ステートフルクラス**です。状態遷移をステートチャート図で設計します。

```
[有効] --decrementVacancy()[vacancy>0]--> [有効] (vacancy--)
[有効] --decrementVacancy()[vacancy=0]--> [無効]
```

ガード条件 `[vacancy > 0]` がポイントで、空席数が0になったら状態が変わる、ということを表現しています。

### 実装例（SeatClass）

```java
public abstract class SeatClass {
    private int price = 0;
    private int vacancy = 0;
    
    public SeatClass(int initPrice, int initVacancy) {
        if (initVacancy < 1) {
            throw new RuntimeException("Vacancy cannot be initialized...");
        }
        price = initPrice;
        vacancy = initVacancy;
    }
    
    public void decrementVacancy() {
        if (vacancy > 0) vacancy--;
    }
    
    public abstract String getClassInfo();  // 子クラスで実装
}
```
## デザインパターンと設計原則

### なぜパターンが必要か

- 設計を個人のノウハウに頼ると、機能ごとにバラバラなアーキテクチャになる
- パターンを使うことで一貫性が生まれ、品質・保守性・生産性が向上する
### Strategyパターン（戦略パターン）

- 「アルゴリズムを切り替えたい」という典型的な問題への解決策

**NG例**：

```java
class Mathematic {
    public Data sort(Data data) {
        switch(settings) {
            case QUICK: return quickSort(data);
            case BUBBLE: return bubbleSort(data);
        }
    }
}
```

- ソートアルゴリズムを追加するたびに `Mathematic` クラスを変更しないといけない。

**OK例（Strategy）**：

```java
class Mathematic {
    Sorter sorter;  // インタフェースに依存
    public Data sort(Data data) {
        return sorter.sort(data);
    }
}

abstract class Sorter {
    public abstract Data sort(Data);
}

class QuickSorter extends Sorter { ... }
class BubbleSorter extends Sorter { ... }
```

- ソートアルゴリズムを差し替えても `Mathematic` を変更する必要がない
- これが**振る舞いのカプセル化**と**ポリモーフィズム**の威力

### SOLID原則

クラス設計の5つの原則。覚え方として頭文字で **SOLID**。

|略称|名前|意味|
|---|---|---|
|**S**RP|単一責任の原則|クラスを変更する理由は1つだけ|
|**O**CP|オープンクローズドの原則|拡張に開き、修正に閉じる|
|**L**SP|リスコフの置換原則|サブクラスはスーパークラスと置換可能|
|**I**SP|インタフェース分離の原則|使わないメソッドへの依存を強制しない|
|**D**IP|依存関係逆転の原則|抽象に依存し、具象に依存しない|

#### オープンクローズドの原則の例

`Client` が `Server` を直接使うとNG。間に `ClientInterface` を挟むことで、`Server` を別のクラスに差し替えても `Client` の変更不要になる。

#### リスコフの置換原則の例

`Square extends Rectangle` は一見正しそうですが、**正方形は幅と高さが等しい**という制約があるため、`setWidth(5); setHeight(4);` を呼ぶと数学的に矛盾します。

→ 解決策：`Rectangle` と `Square` を兄弟クラス（共通の親 `FigureElement` の子）にする。

「is-a 関係」だけで継承を判断してはいけない、という教訓です。

## 5. 演習：改札システムの設計と実装

ここが課題部分ですね。詳しく見ていきましょう。

### クラス構造

```
TicketEntry --- Gate --- Bar
  + entry()     + ticket()    + open()
  + reject()    + pass()      + pass()
                              + close()
                              + alarm()
```

### Gateのステートマシン図

`Gate` は `closed` と `opened` の2状態を持ちます。資料の図を読み解くと、次のようになります。

```
        ticket / bar.open()
closed ─────────────────────► opened
   ▲                            │
   │                            │ pass / bar.close()
   │ ticket / entry.reject()    │
   │ ←──────── closed ◄─────────┘
   │
   └─ pass / bar.alarm()  (closed状態でpassが来たら警報)
```

整理すると次の4つの遷移です。

|現在状態|イベント|アクション|次状態|
|---|---|---|---|
|closed|ticket|bar.open()|opened|
|opened|pass|bar.close()|closed|
|opened|ticket|entry.reject()|opened（自己遷移）|
|closed|pass|bar.alarm()|closed（自己遷移）|

考え方：

- 切符が入った（ticket）→ 開ける
- 通過した（pass）→ 閉める
- 開いてるのにまた切符 → 拒否（reject）
- 閉まってるのに通過しようとした → 警報（alarm）

### Gate.javaの実装

`switch` 文で状態ごとに分岐します。

```java
public class Gate {
    private static final int CLOSED = 0;
    private static final int OPENED = 1;
    private Bar bar = new Bar();
    private TicketEntry entry = new TicketEntry();
    private int state = CLOSED;  // 初期状態

    public void pass() {
        switch (state) {
            case CLOSED:
                bar.alarm();        // 閉まってるのに通過 → 警報
                break;
            case OPENED:
                bar.close();        // 開いてる時に通過 → 閉める
                state = CLOSED;
                break;
            default:
                break;
        }
    }

    public void ticket() {
        switch (state) {
            case CLOSED:
                bar.open();         // 閉まってる時に切符 → 開ける
                state = OPENED;
                break;
            case OPENED:
                entry.reject();     // 開いてるのに切符 → 拒否
                break;
            default:
                break;
        }
    }
}
```

### 発展：Stateパターンとの関係

実は、この `switch` 文での実装は**Stateパターン**を使うとさらに綺麗になります。状態をクラスとして表現し、各状態が自身の振る舞いを持つようにする手法です。SOLID原則の OCP に従う設計ですね。今回の演習では `switch` 文で十分ですが、状態の数や種類が増えてきたら検討する価値があります。
