## システム分析とは何か

### 開発工程の中での位置づけ

- ソフトウェア開発の流れは以下の通り

```
概念モデル → 要求モデル → 分析モデル → 設計モデル → 実装モデル
```

- **概念モデル**: 業務の世界に何があるかを表す。「座席とは何か」「予約とは何か」を整理する。実装の話は一切しない。
- **要求モデル**: 顧客がシステムに何を望むかを表す。「座席をオンラインで予約できるようにしてほしい」など。
- **分析モデル**: 「機能をシステムとしてどう実現するか」を、**実装方法には踏み込まずに**表す。「画面から希望座席を指定すると、予約処理が…」というレベル。
- **設計モデル**: ここで初めて「DBを使う」「CGIで通信する」といった実装方針が入る。
- **実装モデル**: 実際のソースコード。

※ 座席予約システムを例にとる
### 分析モデルの「立ち位置」が一番大事

- 分析モデルは**実装方法を関知しない**
- データベースを使うかどうか、同期通信か非同期か、どの言語で書くか、という話は**まだ書かない**。
- 「機能を実現するための論理的な構造と振る舞い」だけを描くのが分析モデル
### オブジェクト指向分析モデルの中身

- UMLでは分析モデルを2つの側面で表現

|側面|図|役割|
|---|---|---|
|静的側面|**分析レベル・クラス図**|必要なクラスとクラス間の関係|
|動的側面|**コミュニケーション図**（またはシーケンス図）|ユースケースの機能を、オブジェクト間の相互作用としてどう実現するか|

- **どんな部品があって（クラス図）、それらがどう連携して仕事をするか（コミュニケーション図）**を書く
### 分析の手順（3ステップ）

1. **各ユースケースについてコミュニケーション図を作る**
    - (1-1) 図を新規作成
    - (1-2) 必要なクラスのオブジェクトを配置
    - (1-3) オブジェクト間のメッセージを描く
2. **全コミュニケーション図を束ねてクラス図を作る**
3. **各メッセージに対応する操作を、各クラスに追加する**

- 動きから先に決めて、それを集約して静的構造を作るというボトムアップな進め方
- UML 設計の典型的なやり方の一つ
## クラス・オブジェクトの特定

### ロバストネス分析と3種のクラス

- コミュニケーション図に登場するオブジェクトを、必ず次の3種類のいずれかに分類

|種類|アイコン|役割|例|
|---|---|---|---|
|**バウンダリ (boundary)**|⊢〇|アクタとシステムの境界。インタフェース。|画面、外部API、センサ|
|**コントロール (control)**|〇に矢印|処理の調整役。バウンダリの刺激を受けて、エンティティやコントロールを操作する。|「予約処理」「引き出し処理」|
|**エンティティ (entity)**|〇|データの保持。一時的または永続的に記録する。|「座席」「口座」「航空便」|

#### なぜ3つに分けるのか？

- **変化の頻度が違うものを混ぜない**ため
- バウンダリ（画面など）は**よく変わる**。UIは流行り廃りで頻繁に作り直される。
- エンティティ（座席など業務データ）は**ほとんど変わらない**。業務の本質だから。
- コントロールはその間。
#### ステレオタイプ表記

- クラス図上は `<<boundary>>`, `<<control>>`, `<<entity>>` というステレオタイプで表記
- アイコン記法（⊢〇 / 〇に矢印 / 〇）を使ってもOK
- 四角の中にステレオタイプを書いてもOK
### 概念モデルからのコピー

- エンティティの候補は、**概念モデルにすでにあるクラス**
- 概念レベル・クラス図のクラス（航空便、座席、空港、日程、予約 など）をコピー
- それを `<<entity>>` として分析モデルに置く
### 命名のコツ

航空便登録ユースケースの場合:

- バウンダリ → 「航空便管理画面」
- コントロール → 「航空便管理処理」

座席検索/予約のユースケースの場合:

- バウンダリ → 「座席予約画面」
- コントロール → 「座席予約処理」

- 「○○画面」「○○処理」のように、**機能名 + 役割**で命名するときれいに揃う

## 相互作用の分析

### 9つの相互作用パターン

- これは**書いていいパターンの一覧**であり、同時に**これ以外は書いてはいけない**という制約でもある

|#|パターン|図のイメージ|
|---|---|---|
|1|アクタ → バウンダリ（入力）|人 → ⊢〇|
|2|バウンダリ → バウンダリ（画面遷移）|⊢〇 → ⊢〇|
|3|バウンダリ → 自分自身|⊢〇 ↻|
|4|バウンダリ → コントロール（起動）|⊢〇 → 〇|
|5|コントロール → バウンダリ（表示指示）|〇 → ⊢〇|
|6|コントロール → コントロール|〇 → 〇|
|7|コントロール → 自分自身|〇 ↻|
|8|コントロール → エンティティ（利用）|〇 → 〇|
|9|エンティティ → エンティティ（関係）|〇 — 〇|

### 「書いてはいけないこと」

- ロバストネス分析の制約

- ❌ アクタ ↔ コントロール（直接やりとりしてはダメ。必ずバウンダリを経由）
- ❌ アクタ ↔ エンティティ（同上）
- ❌ バウンダリ ↔ エンティティ（コントロールを経由する）
- ❌ エンティティ ↔ コントロール（向きが逆。エンティティから能動的に呼ばない）
#### なぜ禁止？

- 「**画面が直接データに触るな、必ずコントロールを通せ**」というルール
- これを守ると、「画面を変えてもデータ構造に影響しない」「データ構造を変えても画面に影響しない」という分離が実現できる
### 相互作用の例

ATMの「現金引き出し」だと:

```
:利用者 → :現金自動支払画面 → :引き出し処理 → :口座
   1.引き出し額入力   2.引き出す      2.1.残金を減らす
                  ← 2.2.引き出し額表示
```

矢印の向きと番号付けがポイント。**入れ子の処理は番号で表現**します。

- `2: 引き出す` の中で呼ばれる処理は `2.1: 残金を減らす`、`2.2: 引き出し額を表示する`

「航空便を登録する」の例（p.26）も同じ仕組み。`2: 航空便を登録する()` の中で `2.1〜2.10` までの内部処理が走る、という構造になっています。

### 同期 vs 非同期

- **アクタ → バウンダリ** は基本**非同期**（人間の操作はいつ来るか分からない）
- それ以外は基本**同期**
- 迷ったらとりあえず同期にしておけばOK
- 分析レベルではあまり厳密に区別しなくて構わない

### 条件分岐の書き方

- ガード条件を `[ ]` で囲んで書く

- `2.4: [検索条件に合致する空席がある] 空席一覧を表示する()`
- `2.5: [検索条件に合致する空席がない] 警告を表示する()`

また、入れ子のガードも使えます。

- `2.2.3: [空席数 > 0] 出発地を取得する()`

これは「空席数がゼロのときは、出発地などの取得自体しない（無駄な処理を省く）」という意味で、論理的に正しいモデリングです。
## クラス図の拡充

### 手順(2): コミュニケーション図を束ねる

- 全ユースケース分のコミュニケーション図ができたら、それらを**統合**してクラス図を作成
### クラス図作成のポイント

1. **多重度を必ず書く**（1, *, 1..*など）
2. **汎化、集約、コンポジション**も明示する
3. **`<<create>>` 依存関係**: あるクラスが別クラスのオブジェクトを生成し、生成後は参照を持たない場合に使う
4. **アクタは登場させない**（クラス図はシステム内部の話）
5. **概念モデルにあるクラスは、コミュニケーション図に出てこなくても登場させてよい**
6. **関連の方向は明示しない**（分析レベルでは双方向で十分）
### `<<create>>` 関係の意味

これは細かいけど大事な区別です。資料がわざわざ問いかけているのは:

- 座席予約処理は、航空便を**ずっと参照する**のか、**一時的に使う**のか？
- 日程・空港・予約は、生成後も処理クラスから参照されるのか？

答えは「ケースバイケースだけど、**一時的に使うだけなら通常の関連は不要**で、`<<create>>` だけ書けばよい」ということ。

p.32 のクラス図を見ると:

- 座席予約処理 → 予約: `<<create>>`（生成だけ。生成後は座席種別が予約を保持する）
- 航空便管理処理 → 日程, 空港, エコノミークラス, ビジネスクラス: `<<create>>`（生成して航空便に持たせるだけ）
- 航空便管理処理 ⟷ 航空便: 通常の関連（管理処理は航空便を継続的に管理する）

このあたり、設計の話に近いので、そこまで厳密にやらなくても演習レベルではOKです。

### 手順(3): 操作の追加（p.33–34）

最後に、**コミュニケーション図でメッセージを受け取っているクラス**に、メッセージと同名の操作を追加します。

ルール:

- 「作成する」系のメッセージは操作にしなくてよい（コンストラクタ相当だから）
- 各クラスにステレオタイプ、属性、操作を明示する
- アクタは登場させない

p.34 の最終クラス図を見ると、各クラスに操作がきれいに整理されています。例えば:

- `<<boundary>> 座席予約画面`: 座席検索条件を入力する(), 空席一覧を表示する(), 警告を表示する(), 座席を選択する(), 料金を表示する()
- `<<control>> 座席予約処理`: 座席を検索する(), 座席を予約する(), 空席一覧を作成する(), 航空便の空席を確認する()
- `<<entity>> 座席種別`: 値段, 空席数 / 空席数を取得する(), 空席数を1つ減らす(), 値段を取得する()

これを眺めると、**コミュニケーション図のメッセージ = クラスの操作**という対応が一目で分かるはずです。

## システム分析時の留意事項

**全体**
- ユースケースの全振る舞いを網羅している
- データとその置きどころが明確
- 相互作用やオブジェクト数とクラス図の関係・操作・多重度が対応している

**オブジェクト**
- B/C/E の3つ組がそれぞれの役割に合致している
- **コントロールはユースケース当たり5個以内**（多すぎは責務分割の失敗）

**相互作用**
- 基本系列だけでなく代替・例外系列も描く
- ユースケース記述に合致している
- B/C/E 間の許される相互作用のみが書かれている
## まとめ：この章で押さえるべき3点

1. **分析モデルは「実装に踏み込まず、機能の実現方法を表す」モデル**。設計とは違う。
2. **ロバストネス分析の B/C/E 三つ組と、9つの相互作用パターン（裏を返せば禁止パターン）** が骨格。これを守れば自然と疎結合な構造になる。
3. **手順は「コミュニケーション図 → クラス図 → 操作追加」のボトムアップ**。動的側面から静的側面を導出する。
