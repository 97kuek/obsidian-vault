## 高林_MISIA Mood Recommender

### 1. 何を作ったか

**「今の気分を言葉で入力すると、MISIAの曲を推薦してくれるアプリ」**

たとえば「まじで萎えてる」「研究発表前で緊張してる」のような曖昧な気分のテキストを入力すると、それに合ったMISIAの楽曲を1〜3曲、推薦理由つきで返してくれる。

### 2. なぜ作ったか（背景・課題）

- 「今の気分で曲を探したい」という場面は多いが、その気分を具体的な曲名に変換するのは難しい
- 気分→楽曲のマッピングをLLM（Claude）に任せると、**実在しない曲を推薦してしまう**リスクがある（いわゆるハルシネーション）
- アーティスト特化にするには、曲データベース（DB）が必要

### 3. アプローチ・仕組み（技術的な核心）

**LLMとDBの役割分担**がこのシステムの最大の工夫点。

```
ユーザー入力（自然言語の気分）
        ↓
① Claude：感情を言語化（例：「胸の奥で燃え盛る、やり場のない怒り」）
        ↓
② 曲DBの感情ラベルと照合して候補曲を絞り込む
        ↓
③ 候補曲だけをClaudeに渡して推薦文を生成
        ↓
推薦カード表示（曲名・推薦理由・公式ページURL）
```

ポイントは**③でClaudeに自由に曲を選ばせない**こと。先にDBで候補を絞ってからClaudeに渡すことで、実在しない曲が推薦されるのを防いでいる。

### 4. 曲DB（408曲）の設計

各曲に以下の情報を持たせている

|フィールド|内容|
|---|---|
|title|曲名|
|mood|感情ラベル（例：愛、恋愛、温かい、安心）|
|description|曲の説明文|
|source_url|公式ページURL|
|release_date / type|リリース情報|

感情ラベルがマッチングの要であり、**ラベルの設計品質が推薦品質に直結する**と正直に述べている点が誠実。

### 5. その他の工夫

|工夫|内容|
|---|---|
|APIなしデモモード|発表時にAPIキー・通信なしでも動く|
|セキュリティ|APIキーを保存しない設計|
|HTMLエスケープ|入力・出力を安全に表示|
|拡張性|DBを差し替えれば他アーティストにも対応できる|

### 6. 評価の課題

現状、評価が十分にできていないと自己申告している。評価の観点として：

- 感情の解釈が正しいか（例：「緊張」→不安として解釈できているか）
- 推薦曲がDBの408曲の中から選ばれているか
- 推薦理由が入力と対応しているか

これを定量的に評価する仕組みがまだ整っていない。

### 7. 全体の評価（発表者の研究として）

**良い点：**

- LLMに「自由に曲を選ばせない」という設計判断が的確。RAG（Retrieval-Augmented Generation）的なアーキテクチャになっており、実用的な発想
- 出力例が具体的で、アプリとして動いているのが伝わる
- 限界（評価が不十分）を正直に述べている

**今後の改善余地：**

- 感情ラベルの設計をより体系的に（例：VAD空間＝Valence/Arousal/Dominanceなど）
- 評価指標の自動化（推薦されたDB内の曲かどうかを自動チェックするスクリプトなど）
- 推薦の多様性確保（似たような感情ラベルの曲ばかり推薦されないか）

## 越智_AI駆動開発によるチャットボットの構築

### 1. 何を作ったか

**「画像対応マルチモーダルチャットボット」**

テキストだけでなく、画像の入力・解析・生成まで扱えるチャットボット。StreamlitでWebアプリとして実装し、Gemini APIを応答エンジンとして使用している。

### 2. 使用技術の役割分担

|技術|役割|
|---|---|
|Claude Code|システム開発（一部ChatGPTも使用）|
|Gemini API|テキスト応答 + 画像理解|
|Imagen API|画像生成（有料版のみ → 未完了）|
|Streamlit|WebアプリUI|

### 3. システムの処理フロー

```
ユーザー（テキスト入力 or 画像入力）
        ↓
    chatbot.py（メイン制御）
        ↓            ↓
  Gemini API      Imagen API
（応答・画像理解）  （画像生成）
        ↓
  会話履歴保存 / 出力画像保存
```

### 4. プログラム構成（モジュール設計）

責務ごとにファイルを分離した、整理された構成になっている。

```
chatbot.py          ← 全体制御
├── core/
│   ├── gemini.py   ← Gemini APIとの通信
│   ├── imagen.py   ← 画像生成
│   ├── code_runner.py ← AIが生成したPythonコードの実行
│   └── history.py  ← 会話履歴の保存・読み込み
├── ui/
│   ├── chat.py     ← チャット画面
│   ├── sidebar.py  ← 設定サイドバー
│   └── logo.py     ← アイコン表示
├── histories/      ← 会話履歴（JSON）
└── assets/         ← 静的ファイル
```

単一ファイルに全部書かず、機能ごとに分けているのは保守性の面で適切な設計。

### 5. 工夫点（4つ）

**① 画像入出力への対応**

- 画像をアップロードしてGeminiに解析させる
- `/image 犬が走っている` のようなコマンドで画像生成
- 生成画像をファイルに保存して後から再利用可能

**② 会話履歴の保持**

- セッションが終了しても会話内容がJSONに自動保存される
- 過去の履歴を後から読み込んで会話を継続できる

**③ Geminiモデルの自動取得・選択**

- APIキーを入力すると利用可能なモデル一覧を自動取得
- ドロップダウンから選ぶだけでよい → 設定ミスを防止

**④ システムプロンプトのGUI編集**

- 「どう答えてほしいか」をUI上から自由に変更できる
- デフォルトは「日本語で丁寧に、グラフはmatplotlibのコードのみ返す」という設定

### 6. 課題（正直な自己評価）

**① AI出力の不安定性**

- テキスト応答は概ね良好だが、画像処理タスク（モノクロ化、複数条件の画像生成など）では出力品質が安定しない
- LLMの確率的な揺らぎ + プロンプトの解釈の変動が原因

**② UIの最適化不足**

- サイドバーの項目が多くなりすぎてスクロールが必要に
- ClaudeにUI修正を指示しても根本解決には至らなかった
- **「AIはコード生成・機能追加には強いが、実行環境依存の細かいレイアウト調整は人間による確認が依然重要」** という考察は、AI駆動開発の限界を正直に示していて興味深い

**Appendixで明かされるバグの数々（率直な開示）**

- 画像がUIには表示されているのにGeminiに送信されていなかった
- モノクロ化でRGB→グレースケール変換が未実装だった（`cmap='gray'`だけでは変換にならない）
- 複数画像生成時にファイル名が衝突していた
- 会話履歴に画像データが保存されておらず、読み込み時にエラーが出ていた
