## プログラムの作法
### 環境構築・実行
```bash
# b4studyという名前のpython3.10の仮想環境を新しく作成
conda create -n b4study python=3.10

# 作成した環境に入る
# ターミナルの先頭に(b4study)と表示されていれば有効
conda activate b4study
```
### GPU使用
- 使う前に以下のコマンドで空きを確認する
```bash
# GPUの状態を一覧表示するコマンド
nvidia-smi
```
- `Memory-Usage`: VRAM（GPU専用メモリ）の使用量と総容量。モデルや学習中のテンソルはすべてVRAMに載るので、ここが満杯だとOut of Memoryエラーで自分のプログラムも他人のプログラムも落ちる
- `GPU-Util`: GPUの演算コアがどれだけ稼働しているかの割合。これが高いということはGPUが忙しく計算している状態なので、そこに自分のプロセスを追加すると時分割で交互に実行されて両方とも遅くなる
- `Memory-Usage`に空きがあっても`GPU-Util`が高ければ使ってはいけない。逆に、VRAMの「現在値」は瞬間値であって、学習中にピークでもっと増えることがある
- 以下のコマンドでGPU番号を指定できる
```
CUDA_VISIBLE_DEVICES=1 python demo.py
```
- 下部の**Processes**セクションには、どのプロセスがどのGPUをどれだけ使っているかが表示される。python3やpythonというプロセス名が他人のものかどうかはPIDで判断できる。
### tmux
- サーバー上に「仮想ターミナル」を作って常駐させるツール
- ディープラーニングの学習は数時間〜数日かかることが普通なので、その間ずっとSSH接続を維持し続けるのは現実的ではない。Wi-Fiが途切れる、PCがスリープする、VPNが切れる、そういった瞬間に数時間分の学習が無駄になる。
- tmuxを使うと、サーバー上に独立した仮想ターミナル（セッション）が生まれ、SSH接続が切れてもそのセッションはサーバー上で動き続ける。後からSSHで再接続して、そのセッションに戻れば、学習の途中経過もそのまま見える。
#### 基本的な使い方の流れ
まずSSHでサーバーに接続した後、学習を始める前にtmuxセッションを作る。
1. セッションの作成
```
tmux new -s mnist_train
```
- `-s` はセッション名を指定するオプション
- 名前をつけておくと後で何の学習を回していたか分かりやすい
- この中でいつも通り `python demo.py` などを実行すれば、学習が始まる
2. デタッチ（セッションから離脱）
- 学習が動き始めたら、セッションから離脱できる。
```
Ctrl+b を押して、指を離してから d を押す
```
3. セッション一覧の確認
```
tmux ls
```
- 現在サーバー上に存在するtmuxセッションの一覧が表示される。出力例としては `mnist_train: 1 windows (created Wed Apr 23 14:30:00 2026)` のように、セッション名と作成日時が出る。共有サーバーでは他の人のセッションも見えるので、誰が何を回しているかの参考にもなる。
4. アタッチ（セッションに再接続）
```
tmux attach -t mnist_train
```
- `-t` はターゲットのセッション名を指定するオプション
- これで先ほどのセッションに戻ると、学習のログがずらっと流れている状態がそのまま見える
5. セッションの終了
- 学習が完全に終わって、もうそのセッションが要らなくなったら、セッション内で `exit` と打つか、`Ctrl+d` でシェルを閉じれば、セッション自体が消える。
- セッションの外から消したい場合は以下。
```
tmux kill-session -t mnist_train
```
### 実際の運用パターン

典型的な流れとしては、まずSSHでサーバーに入り、`nvidia-smi` でGPUの空きを確認する。空いているGPUがあれば `tmux new -s experiment_01` でセッションを作り、`CUDA_VISIBLE_DEVICES=1 python train.py` のように使うGPU番号を指定して学習を開始する。学習のログが流れ始めたのを確認したら `Ctrl+b → d` でデタッチしてSSHを切断。

翌日SSHで再接続して `tmux attach -t experiment_01` で戻り、学習が正常に進んでいるか確認する。終わっていたらログを確認して `exit` でセッションを閉じる。

この習慣を徹底するだけで「SSH切れて学習が飛んだ」という事故がゼロになるので、研究室では必ずtmux内で実行することがルールになっている。
### デバイス転送
- `PyTorch`ではテンソルとモデルが同じデバイスにいないと計算できない
- `model.to(device)`でモデルのパラメータをGPUに送り、学習ループの中で`images.to(device)`,`labels.to(device)`でデータもGPUに送る
- 片方を忘れると`expected all tensors to be on the same device`というエラーが出る
### configファイル
- 学習率、バッチサイズ、エポック数、モデル名などのハイパーパラメータをハードコードしていると、パラメータを変えるたびにコードを書き換えることになる
- `yaml`ファイルに外だしすることで、コードは1つのまま`config`ファイルだけ差し替えれば実験条件を変えられる
- 実験ごとに`config`ファイルに保存しておけば、後から結果を再現できる
