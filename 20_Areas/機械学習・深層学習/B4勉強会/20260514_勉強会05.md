## 籠_SpeechLLM

### SpeechLLMとは何か

#### 定義

- **音声入力をLLMが理解できる形に変換し、音声から理解・推論・応答を行う枠組み**のこと
- **狭義**：音声を入力してテキストを出力するもの（音声認識、感情認識など）
- **広義**：音声を入力して音声を出力する対話システムや、双方向同時通話（full-duplex）まで含む
#### 音声が持つ情報

- 音声はテキストより豊富な情報を含む

|種類|内容|例|
|---|---|---|
|言語情報|何を言ったか|単語・文法|
|**パラ言語情報**|**どう言ったか**|**感情・声色・話速**|
|**非言語情報**|**周囲で何が起きているか**|**環境音・ノイズ**|

- 後の2つが重要で、テキストに変換した時点で失われやすい情報
- 「大丈夫です」という言葉でも、泣きながら言っているのか笑いながら言っているのか、テキストだけでは判断できない
#### 発展の流れ

```
従来型（ASR→NLU→TTS）
  ↓ LLMを導入
Cascade型（ASR→LLM→TTS）
  ↓ 音声を直接LLMに統合
End-to-End型（音声トークン→LLM→音声）
```
### Cascade型

#### 基本パイプライン

- 既存の技術を**3つ直列につなぐ**方式

```
入力音声 → [ASR: 文字起こし] → [LLM: 応答生成] → [TTS: 音声生成] → 出力音声
```
#### 利点

- 既存のWhisper（ASR）やChatGPT（LLM）などをそのまま流用できる
- モジュールを個別に交換・改善しやすい
- モジュール単体で評価できる
#### 課題（重要）

- **ASR誤りの伝播** 文字起こしが間違えると、後段のLLMがいくら賢くても誤った前提で回答してしまう
- **音声固有情報の損失** テキスト化の時点でパラ言語・非言語情報が消える、感情や声色がわからなくなる
- **逐次処理による遅延** ASR完了→LLM生成→TTS生成という3ステップを全部待つ必要があるため、応答が遅れる。相槌のような即時反応が難しくなる。
- **Turn-takingの難しさ** これが対話においては特に根本的な問題

#### Turn-takingとは

- 「**いつ聞くか・いつ話すか・割り込むか**」を制御する問題
#### 具体例：AudioGPT

- ChatGPTを司令塔として、ASR・TTS・各種音声処理モデルを接続したシステム
- LLMがタスクを判断し、適切なモデルに処理を投げる設計
### End-to-End型

#### 基本発想

- **音声をテキストに変換せず、音声特徴や音声トークンをLLMの内部で直接扱う**こと
- Cascade型の課題を克服しようとする方式
#### 2種類のアーキテクチャ

**Speech-aware text LM（音声理解向き）**

```
入力音声 → [Encoder] → [Adapter/Projector] → [LLM] → 出力テキスト
```

- 音声をエンコードしてLLMに流し込む
- 出力はテキスト

**Speech+text LM（音声対話・生成向き）**

```
入力音声 → [Tokenizer/Encoder] ┐
入力テキスト → [Tokenizer]     ┤→ [LLM] → [Decoder/Vocoder] → 出力音声
                                           → [Decoder]         → 出力テキスト
```

- 音声もテキストもトークン化してLLM内部で統合し、音声を直接出力

#### Cascade型との比較まとめ

| 観点   | Cascade型        | End-to-End型             |
| ---- | --------------- | ----------------------- |
| 構成   | 分離（ASR+LLM+TTS） | LLM内部に統合                |
| 実装   | 簡単              | 難しい                     |
| 音声情報 | テキスト化で失われる      | 保持できる可能性あり              |
| 遅延   | 増えやすい           | 低遅延化の可能性あり              |
| 対話性  | Turn-takingに弱い  | Full-duplexで自然な対話の可能性あり |

#### 具体例：Moshi

- End-to-End型の代表例
- 音声とテキストを同一モデル内で同時に扱うリアルタイム音声対話モデル
- 4つの主要コンポーネントからなる

**① Mimi（音声Codec）** 音声波形 ↔ 離散トークン列の双方向変換を担います。24kHzの音声を12.5Hzのトークン列に圧縮します（1ステップ≒80ms）。8つのコードブックを持ち、第1が意味情報（semantic）、残りが音質情報（acoustic）を担います。

**② RQ-Transformer** 時間方向（過去の会話文脈）と同一時刻内の複数トークン（text→semantic→acoustic）を2段階で生成します。

- **Temporal Transformer**：過去の会話文脈を扱う
- **Depth Transformer**：同一時刻内の複数トークンを階層的に生成

**③ Helium（テキストLLM backbone）** 7Bパラメータ規模のテキストLLMで、知識・推論・質問応答能力を担います。Moshiの基盤となっています。

**④ Multi-stream（full-duplexの核心）** ユーザ音声とMoshi音声を**別々のストリームとして並列にモデル化**します。これにより「ユーザが話し終わるのを待ってから返答する」必要がなくなり、割り込みや相槌を自然に扱えます。

加えて、**Inner Monologue**という仕組みがあり、Moshi自身の発話についてはtext tokenを先に生成してから音声tokenを生成することで、言語的一貫性を高めています（外部ASRに依存しないため、ユーザ側streamにはtext tokenを置きません）。

#### 音声の表現方法（補足）

- End-to-Endモデルが音声を扱う際の表現は大きく2種類ある

|表現|形式|長所|短所|用途|
|---|---|---|---|---|
|**連続表現**|encoderの埋め込みベクトル|音声情報を豊富に保持|Adapterが必要|感情・話者識別など理解タスク|
|**離散表現**|量子化されたトークン列|LLMのnext-token predictionに乗せやすい|量子化で情報損失|音声生成・対話|

### 現在の課題

#### ① 音声入力による意味推論能力の劣化

同じ内容でも、テキスト入力と音声入力では正解率が違います。Qwen2.5-OmniのSLU（Spoken Language Understanding）のOverall Accuracyを見ると、テキスト入力より音声直接入力の方が低い傾向があります。音声の特殊な表現空間がLLMの推論能力を一部打ち消してしまう可能性があります。

#### ② Instruction sensitivity（指示文感度）

同じ音声・同じタスクでも、指示文（プロンプト）の表現を変えるだけで性能が大きく変動します。Qwen2-AudioはASR（文字起こし）タスクで、デフォルト指示ではWER（誤り率）4.89なのに複雑な指示だと72.20まで劣化しています。テキストLLMでも同様の問題はありますが、音声LLMではより顕著です。

#### ③ データセット・評価の難しさ

- ASRのWER（単語誤り率）のような単一指標では性能を表せない
- 音声対話では「言語的知能・音声品質・遅延・対話性」など多軸で評価が必要
- 感情理解や割り込み対応の評価が特に難しい
- 高品質な多ターン音声対話データが不足している
### まとめ

> **音声対話AIは「テキストをしゃべるだけ」の段階から、「音声そのものを理解・生成する」段階へ移行しつつあるが、推論能力の維持・指示追従性・評価方法など解くべき課題がまだ多い**

## 岩本_VLM
### VLMとは何か

#### 定義

- **画像と言語を同時に扱えるマルチモーダルAIモデル**のこと
- 従来、AIの研究は大きく2つの分野に分かれていた

|分野|得意なこと|出力|
|---|---|---|
|**CV（Computer Vision）**|物体の検出・分類|ラベル・座標|
|**NLP（自然言語処理）**|翻訳・要約・対話|テキスト|

- VLMはこの2つを統合し、「**画像を見て言葉で説明する**」「**画像について質問に答える**」といったことが可能に
#### VLMが解ける代表的なタスク

- **ゼロショット画像分類**：事前にラベルを用意しなくても「これは猫ですか？」と自然言語で聞いて分類
- **画像キャプション生成**：画像の内容を自動的に文章で説明
- **VQA（視覚的質問応答）**：「このバナナの値段は？」「この標識には何と書いてある？」など
- **OCR・チャート理解**：画像内の文字やグラフを読み取る
- **物体検出・セグメンテーション**：「どこに」「何が」あるかをピクセル単位で認識
- **画像テキスト検索**：テキストで画像を検索したり、その逆をしたりする
### VLMの基本構造

- VLMには大きく**2世代のアーキテクチャ**が存在
#### 従来型アーキテクチャ（ゼロから学習するタイプ）

```
テキスト → [Text Encoder] → テキストベクトル ┐
                                              ├→ [対照学習] → VLM
画像   → [Vision Encoder] → 画像ベクトル    ┘
```

- 対照学習がコア
- 「犬」というテキストと犬の画像を、数学的な空間上で**近い位置に**配置し、関係ない組み合わせは**遠い位置に**なるよう学習
- これにより、「犬の画像≒犬というテキスト」という対応関係をモデルが覚える
- **メリット**：画像とテキストの対応を基礎から学ぶため非常に強固
- **デメリット**：天文学的な量のデータと計算資源が必要。開発コストが非常に高い
#### 最新アーキテクチャ（既存LLMを活用するタイプ）

- 現在の主流
- 「すでに賢いLLMに、目を与える」発想

```
テキスト ──────────────────────────→ [LLM] → 出力テキスト
                                       ↑
画像 → [Vision Encoder] → [Projector] ┘
```
#### コンポーネント①：Vision Encoder（視覚の抽出器）

- 画像のピクセルデータを、意味を持つ特徴ベクトルに変換
- 現在の主流技術は**ViT（Vision Transformer）**
1. 入力画像を**小さな正方形のパッチ（例：16×16ピクセル）に分割**
2. 各パッチを「パズルのピース」として扱い、**TransformerのSelf-Attention**で「このパッチとあのパッチはどう関係しているか」を計算
3. 出力として、「ここにふわふわした物体がある」「ここは尖っている」といった**数百個の視覚特徴ベクトル**が得られる
- 単なる色の集まりだった画像が、**意味のある特徴表現**に変換される
#### コンポーネント②：Text Encoder（言語の抽出器）

- 自然言語テキストを意味を持つベクトルに変換
- 従来型では独立したモジュールとして重要だが、最新型ではLLM内部の埋め込み層がこの役割を兼ねる
- BERTに近いTransformer構造を使い、文章をトークンに分割してベクトル化
- **対照学習で画像ベクトルと同じ空間に配置**するために必要

#### コンポーネント③：Projector（異文化の翻訳機）

- VLMの最新アーキテクチャにおける**最重要パーツ**
- 次元も、意味空間もまったく異なるため、Vision Encoderが出力した視覚ベクトルを、そのままLLMに渡しても理解できない
- Projectorは視覚ベクトルを「LLMが読める言語」に変換する

|種類|特徴|
|---|---|
|**Linear Projector**|シンプルな全結合層。軽量だが情報の圧縮はしない|
|**Perceiver Resampler**|固定長のトークンに圧縮。可変長の入力に対応|
|**Q-Former**|学習可能なクエリで必要な情報だけを抽出。最も高度|
#### コンポーネント④：LLM（推論エンジン）

- Projectorから来た視覚情報とテキストを統合し、最終的な回答を生成
- LLMの視点からは、画像トークンもテキストトークンも**同じトークンの列**として見えている

```
入力イメージ：
[画像トークン1][画像トークン2]...[画像トークン256] + "この画像に何が写っていますか？"
                   ↓
LLMが処理
                   ↓
出力："犬が写っています。"（1単語ずつ確率的に生成）
```

### 背景と歴史（5つの発展フェーズ）

#### 第1期（2021年〜）：視覚と言語の「共通言語」の獲得

- CLIP（OpenAI）、SigLIPが代表例
- 超大規模な「画像＋テキスト」ペアを使った対照学習で、画像とテキストを同じ意味空間に配置
- テキストを「生成」することはできず、検索・分類に留まった
- モダリティ（画像と言語という異なる種類のデータ）の壁を越える「アライメント基盤」を業界に提供した
- 後の全てのVLMの土台に

#### 第2期（2022年春）：文脈理解とマルチモーダル対話

- Flamingo（DeepMind）が代表例
- ブログ記事のように「文章と画像が交互に出てくるWebデータ」で学習し、文脈を理解できるようにした
- 動画フレームや高解像度画像を扱うと、視覚特徴の長さが可変になりLLMの負荷が膨大になることが課題
- Flamingoはこれを**Perceiver Resampler**で解決
- あらかじめ用意した固定数（64個）の「学習可能なクエリ」が、大量の視覚情報からcross-attentionで重要なエッセンスだけを吸い上げ、常に固定長のトークンに圧縮

また**ゲート機構**という工夫もあります。Cross-Attentionの出力に「調整弁」を設け、学習初期値を0にすることで、最初は元のLLMと同じ振る舞いをし、学習が進むにつれて「画像情報をどれだけ取り込むか」を徐々に学習させました。これにより既存LLMの能力を壊さずに視覚能力を追加できます。

**最大の功績：** 画像とテキストを並べるだけで新タスクを学習できる（Few-shot能力）を証明。

#### 第3期（2023年初頭）：生成能力の獲得と「情報抽出」

- BLIP、BLIP-2が代表例
- 凍結された（学習済みで重みを固定した）画像エンコーダとLLMを、**Q-Former**という翻訳モジュールで接続しました。

**Q-Formerの学習は2段階です：**

**第1段階（LLMなし）：** 画像エンコーダとQ-Formerだけで、3つのタスクを同時に最適化します。

- **ITC（対照学習）**：画像特徴とテキスト特徴を空間上で近づける
- **ITG（テキスト生成）**：クエリが抽出した視覚情報だけからテキストを生成させる
- **ITM（マッチング）**：画像とテキストのペアが一致しているかを二値分類する

この3タスクで、クエリが「テキストと関連性の高い視覚情報」を抽出できるよう鍛えます。

**第2段階（LLM接続）：** 第1段階で整列された画像トークンを線形層でLLMの次元に投影し、テキストトークンと連結してLLMに入力します。

**最大の功績：**「画像エンコーダ ＋ Projector ＋ LLM」という**現代VLMの基本アーキテクチャを確立**した。
#### 第4期（2023年中頃）：極限のシンプル化

- LLaVaが代表例
- 複雑なQ-Formerを捨て、**単一の全結合層（Linear Layer）だけ**でVision EncoderとLLMをつないだ
$$
H_v=W \cdot Z_v
$$
- Vision Encoderの出力Zᵥに重み行列Wを掛けるだけで、LLMが理解できる空間に変換する
- **メリット**：計算負荷が低く、情報の損失も少ない
- **デメリット**：パッチ数がそのままLLMに渡されるため、高解像度画像では計算コストが上がる
- **最大の功績：**「LLMが賢ければ、つなぎ目はシンプルで十分」ということを証明し、高性能VLMの開発コストを大幅に下げた。
#### 第5期（2024年〜）：高解像度・動画・ネイティブ対応

- Qwen2-VL、InternVLシリーズが代表例
- 画像を固定サイズにリサイズせず、**任意の解像度・アスペクト比のまま**処理できるようにした
- さらに長時間動画の理解にも対応
- 画像を動的にパッチ分割して直接入力（リサイズなし）
- 高解像度・動画化による「視覚トークンの爆発的増加」を防ぐため、意味を保ったままトークンを圧縮
- 2D空間の相対位置や、動画のタイムスタンプ・フレーム順をモデル化
- リサイズによる情報欠落がなくなり、極小の文字（OCR）や細部の認識精度が飛躍的向上
- 長時間動画も連続した視覚情報として処理可能に
- 現実世界の画像・動画をありのままで捉えることが可能になり、高度な視覚推論と計算効率の両立を実現。
### まとめ

> **「目（Vision Encoder）で見て、翻訳機（Projector）で言語化し、脳（LLM）で考える」という構造が現代VLMの本質であり、その翻訳機の進化がVLMの歴史そのものだった**

```
CLIP（2021）：まず画像と言語を同じ空間に配置することを学んだ
     ↓
Flamingo（2022）：文脈理解とFew-shot能力を獲得した
     ↓
BLIP-2（2023初）：現代VLMの基本アーキテクチャを確立した
     ↓
LLaVa（2023中）：「シンプルでいい」ことを証明し、民主化した
     ↓
Qwen2-VL等（2024〜）：高解像度・動画に対応し、現実世界へ適用拡大
```
#### 今後の課題

- **ハルシネーションの抑制**
- **計算コストの削減**

## AI駆動開発

### AI駆動開発とは

**パラダイムシフト**

- 従来の開発は「人間が考えて、書いて、デバッグする」という流れ
- AI駆動開発では「人間が意図を伝えて、AIが生成して、人間がレビューする」に変化
- エンジニアの仕事の重心が「コードを書く」から「指示してレビューする」にシフト

**3つのレベル**

| レベル       | 内容           | 例                   |
| --------- | ------------ | ------------------- |
| L1 補完     | 行・関数単位で候補を出す | GitHub Copilot      |
| L2 対話     | チャットで相談・生成   | ChatGPT, Gemini     |
| L3 エージェント | タスク単位で自律実行   | Claude Code, Cursor |

**LLMとAIエージェントの違い**

- LLMは「テキストを入力したら出力が返る」1往復の生成エンジン
- エージェントはそれに「Plan → Act → Observe」のループが加わったもの
- 「テストを書いて」と指示した場合、LLMはコードを返すだけ
- エージェントはコードベースを探索→テスト作成→実行→失敗修正まで自走する
### 主要ツール図鑑

**各社ツールの整理**

- **Anthropic**: Claude Code（CLI）
- **OpenAI**: Codex CLI（CLI）
- **Google**: Gemini CLI（CLI）、Antigravity（IDE）
- **AWS**: Kiro（IDE、仕様駆動に特化）
- **Anysphere**: Cursor（IDE）

**CLI 3強の使い分け**

|ツール|強み|向いてる用途|
|---|---|---|
|Claude Code|複雑な多ファイル編集、計画力が高い|大規模リファクタ、複数ファイルをまたぐ改修|
|Codex CLI|高速・サンドボックス|単一ファイルの実装、PR自動化|
|Gemini CLI|100万トークンのコンテキスト、無料枠が大きい|大規模コードベースの調査|
- どのツールも「Webサーチ・MCP・ファイル/ターミナル操作・並列実行」は標準装備
- 差がつくのはその上に乗せるワークフローの設計思想
- Claude Code → Plan→Execute→Verifyループ、CLAUDE.mdで文脈を永続化
- Kiro → requirements.md / design.md / tasks.mdの仕様書駆動
- Cursor → クラウドエージェントで長時間タスクをオフロード
### 実践ノウハウ

**CLAUDE.mdとは**

- プロジェクトのルールを書いておくファイルで、Claude Codeが毎ターン自動で読み込む

```
## プロジェクト概要
## ディレクトリ構成
## コーディング規約（Python 3.11, 型ヒント必須など）
## 実験管理ルール（WandBのプロジェクト名、命名規則）
```

- ポイントは**200行程度に薄く保つこと**。毎回トークンを消費するので、大きな仕様は`spec/`フォルダに外出しして、CLAUDE.mdからポインタだけ張るのが推奨です。

**仕様駆動開発（Spec-Driven Development）**

「Vibe Coding（雰囲気で指示する）」の対極にある考え方です。

- Vibe Coding → ふんわり指示→AIが空気読んで生成→レビューせず採用。500行を超えると破綻しがち
- 仕様駆動 → 仕様書を先に書く→AIが仕様に従って生成→仕様とコードを継続的に同期

仕様書の構造はどのツールも共通で「**What（要件）→ How（設計）→ Steps（タスク分解）**」に収束します。

研究での応用例：実験スクリプトを書く前に「データ仕様・評価指標・期待出力」をspec.mdに書いてからAIに投げると、ズレた実装が減ります。

価値が下がる：定型コードを速く書く能力、APIの暗記

## Appendix: DS向けAI駆動の現状

SWE（ソフトウェア開発）とDS（データサイエンス・研究）では、AI駆動の進み方が異なります。

**なぜDSはSWEより難しいか**

SWEは「テストが通るか」で機械的に正誤判定できるため、AIが自律的にPlan→Act→Observeループを回せます。一方DSは「この分析が正しいか」の判定が本質的に難しく、「コードが動く」と「結論が正しい」の間に大きなギャップがあります。ドメイン知識を持つ人間のレビューが不可欠です。

**DS向けツールの4レイヤ**

|レイヤ|代表ツール|
|---|---|
|ノートブック内蔵AI|Deepnote Agent, Hex, Cursor for Jupyter|
|対話型データ分析|ChatGPT Advanced Data Analysis, Claude|
|実験管理|W&B, MLflow, Neptune.ai|
|研究プロセス自動化|Sakana AI Scientist v2, Elicit|

Sakana AI Scientist v2はNature掲載・ICLR採択まで達成しましたが、独立評価では42%の実験がコード実行で失敗・文献の誤分類・ハルシネーションなども報告されており、まだ「当面は半自動化が現実解」という結論です。
