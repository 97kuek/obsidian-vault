# 情報通信ネットワークA：DNS

- 親ノート: [[情報通信ネットワークA]]

## DNS（ドメイン名と名前解決）

### なぜDNSが必要になったのか

#### 「名前」と「アドレス」は別物

- インターネット上の通信先は本来 **IPアドレス**（`172.26.235.197`）で指定
- しかし数字の羅列は人間には覚えにくい
- そこで端末に**わかりやすい名前 = ホスト名**をつけたい、という動機が生まれた
- ホスト名はPCなどの端末につける名前
- 名前をつけただけでは通信できない
- 最終的には **名前 → IPアドレス** の変換が必要で、そのための**対応表**をどこかで管理する必要がある

#### 初期の方式:HOSTSファイルによる集中管理

- インターネット黎明期は、名前とIPアドレスの対応を書いた **HOSTSファイル** を各端末が手元に持っていた
- このファイルは **SRI-NIC**が一括管理・公開していた(=**集中管理**)
- 各端末はそれを **FTPでダウンロード**して使っていた

#### 集中管理による問題点

- 問題はホスト数の爆発的増加
- 1981年に213台だったホストは、わずか数年で数十万台に
- この規模では**人手の集中管理は破綻**する
	1. **スケーラビリティがない** — 数が多すぎて一箇所で管理しきれない。名前が衝突したときの裁定も面倒。
	2. **障害に弱い(ロバスト性がない)** — 管理が止まると実質インターネット通信が止まる。これを **SPOF(Single Point of Failure:単一障害点)** と呼び、これを避けた設計が必須。

#### 解決策

- そこで導入されたのが**ドメイン名**
- 例:`www.comm.waseda.ac.jp`
- 名前空間を**ピリオドで区切って階層化**する
- これにより **① 衝突管理が容易**(同じ階層内でだけ重複しなければよい)、**② 分散管理が可能** になった
- この発想が **分割統治**
- トップが全部を管理するのではなく、階層ごとに置かれた組織に権限を**委任**し、各組織が自分の担当範囲だけを管理

### ドメイン名空間の階層構造

- ドメイン名は右から左へ階層が深くなる
- 一番右(の見えない部分)が頂点の **root**

```
                    root（頂点）
                   /     \
        TLD →    jp       com
                / \         \
       2LD →  ac   co       google
              /
       3LD → waseda
              /
            cs / comm
              /
            www / nsl
```

- `nsl.cs.waseda.ac.jp` は `cs.waseda.ac.jp` のサブドメイン
- `cs.waseda.ac.jp` は `waseda.ac.jp` のサブドメイン … というように、上位ドメインの中に下位(サブドメイン)が入れ子になっている
- この階層構造のおかげで名前付けの自由度が上がる
- サブドメイン内で衝突しなければよいので、`www.cs.waseda.ac.jp` と `www.comm.waseda.ac.jp` のように `www` を重複して使える

#### トップレベルドメイン(TLD)の種類

- 一番上の階層(`.jp` や `.com`)が **TLD** で、5種類に分類される

| 種類                      | 意味             | 例                            |
| ----------------------- | -------------- | ---------------------------- |
| **ccTLD**(country code) | 国コード(ISO国コード)  | `.jp` `.kr` `.cn` `.us`      |
| **gTLD**(generic)       | IANAが管理する汎用    | `.com` `.net` `.org` `.info` |
| **sTLD**(sponsored)     | gTLDのうち特定組織用   | `.coop` `.edu` `.gov` `.mil` |
| **IDN ccTLD**           | 国際化したTLD       | `.日本` `.韓国`                  |
| **infrastructure TLD**  | 1つだけ存在・逆引き等に利用 | `.arpa`                      |

#### JPドメインの2LD(セカンドレベルドメイン)

- `.jp` の直下の階層には、組織の種別を表す2LDがある

|2LD|由来|用途|
|---|---|---|
|`ac`|academic|大学等|
|`co`|corporation|企業|
|`ed`|education|学校(大学以外)|
|`go`|government|政府機関|
|`or`|organization|法人・組織|
|`ne`|network|ネットワークサービス提供組織|
|`gr`|group|任意団体・個人|
|`ad`|administrative|JPNIC会員|
|`lg`|local government|地方公共団体|

- このほか地域ドメイン(`tokyo.jp` など)や汎用ドメイン(`waseda.jp` など)もある
- ドメイン名はURL(`https://www.waseda.jp`)、メールアドレス(`tmori@waseda.jp`)など、ホスト名を指定するアプリ全般で使われる

### ドメイン名の登録・管理

#### レジストリ・レジストラ・レジストラント

- ドメイン名空間の各階層には管理組織が存在
- 特にTLDの管理をめぐって3つの主体が登場する

|主体|役割|
|---|---|
|**レジストリ**(registry)|TLDを管理する組織。ドメイン名DBの大元を運用|
|**レジストラ**(registrar)|レジストリから登録業務を委任された企業(窓口)|
|**レジストラント**(registrant)|ドメイン名を取得する登録者(ユーザー)|

1. レジストラント(`example.jp` が欲しい)
2. 複数のレジストラに問い合わせ(「登録料1000円」「1200円」「取り扱いなし」など条件が違う)
3. 選んだレジストラ経由でレジストリのDBに登録、という流れ

- **レジストリの役割**は、ドメイン名DBの運用管理、登録ポリシーの策定、登録申請の受付、**ネームサーバの運用**、Whoisサービスの提供など

|分類|TLD|レジストリ|
|---|---|---|
|gTLD|`.com` `.net`|VeriSign Inc.|
|gTLD|`.org`|Public Interest Registry|
|ccTLD|`.jp`|JPRS(日本レジストリサービス)|
|ccTLD|`.cn`|CNNIC|

#### 登録時に何を登録するのか

- ドメイン名を登録するときは、ドメイン名だけでなく**そのドメインを管理するネームサーバの情報**も登録する
- 登録ドメイン名:`example.jp`
- ネームサーバ:`ns.example.jp`(IPアドレス `172.16.4.2`)
- このネームサーバが実際にDNSサーバとして動作し、そのドメインの情報を返す担当になる

#### インターネットのガバナンス:ICANN / IANA

- 世界全体で統一ルールがないとシステムが破綻するため、**ガバナンス**が重要
- **ICANN**(The Internet Corporation for Assigned Names and Numbers):米国で設立された非営利法人。ドメイン名やIPアドレスなどの識別子をグローバルに管理・調整
- **IANA**(Internet Assigned Numbers Authority):実際の割り当て業務を行う組織機能。ICANNの子会社PTIが運用
- IANAは**DNSルートサーバの運用・調整**や**ポリシー策定**を担う

#### ルートサーバ

- ドメイン名階層の頂点(root)に対するネームサーバが**ルートサーバ**
- 各TLDのネームサーバの情報を管理している(=「`.jp` のことは誰に聞けばいいか」を知っている)
- **論理的には全世界で13個**(A〜M)存在
- ただし**物理的には1000以上のインスタンス**が世界中に分散配置されている → きわめて高い冗長性でDNSを高信頼化(この分散には後述の **anycast** を使う)

### DNSによる名前解決

#### DNSとは

- **名前解決** = ドメイン名をIPアドレスに変換するシステム
- ほとんどのインターネットアプリはDNSなしには動かない
- 各階層のネームサーバが**協調して**動作する
- **DNSは地球規模の大規模分散システム**
- 仮に米国のDNSサーバが全部止まってもDNS全体は動き続ける
- ドメインごとにネームサーバが名前を管理し、下位ドメインへ管理を移譲でき、冗長性によってSPOFを回避している

#### DNSの3つの構成要素

|構成要素|種別|役割|説明|
|---|---|---|---|
|**スタブレゾルバ**|クライアント|情報を**要求**|OSやブラウザ内部で動く簡素なDNSクライアント。キャッシュも持つ|
|**フルレゾルバ**|サーバ|端末の**代理で名前解決**|組織やプロバイダが提供するサーバ。`8.8.8.8` のようなパブリックDNSもこれ|
|**権威サーバ**|サーバ|情報を**提供**|各ドメインのネームサーバ。委任されたゾーンの情報を保持・管理|

#### エンドユーザから見た基本動作

ユーザーがブラウザに `www.waseda.jp` と入力すると

1. ブラウザ内の**スタブレゾルバ**が「`www.waseda.jp` は?」と**フルレゾルバ**(いわゆるDNSサーバ/キャッシュサーバ)に尋ねる
2. フルレゾルバが `104.16.38.132` を返す
3. ブラウザはそのIPアドレスに向けてTCP通信(Port 443)を開始し、HTTPSで通信する

- ユーザーから見える範囲は「スタブ→フルレゾルバに聞いて、答えをもらう」だけ
- **面倒な探索はフルレゾルバが代行**している

#### フルレゾルバの動作

- `www.comm.waseda.ac.jp` を解決する例で、フルレゾルバが**rootから順に下へたどっていく**様子を追っていく

1. スタブレゾルバ → フルレゾルバへ「`www.comm.waseda.ac.jp` は?」と問い合わせ
2. フルレゾルバ → **ルートサーバ**へ問い合わせ。rootは「`.jp`/`ac.jp` のことは `e.dns.jp` などに聞け」と**次の場所を案内**(IPは答えない)
3. フルレゾルバ → **e.dns.jp**(jp/acの権威サーバ)へ。「`waseda.ac.jp` は `ns.cfi.waseda.ac.jp` に聞け」と案内
4. フルレゾルバ → **ns.cfi.waseda.ac.jp**へ。「`comm.waseda.ac.jp` は `katto.comm.waseda.ac.jp` に聞け」と案内
5. フルレゾルバ → **katto.comm.waseda.ac.jp**へ。ここでようやく `www` のIP = **`133.9.250.196`** が返る
6. フルレゾルバ → スタブレゾルバへ最終的なIPを回答

- **反復的(iterative)問い合わせ**
	- 各権威サーバは「答えそのもの」ではなく「**次に聞くべきサーバ**」を返すだけ
	- 最終段(`comm`の権威サーバ)だけが本当のIPを答える
	- 地道に root から下へたどっていくのがDNSの基本動作
- 探索の手間はすべて**フルレゾルバが代行**し、スタブレゾルバ(=ユーザー端末)は最初に1回聞いて最後に答えをもらうだけ
- フルレゾルバが最初にrootへ行けるのは、**Root Hints**(13台のルートサーバのIPを書いた初期設定ファイル)を持っているから
- 「最初の取っ掛かり」だけは固定で知っている

### DNSの機能

#### ① リソースレコード(Resource Record, RR)

- DNSはホスト名→IPの変換だけでなく、用途別に**様々な種類の情報**を提供
- 種類を**レコードタイプ**として定義

|タイプ|ID|意味|
|---|---|---|
|**A**|1|IPv4アドレス(最もよく使う)|
|**AAAA**|28|IPv6アドレス(quad A)|
|**NS**|2|ネームサーバ|
|**CNAME**|5|カノニカルネーム(別名)|
|**MX**|15|メール受信サーバ(mail exchange)|
|**PTR**|12|逆引き(pointer)|
|**TXT**|16|任意の可読データ(送信者認証などに利用)|

#### ② DNSメッセージ形式

DNSのメッセージは**先頭12バイトが固定のヘッダ**で、その後ろに可変長の本体が続く。

```
+----------------+----------------+
| クエリ識別番号  |     フラグ      |  ← 12バイト固定ヘッダ
+----------------+----------------+
|   クエリ数      |     応答数      |
+----------------+----------------+
|  権威RR数       |    追加RR数     |
+----------------+----------------+
| クエリ部（質問。可変長）          |
+---------------------------------+
| 応答部（応答RR。可変長）          |
+---------------------------------+
| 権威サーバ情報（可変長）          |
+---------------------------------+
| 追加情報（可変長）                |
+---------------------------------+
```

**クエリのデータ構造(`www.waseda.jp` のAレコードを引く例):**

ドメイン名は「**ラベルの長さ + ラベル文字列**」の繰り返しで表され、最後は `00` で終わる。

```
03 77 77 77   06 77 61 73 65 64 61   02 6a 70   00   00 01   00 01
 │  w  w  w     │  w  a  s  e  d  a    │  j  p   │     │       │
 3文字 "www"    6文字 "waseda"       2文字"jp"  終端  タイプ  クラス
                                                      (00 01  (00 01
                                                       =A)    =通常1)
```

- 先頭の数字 = **次に続く文字数**(`03`なら3文字)、ピリオドは使わず長さで区切る
- 末尾 `00` でドメイン名の終わり
- `00 01` = タイプ(ここではA)、`00 01` = クラス(通常は1)

**応答のデータ構造**はクエリと似ているが、**TTL(生存時間)**、リソースデータ長、リソース本体が加わる。応答例ではドメイン名の繰り返しを避けるため、先頭で `c0 0c`(pointer)を使って既出のドメイン名を参照するなどの圧縮も行われる。

#### ③ キャッシュとTTL

毎回rootからたどると非効率なので、**キャッシュ**で結果を再利用する。

- **キャッシュ** — 権威サーバから受け取ったデータを、フルレゾルバが一定期間メモリに保存しておく仕組み
- **TTL(Time To Live)** — そのデータをキャッシュとして保持してよい**最大時間**。各レコードに付いている
- **ネガティブキャッシュ** — 「存在しなかったドメイン名」という情報もキャッシュする(存在しない名前を何度も探しに行かないため)

> [!example] 演習（スライド84）
> `www.comm.waseda.ac.jp` と `www.waseda.jp` のTTLを比較する。TTLが長いほど更新の反映は遅いが負荷は軽く、短いほど変更を素早く反映できるが問い合わせ頻度は増える。このトレードオフがTTLの差の背景にある。

#### ④ 正引きと逆引き

|用語|意味|
|---|---|
|**正引き**|ドメイン名 → IPアドレス(通常の名前解決)|
|**逆引き**|IPアドレス → ドメイン名|

逆引きは「学籍番号から氏名を得る」イメージ。重要な注意点:

- 逆引きは**正引きとは別のメカニズム**を使うため、必ずしも**1:1に対応しない**
- 1つのIPに複数のドメイン名が対応することも、逆引きが存在しないこともある

- **逆引きの仕組み**
  - 特別なドメイン `in-addr.arpa` を使い、IPアドレスを上位ビットから1バイト単位でたどる。
  - `133.9.81.100` の場合、IPアドレスを逆順にした `100.81.9.133.in-addr.arpa` を用いる。
  - 各NSサーバのPTRレコードを正引きと同じ要領でたどり、`dhcp100.goto.info.waseda.ac.jp` のようなドメイン名を得る。

#### ⑤ その他の機能・拡張

|機能|内容|
|---|---|
|**anycast**|ルートサーバはanycastで運用 → 同じIPを世界中に分散配置し、高い冗長性と速い応答を実現|
|**IDN**|国際化ドメイン名(日本語などのドメイン)|
|**SPF / DKIM**|TXTレコードを使ったメール送信者認証|
|**DNSSEC**|DNS応答の改ざんを防ぐ署名の仕組み|
|**DoH / DoT / DoQ**|DNS通信をHTTPS / TLS / QUICで暗号化(盗聴・改ざん対策)|
