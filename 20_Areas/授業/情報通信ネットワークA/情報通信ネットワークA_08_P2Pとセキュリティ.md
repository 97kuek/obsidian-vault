# 情報通信ネットワークA：P2Pとセキュリティ

- 親ノート: [[情報通信ネットワークA]]

## P2P・セキュリティ技術

### P2Pとは何か

- **P2P (Peer-to-peer)** は各ピアが相互にタスクやリソースを融通し合うモデル（Peer＝対等な仲間）


|         | クライアントサーバ型 | P2P型      |
| ------- | ---------- | --------- |
| リソースの所在 | サーバに集中     | 各ピアに分散    |
| 役割      | 提供/受信が固定   | 各ノードが両方行う |
| 性質      | 中央集権的      | 非中央集権的    |

### オーバーレイネットワーク

- P2Pはアプリケーション層で独自の**オーバーレイ・ネットワーク**を構成する。ピア間通信にはTCP、UDP、QUICなどを利用できる
- 下に物理的なIPネットワークがあり、その上に重ねる二層構造
- 論理層で「隣のピア」でも物理層では何台ものルータを経由している、というズレがポイント

### P2Pの3つのアーキテクチャ

- インデックス情報を**誰が管理するか**で分類している
	- **ピュアP2P**：各ノードが索引や探索機能を分散保持する。参加時には既存ピアを知るためのブートストラップ手段が必要。例 Gnutella, Freenet
	- **中央集中型P2P**：中央サーバが索引を一元管理（実ファイル転送はピア同士）。Napsterなど
	- **ハイブリッド型P2P**：**スーパーノード**（安定的な高性能ピア）が索引を管理。例 Skype, Kazaa
- BitTorrentはトラッカーを使う構成に加え、DHTやピア交換も利用できるため、単純な中央集中型だけには分類できない

### P2Pの特徴と応用

- 一般ユーザの草の根的活動で構成
- 多端末の資源共有で高効率・高スケーラビリティ
- 一方で匿名性が高く悪用・検閲回避にもなる
- 応用：ファイル共有、コンテンツ配信、リアルタイム通信、暗号通貨など。特定サービスのP2P採用状況は時期によって変わる

### ビットコインとブロックチェーン

- Satoshi Nakamoto が2008年に発表（"Bitcoin: A **Peer-to-Peer** Electronic Cash System"）
- P2P＋ブロックチェーンで暗号通貨取引を実現
- **ブロックチェーン＝分散台帳技術**：取引履歴の台帳を中央集権でなく参加ノード全体で共有
- **構造**：各ブロックはトランザクション(TX)群と**前ブロックのハッシュ値**をヘッダに含む。1ブロック改ざんすると以降全ブロックのハッシュが連鎖的に変わり検知できる
- **マークル木**：ブロック内のTXをハッシュツリーで管理。TX改ざんで **Merkle root** が変わり、計算済み nonce が無効化 → 改ざん検出
- **Proof of Work (PoW)・マイニング**：nonce計算に膨大なリソースが必要。マイナーに成功報酬がBTCで支払われ、計算した証拠＝PoW

### 暗号の表現と歴史

```
平文 ──[暗号化:暗号鍵]──→ 暗号文 ──[復号:復号鍵]──→ 平文
```

- 代表：バーナム暗号（ワンタイムパッド）、共通鍵暗号（DES/3DES, AES, RC4）、公開鍵暗号（RSA, 楕円曲線暗号）
- **バーナム暗号／ワンタイムパッド**：XORを使う方式のうち、鍵が真にランダムで平文と同じ長さを持ち、一度しか使われない場合をワンタイムパッドと呼び、情報理論的安全性を持つ。同じ鍵を再利用すると安全ではない

### 共通鍵暗号 vs 公開鍵暗号

- **共通鍵暗号**：暗号化と復号に同じ鍵（対称鍵）。速いが**鍵配送問題**（鍵を安全に渡す手段が難しい）
- **公開鍵暗号**：公開鍵と秘密鍵のペアを使う非対称暗号。暗号化、鍵共有、電子署名などの用途があるが、すべての公開鍵方式が同じ操作を提供するわけではない。公開鍵の真正性を確認する仕組みも必要である

### RSA暗号

- Rivest, Shamir, Adleman に由来（1977年、2002年チューリング賞）。**巨大数の素因数分解の困難さ**が安全性の根拠。処理は遅いので**ハイブリッド暗号**で使う

### 電子署名

- **秘密鍵（署名鍵）で署名し、公開鍵（検証鍵）で検証する**仕組みである。「公開鍵暗号を逆向きに使う」という説明はRSAの直感には使えるが、ECDSAなど一般の署名方式には当てはまらない
- **仕組み**：メッセージのハッシュ値を秘密鍵で署名し、本人性検証と改ざん検知を実現（ハッシュはSHA-256、署名はRSA/DSA/ECDSA）

```
【署名側】 文書 →ハッシュ→ ハッシュ値 →秘密鍵で署名→ 電子署名
【検証側】 文書と署名を、公開鍵と署名アルゴリズムで検証
          検証成功なら、対応する秘密鍵の保持者が署名し、署名後に文書が変わっていないと判断
```

### 暗号学的ハッシュ関数

- 任意長のビット列を固定長に圧縮
- 主な性質は、入力の小さな変化で出力が大きく変わること、原像計算が困難であること、同じ出力を持つ異なる入力を見つけることが困難であること
- 有限長の出力なので衝突自体は必ず存在する。安全性は、実用的な計算量で衝突を見つけにくいことに基づく

### 応用 — TLS（ハイブリッド暗号の実例）

- 公開鍵暗号は安全だが遅い
- 共通鍵暗号は速いが鍵配送が問題
- → **鍵共有と認証に公開鍵技術を使い、データ本体は高速な共通鍵暗号で保護する**
- TLS 1.3の代表的なサーバ認証では、次の役割を分けて理解する
  1. クライアントとサーバが対応バージョン、暗号方式、鍵共有情報を交換する
  2. (EC)DHEなどから共有秘密を導出する
  3. サーバが証明書と電子署名によって、自身の公開鍵とハンドシェーク内容を認証する
  4. 導出した共通鍵で以後の通信をAEAD暗号により保護する
- TLS 1.2以前のRSA鍵交換ではpre-master secretをサーバ公開鍵で暗号化する方式もあったが、TLS 1.3では採用されていない
