# 情報通信ネットワークA：文字コードとWeb技術

- 親ノート: [[情報通信ネットワークA]]

## 文字コード技術

### 文字コードとは

#### 符号化(エンコード)という考え方

- 文字コードとは、**コンピュータで文字を処理するために、文字に番号を割り当てる規則・規格**のこと
- 文字コードが表すのは「あ」の形そのもの（図形情報）ではなく、**文字を抽象化した情報**。「あ」という文字の"概念"に番号を振っているのであって、フォントの見た目とは別物
- 符号化(coding)とは、文字を {0, 1} のビット列で表現すること

#### 文字集合 vs 符号化文字集合

|用語|英語|意味|
|---|---|---|
|**文字集合**|character set|システムで扱う文字の"集まり"。番号はまだ振っていない|
|**符号化文字集合**|coded character set|文字集合を定義し、**各文字にコード（ビット組合せ）を割り当てた**もの|

- 符号化文字集合は抽象文字へコードポイントを割り当てる。各コードポイントを示す表を**コードチャート**と呼ぶ
- 見た目が同じ文字列でも、合成済み文字と結合文字列のように複数のコードポイント列で表せる場合がある。比較時にはUnicode正規化が必要になることがある
- UnicodeとISO/IEC 10646は別組織が保守する規格だが、コードポイントの割り当ては同期されている

#### 符号化文字集合 vs 文字符号化方式

|用語|意味|
|---|---|
|**符号化文字集合**|コードを割り当てた文字の集合そのもの|
|**文字符号化方式**（文字エンコーディング）|コードポイントや符号位置を、保存・通信に使うコード単位やバイト列へ変換する方法|

後から「複数の文字集合を組み合わせたい」「互換性を保ちたい」という要求が出て、**符号化方式**が考案された。

#### ASCII (ISO/IEC 646)

- 最初に標準化された符号化文字集合。**ASCII** = American Standard Code for Information Interchange
- ANSI が策定、**1963年初版**。**7ビット**（128文字）。国際標準化したのが **ISO/IEC 646**（各国版が定義）
- **符号位置(code position)**：コード表の「列/行」。例 小文字 `m` は6列13行で「6/13」、ビット列 `1101101`（7ビット）

#### なぜ7ビットだったのか

- 当時は**データリンク層のコストが高く**、なるべく少ないビットで送りたかった
- 1バイト=8ビットのうち**最上位ビットをパリティチェック**（誤り検出）に使っていた
- バイナリは uuencode 等でいったん ASCII(7ビット) に変換して送信

### 制御文字と図形文字

- **図形文字**：画面に表示できる普通の文字／**制御文字**：改行など出力を制御する特殊コード

|制御文字|略号|ASCII|説明|
|---|---|---|---|
|ベル|BEL|0x07|ビープ音|
|水平タブ|HT|0x09|`\t`|
|改行|LF|0x0A|行送り|
|復帰|CR|0x0D|行頭へ移動|
|エスケープ|ESC|0x1B|後続文字を特別解釈（エスケープシーケンス）|

**改行コードはOSで違う**：LF(0x0A)＝UNIX系／CR+LF(0x0D 0x0A)＝MS-DOS・Windows／CR(0x0D)＝旧Mac OS。

#### 内部コードと外部コード

- **内部コード**＝システム内部の処理用／**外部コード**＝入出力用。両者は一致していなくてよい
- 現代の主要OSや言語処理系はUnicodeを扱うが、内部表現はUTF-8、UTF-16、UTF-32や独自表現など実装によって異なる。ファイルやWebではUTF-8が広く使われる

#### 文字コードの変遷

|段階|時期|内容|
|---|---|---|
|(1) ASCIIと各国対応|1960年代～|アメリカで誕生 → 他国へ|
|(2) 複数バイト文字集合|1970年代～|アジアの文字（漢字）への対応|
|(3) 国際符号化文字集合|1990年代～|世界で統一された文字へ|

### 代表的な符号化文字集合と符号化方式

#### (1) ASCII → 各国言語への対応

- ASCIIは英語向けなので各国が**各国版ISO/IEC 646**を作成（JIS X 0201＝日本カナ、DIN 66003＝独、BS 4730＝英ポンド記号）。一部を自国文字に置き換えたため後に互換性問題を起こす
- 8ビットに拡張した **ISO/IEC 8859シリーズ**（Latin-1…）でラテン用字系191文字に対応
- **JIS X 0201**：日本の出発点で「文字化け」の遠因。7ビット部はASCIIベースだが**`\`を`¥`、`~`をオーバーラインに置換**。8ビット目が1の領域に**半角カタカナ**を配置

#### (2) 複数バイト文字集合と ISO/IEC 2022

- 漢字は1バイト（256通り）では足らず**2バイト文字**を提案。各国版切替や多言語同時表示を解決する**符号化方式**が **ISO/IEC 2022**
- インターネット想定で当初は**7ビット符号**。**エスケープシーケンス（ESC=0x1B）で文字集合を切り替える**のが核心

**GL/GR領域**（1バイトを4区画に分ける）:

```
0x00–0x1F  CL  制御文字（左）
0x20–0x7F  GL  図形文字（左）← 7ビットで使える範囲
0x80–0x9F  CR  制御文字（右）
0xA0–0xFF  GR  図形文字（右）← 8ビット目が1
```

**94×94の2バイト文字集合**：各バイトとも `0x21`–`0x7E`（GL範囲94通り）を使い、94×94=8,836文字を表現。

#### JIS X 0208 と区点番号

- 日本語の漢字等を収めた**2バイト符号化文字集合**（ISO/IEC 2022準拠、1978制定）。**94行×94列**を**区点番号**（区＝行、点＝列）で表す
- 変換ルール（GLに呼び出した場合）：`バイト値 = 0x20 + 区点番号`
- 例「森」＝31区25点 → `0x3F39`（`00111111 00111001`）／「愛」＝16区6点 → `0x3026`

#### ISO-2022-JP ← メールで日本語を送る方式

- **ASCII + JIS X 0201ラテン + JIS X 0208** を混在できる**7ビット符号化方式**（半角カナは非サポート）。電子メール用に考案（**RFC 1468**、JUNETコード、村井純先生ら）
- エスケープシーケンスで切替：`ESC $ B`(1b 24 42)→JIS X 0208／`ESC ( B`(1b 28 42)→ASCII。例「森」: `1b2442 / 3f39 / 1b2842 / 0a`

#### (3) Unicode と ISO/IEC 10646

- 文字集合を切り替えるのではなく、**1つの大きな表に全文字を入れる**発想。同時期に2規格が並行：

||ISO/IEC 10646|Unicode|
|---|---|---|
|立場|国際標準（公的）|民間コンソーシアム|
|初期コード長|**4バイト**|**2バイト**|
|ISO/IEC 2022互換|あり|なし|

- 最終的に **ISO/IEC 10646 を Unicode に合わせる**形で妥協し同一コード表に。このとき日中韓台の漢字を統合（**CJK統合漢字**）→ JIS X 0208資産がそのまま使えない・字形差などの問題
- 2026年7月時点の正式版はUnicode 17.0.0である。次版のベータではなく、公開済みの正式版を基準にする
- **BMP（基本多言語面）**はU+0000–U+FFFFの範囲である。Unicode全体はU+10FFFFまであり、UTF-16ではBMP外をサロゲートペアで表す

#### UTF-16 と UTF-8

- **Unicode**＝コードポイント（文字集合）／**UTF-16・UTF-8**＝実際のバイト列への変換**方式**
- UTF-16：16ビット符号単位、拡張部はサロゲート（Javaの内部表現）／UTF-8：**可変長**

**UTF-8 の符号化規則**（暗記）:

|バイト数|コードポイント範囲|ビットパターン|
|---|---|---|
|1バイト|U+0000 – U+007F|`0xxxxxxx`|
|2バイト|U+0080 – U+07FF|`110xxxxx 10xxxxxx`|
|3バイト|U+0800 – U+FFFF|`1110xxxx 10xxxxxx 10xxxxxx`|
|4バイト|U+10000 – U+10FFFF|`11110xxx 10xxxxxx 10xxxxxx 10xxxxxx`|

- **1バイト文字（U+0000–007F）は ASCII とまったく同じ**＝後方互換
- バイト長で先頭バイトのパターンが決まり（2バイト→`110`、3バイト→`1110`）、**2バイト目以降は必ず `10`** で始まる → 途中バイトと先頭を区別できる

**「森」を UTF-8 でエンコード**:

```
① コードポイント   U+68EE
② 範囲判定         U+0800–U+FFFF → 3バイト
③ 2進数化         0x68EE = 0110 1000 1110 1110
④⑤ 4/6/6に分割    0110 | 100011 | 101110
⑥ 枠にはめる       1110 0110  10 100011  10 101110
⑦ 16進数化         E6 A3 AE
```

→ 「森」の UTF-8 は **E6 A3 AE**。

#### 絵文字

- もとは日本の携帯電話文化から普及し、Unicodeへ収録された。絵文字にはBMP内・BMP外の両方があり、1コードポイントとは限らない。肌色修飾子、異体字セレクタ、ゼロ幅接合子などを組み合わせた列で1つの表示を作る場合がある

## Web技術

### Webとは何か

- **WWW（World Wide Web）** は1989年にCERNのティム・バーナーズ＝リーが発明した、ハイパーリンクで相互接続された情報空間。通信は **HTTP/HTTPS**（アプリケーション層）
- **Web ≠ インターネット**：インターネットは下層（IP・TCP等）の基盤、Webはその上で動く**1つのアプリケーション**にすぎない

### HTTPとハイパーテキスト

- **HTTP**（HyperText Transfer Protocol）はWWWの通信プロトコル。ハイパーテキスト＝「参照付きの電子的文章」。文書記述には **HTML** を用いる
- HTTPはWeb以外（ヘルプ・辞書・電子書籍・PDF）でも使われる

### Webが革新的だった理由

- ハイパーリンクにより、**異なるサイト・メディア・人々をつなぐリンク構造**を提供。サイト間リンクが導線を作り（人気サイトに人が集まる）、広告収入モデルやソーシャルメディアを生んだ

### HTTPの歴史

|バージョン|時期|ポイント|
|---|---|---|
|HTTP/0.9|1991|最初期|
|HTTP/1.0|1991–1995 (RFC 1945)||
|HTTP/1.1|1996–1997 (RFC 2068)|長く主流|
|HTTP/2|2015、2022改訂 (RFC 9113)|バイナリフレーミングと多重化。元はGoogle SPDY|
|HTTP/3|2022 (RFC 9114)|HTTPの意味論を、UDPベースのQUIC上で転送|

### URL と URI

- **URL**（Uniform Resource Locator）は上位概念 **URI**（Uniform Resource Identifier）の一例。構文 `scheme:[//authority]path[?query][#fragment]`
- 例 `https://nsl.cs.waseda.ac.jp/tokyo2020/`：`https`=scheme（暗号化http）／`nsl.cs.waseda.ac.jp`=ドメイン（ホスト）／`/tokyo2020/`=ファイルパス。＝「ホストのサーバからファイルパスのオブジェクトをhttpsで取得(GET)する」指定

### HTML

- Web文書の構造と意味を記述する**マークアップ言語**である。現在はWHATWGの **HTML Living Standard** が継続的に更新されている。W3CがHTML5を「無効な技術」にしたという意味ではなく、独立した版番号のスナップショットよりLiving Standardへ標準化の中心が移った

### HTTPリクエストメソッド（RFC 9110）

|メソッド|意味|
|---|---|
|**GET**|URLで指定したデータを要求（取得）|
|HEAD|GETと同じだがヘッダのみ取得|
|**POST**|データを送信（name, value の組）|
|PUT|指定ファイルを置き換える|
|DELETE|指定ファイルを削除|
|TRACE|リクエスト内容をそのまま返す（エコー）|
|OPTIONS|オプションの通知|
|CONNECT|HTTPトンネリング（proxy + SSL/TLS時など）|

### HTTPステータスコード（RFC 9110）

- 3桁の数字。**百の位で大分類**（2xx=成功, 3xx=リダイレクト, 4xx=クライアント側エラー, 5xx=サーバ側エラー）

| コード | 名称 | 説明 |
|---|---|---|
| 200 | OK | 成功 |
| 206 | Partial Content | 部分的なコンテンツ（途中から開始） |
| 302 | Found | 一時的なリダイレクト |
| 401 | Unauthorized | 有効な認証資格情報がなく、認証が必要 |
| 403 | Forbidden | サーバは要求を理解したが実行を拒否 |
| 404 | Not Found | リソースが存在しない |
| 500 | Internal Server Error | サーバ内部のエラー |
| 503 | Service Unavailable | 一時的に利用不可（高負荷など） |

- **401は認証の提示・再試行に関係し、403は認証済みかどうかにかかわらず要求を拒否する場合に使われる**

### WWWのアーキテクチャ

- 基本は**クライアント・サーバモデル**（**サーバ**＝リソース/サービスを提供、**クライアント**＝要求する、という非対称な関係）。サーバは到達可能なIPさえあればどこにあってもよい

### 発展技術が必要になった背景

- **規模**：利用者・サービス・トラフィックが大幅に増え、単一サーバでは可用性と性能を維持できない
- **質**：静的→動的コンテンツ、広告収入モデル、一極集中化
- 対応技術は (1)アーキテクチャの進化 (2)分散技術 (3)高速化 の3本柱

### (1) アーキテクチャの進化（三層モデル）

- 単純なHTML・画像配信なら**二層**（クライアント・サーバ）で十分だが、DB検索や動的生成には**三層アーキテクチャ**で処理を分散しスケーラビリティを上げる

|層|担当|実装例|
|---|---|---|
|プレゼンテーション層|データ表示（クライアント）|ブラウザ|
|アプリケーション層（ミドル層）|データ加工|Webサーバ Apache／アプリサーバ JavaEE, .NET, Rails, Node.js|
|データ層|データアクセス|DBサーバ MySQL, Oracle|

- 流れ：クライアントがリクエスト → Webサーバがクエリ → DBサーバが検索結果 → Webサーバがデータ加工してHTML出力
- **動的ウェブコンテンツ**：サーバ側（アプリサーバ）とクライアント側（**JavaScript**）の両方で動作。特に **AJAX** はクライアント側JSで**サーバとの非同期通信**を実現

### (2) 分散技術

- **ローカルな分散（サーバファーム）**：複数サーバのPCクラスタが単一サーバのように振る舞う。負荷分散に**ロードバランサ**や**DNSラウンドロビン**。**リバースプロキシ**でアクセス頻度の高いリクエストをフロントでキャッシュ
- **グローバルな分散（CDN）**：遅延、負荷、障害、契約などを考慮して配信拠点を選ぶ。コンテンツの事前配置やキャッシュ、DNS応答、anycastなど複数の仕組みを組み合わせる

### (3) 高速化

- 講義で引用される「100ミリ秒の遅延増加で売上が1%低下」という数値は、特定企業・時点の経験則として扱い、一般法則とはみなさない
- **① キャッシュ**：アクセスは偏る（**Zipf則**：少数のオブジェクトに集中）のでキャッシュが効く。制御は**HTTPの Cache-Control ヘッダ**（no-store／no-cache／public／private／max-age）などで行う。静的か動的かだけで決まるのではなく、応答内容と更新要件に応じてキャッシュ可否を指定する
- **② プロトコル改良**：HTTP/1.1パイプラインは先行応答待ちによるHOLブロッキングがあり普及しなかった。複数TCP接続による並列化を経て、HTTP/2は1接続上で複数ストリームを多重化した
- **HTTP/2が導入した概念**：バイナリフレーミング、多重化ストリーム、フロー制御、優先度、HPACKによるヘッダー圧縮、サーバープッシュ
- HTTP/2はHTTPレベルの逐次処理を改善するが、1本のTCP接続を共有するためTCPのHOLブロッキングは残る。HTTP/3はQUICの独立ストリームによって、あるストリームの損失が他ストリームを止める問題を軽減する
