# オペレーティングシステムA：デバイス管理

- 親ノート: [[オペレーティングシステムA]]
- 担当: 中島（OS内部・仮想化）

## I/Oアーキテクチャ

- CPU・メモリ・グラフィックメモリが**バス**でつながり、その先に**バスコントローラ**を介してデバイスがぶら下がっている
- CPU とデバイスのやり取りには、2つの経路がある
  - **I/O ポート**（データのやり取り口）
  - **割り込み**（デバイス → CPU への「イベント発生したよ」という通知）

## デバイスドライバ

```
アプリケーション
   ↓（標準インタフェース）
ファイルシステム／ネットワークシステム（カーネル内）
   ↓
デバイスドライバ
   ↓（デバイス独自プロトコル）
デバイス
```

- ポイントは、**アプリ側は標準インタフェースで統一されていて、デバイス側だけがデバイス独自のプロトコル**になっていることである
- つまりデバイスドライバが「翻訳係」となり、デバイスごとの違いを吸収してくれる
- 新しいデバイスを使いたければ、対応するドライバを差し込めばよい、という設計である

## I/Oインタフェース

- Unix では、デバイスも**ファイルのように扱う**
  - `/dev/device` のような名前でデバイスを表す
  - 各デバイスは**メジャー番号**と**マイナー番号**を持つ
  - カーネル内部ではこの番号でドライバを識別する（メジャー番号＝どの種類のドライバか、マイナー番号＝その中のどの個体か、というイメージ）

## デバイスアクセスの方法

- デバイスとやり取りするには、2つの観点がある

### (1) I/Oイベントの通知（デバイスの状態をどう知るか）

- エラー通知・データ利用可能通知・状態通知などを受け取る方法として、**ポーリング**と**割り込み**の2種類がある

### (2) データの転送（実際にデータをどう運ぶか）

- **Read/Write**：通常の読み書き
- **プログラマブル I/O**：CPU が1バイトずつ自分で運ぶ方式
- **DMA（Direct Memory Access）**：CPU を介さず、デバイスが直接メモリへデータを書き込む方式。大量データを運ぶとき CPU の負担を大きく減らせる

### デバイスのアクセス：2つの方式

- デバイスのレジスタにアクセスする方法も2通りある

|方式|しくみ|アクセス手段|
|---|---|---|
|**I/Oポート方式**|メモリ空間とは別に「I/O空間」があり、各デバイスにポートを割り当てる|専用命令（x86なら `in` / `out`）が必要|
|**Memory Mapped I/O**|デバイスの一部を「メモリの一部」として見せる|普通のメモリアクセス命令でOK（ただしアクセスコストは異なる）|

## ポーリング

- **ポーリング**は、「準備できた？まだ？」と**CPU が繰り返し確認し続ける**方式である

```c
while(flagPort != READY);   // 準備フラグが立つまでひたすら待つ
mem = dataPort;             // 準備できたらデータを読む
```

- この方法はシンプルだが、**待っている間ずっと CPU を使い続けるので無駄が大きい**のが欠点である
- 「他の処理の合間にチェックする」工夫も紹介されているが、それをやるとプログラムの構造が複雑で見通しが悪くなる

## 割り込み

- ポーリングの無駄を解決するのが**割り込み**である
- 考え方は次のとおり
  1. デバイスドライバは I/O を設定したあと、`wait(&device)` で**眠って（CPU を手放して）**他の処理に譲る
  2. デバイスの処理が終わると、デバイスが CPU に**割り込み**をかける
  3. **割り込みハンドラ**が起動し、`wakeup(&device)` で眠っていた処理を起こす

```
デバイスドライバ側          割り込みハンドラ側
I/Oポートの設定
wait(&device);   ←眠る    interrupt_handler() {
                              wakeup(&device);   ←起こす
                           }
```

- こうすれば、**待っている間 CPU は他の仕事ができる**ので、ポーリングのような無駄使いが起きない
- これが現代の OS で割り込みが広く使われる理由である
