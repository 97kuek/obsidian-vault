# オペレーティングシステムA：プロセス管理

- 親ノート: [[オペレーティングシステムA]]
- 担当: 中島（OS内部・仮想化）

## プロセスとは何か、どう作られるか

### そもそもプロセスとは

- **プログラム**と**プロセス**は別物である
  - **プログラム** … ディスク上に置かれた「命令の集まり（静的なファイル）」
  - **プロセス** … プログラムを実行する主体。仮想アドレス空間、CPU状態、開いているファイルなどの実行状態を持つ
- 実行待ちやI/O待ちのプロセスも存在するため、プロセスを単に「現在CPUで実行中のプログラム」と定義すると狭すぎる
- OS は複数のプロセスを**並行**に進められる。複数コアがあれば、複数プロセスを物理的に**並列**実行することもできる
- 複数のプロセスが**同じプログラム**を実行することもできる
- 各プロセスは**プロセスデータ構造**で管理される

### プロセスの実行環境

- まず単一CPUコアを考えると、そのコア上である瞬間に実行できるプロセスは1つであり、プログラムカウンタやスタックポインタなどの実レジスタも1組である
- OS が短い時間でプロセスを切り替えるため、複数のプロセスが並行して進んでいるように見える
- マルチコアCPUでは、各コアがレジスタを持つため、コア数までのプロセスを実際に並列実行できる。それでもプロセス数がコア数を超えれば時間分割が必要である

### プログラムが実行される仕組み

- 高級言語（C, C++）はコンパイラで機械語に変換され、リンケージでライブラリが結合され、最終的にこの形で実行される

### システムコール

- **「OS はどこにあるのか？」**
  - 答えは「OS もプログラムであり、メモリの中にある」
- ではアプリはどうやって OS の機能を使うのか
  - それが**システムコール**である
- ただし、ただの関数呼び出しとは異なる
  - OS の機能は**カーネルモード**という特権状態でしか実行できないからである
- システムコールの流れは次のとおり
  1. アプリは引数と**システムコール番号**を規約に従ってレジスタなどへ置き、システムコール用の命令を実行する
  2. トラップ命令によって**ユーザモードからカーネルモードに切り替わる**
  3. カーネル内の決まった番地（ディスパッチコード）にジャンプする
  4. システムコール番号を使って、適切な処理（open, close, fork など）に振り分ける

### プロセスの生成

#### fork（自分のコピーを作る）

- 親と同じ内容から開始する、独立した子プロセスを作る
- コード・データ・スタックは意味上複製されるが、通常は **Copy on Write** によって物理ページを一時的に共有する
- 子用のプロセス管理情報とページテーブルを用意し、CPUコンテキストを引き継ぐ。親子は別のPIDと別の仮想アドレス空間を持つ
- 戻り値が親と子で違う
  - **親プロセスには子の PID、子プロセスには 0** が返る
  - これによって「いま自分は親なのか子なのか」をプログラムが判定できる

#### exec（中身を入れ替える）

- 現在のプロセスの**プロセスイメージ（仮想アドレス空間の内容）を別のプログラムへ置き換える**
- execは新しいプロセスを作らない。成功後もPIDは変わらず、実行するプログラムだけが置き換わる
- 新しいコード・データ・スタックを構成し、実行ファイルのエントリポイントから実行を開始する
- 開いているファイル記述子は原則として引き継がれる。ただし `close-on-exec` が設定されたものは閉じられる

```
shell() {
    プロンプトの出力
    コマンドの入力
    if (fork()) {        // 親プロセス（戻り値=子のPID、つまり真）
        wait();          // 子の終了を待つ
    } else {             // 子プロセス（戻り値=0、つまり偽）
        exec(コマンドのファイル);  // lsなどに変身して実行
    }
}
```

- つまり「**fork で自分を複製し、複製した子の方を exec で目的のプログラムに変身させる**」というのが UNIX のプロセス生成の基本パターンである
- `ls > aaa`（出力をファイルにリダイレクト）のような処理も、fork の後・exec 前に `open` や `dup2` でファイルディスクリプタを差し替えることで実現される

### プロセスデータ構造

- 各プロセスは以下を持つ
  - **仮想アドレス空間**（コード・データ・ヒープ・スタックなど）
  - **保存されたCPUコンテキスト**（レジスタ、プログラムカウンタ、スタックポインタなど）
  - **PID**（プロセス識別番号、例：1010, 829）
- OSのプロセス制御ブロックは、これらの実体そのものを一箇所へ格納するのではなく、CPU状態、スケジューリング状態、アドレス空間への参照、開いているファイルなどの管理情報をまとめる

## どう切り替え、どう順番を決めるか

### 保存されたCPUコンテキストと実レジスタ

- 各プロセスについて、実行を再開するために必要なレジスタ値が**CPUコンテキスト**として保存される
- 単一コアの CPU がその瞬間に使う**実レジスタ**は1組である
- プロセス A の実行を再開する際、OSは保存されていたAのCPUコンテキストを実レジスタへ復元する
  - **時間的プロセス管理** … 1つの CPU を時間で分け合う。プロセスごとの仮想 CPU が、物理 CPU に順番にマッピングされる
  - **空間的プロセス管理** … 各プロセスの仮想メモリを、物理メモリに割り当てる（メモリ管理の話につながる）

### コンテキストスイッチ

- **コンテキストスイッチ**とは「いま実行中のプロセスを別のプロセスに切り替えること」である

#### いつ起きるか

- 実行中のプロセスが**ブロック**したとき（入力待ちなどで進めなくなる）
- 実行中のプロセスが**一定時間以上**実行されたとき（時間切れ）

#### 重要な原則

- コンテキストスイッチは**常にカーネル内で発生**する
- ブロックはシステムコール中に起こり、時間切れは割り込みによって起こるからである

#### Wait/Wakeupの仕組み

> ここでのWait/Wakeupは、カーネル内部の「待機状態へ移す／実行可能状態へ戻す」という概念的な操作である。子プロセスの終了を回収する `wait()` システムコールとは役割が異なる。

1. プロセス A を実行 → A が Wait を実行
2. **実レジスタの内容をAのCPUコンテキストとして保存**
3. **Bの保存済みCPUコンテキストを実レジスタへ復元** → プロセス B を実行
4. B が A を Wakeup（再び実行可能に）し、自分は Wait
5. 実レジスタをBのCPUコンテキストとして保存して、Aに戻る

- **Wait と Wakeup**という2つの操作が切り替えのカギである
  - Wait … 現在のプロセスを止めて他に実行を移す（→コンテキストスイッチ）
  - Wakeup … Wait で止まっているプロセスを再び実行可能にする（→スケジューリング）
- 一方で、**コンテキストスイッチのオーバヘッド（コスト）**も指摘されている
  - **パイプライン**が乱れる（CPU の命令先読みが無駄になる）
  - **キャッシュメモリ**が効きにくくなる（プロセスが多いと、各プロセスのよく使うデータをキャッシュに保てない）

### スケジューリング

#### プロセスの状態

- プロセスは3つの状態を行き来する
  - **実行中（Running）** … いま CPU を使っている
  - **実行可能（Ready）** … いつでも動けるが、CPU の順番待ち
  - **ブロック（Blocked）** … 入力待ちなどで動けない
- 主な遷移は次のとおりである
  - 実行可能 → 実行中：スケジューラに選ばれる
  - 実行中 → 実行可能：時間切れや、より高優先度の処理によってプリエンプトされる
  - 実行中 → ブロック：I/O完了などを待つ
  - ブロック → 実行可能：待っていたイベントが完了する
- 実行可能状態のプロセスは **Ready キュー**につながれて順番を待つ

#### Preemptive か Non-Preemptive か

- **Preemptive** … 実行中プロセスが自発的に待機しなくても、タイマ割り込みや優先度などを契機にOSがCPUを取り上げられる。応答性と公平性を確保しやすい
- **Non-Preemptive** … いま動いているプロセスが**自分でブロックするか終了するまで**切り替えない。シンプルだが、長く居座られると他が待たされる

#### スケジューリングアルゴリズム

- **Non-Preemptive 系**
  - **FIFO** … 到着順。来た順に処理
  - **SJF** … 実行時間が短いものを優先
- **タイムシェアリング系**
  - 代表例は**ラウンドロビン**である。実行可能プロセスへ短い時間片を順番に割り当てる
  - 実際のOSでは優先度や重みも組み合わせるため、必ずしも全プロセスへ完全に均等なCPU時間を与えるわけではない
- **優先度順**
  - キュー内を優先度順に並べ、高いものから実行
  - **UNIX スケジューラ**の巧妙な点
    - **I/O 待ちから復帰したプロセスは優先度が上がる**（対話的な処理をキビキビ動かすため）
    - **長く実行したプロセスは優先度が下がる**（CPU を独占させないため）
- **リアルタイム系**
  - プロセスが**周期的**に実行され、それぞれ**デッドライン**を持つ
  - **EDF** … 締め切りが最も早いものを最優先
  - **RM** … 周期が最も短いものを最優先

### スレッド

- プロセスの中で、**コードとデータは共有しつつ、PC・SP・スタックだけを別々に持つ**実行の流れである

#### なぜスレッドが必要か

- **スループット向上** … マルチスレッドサーバ（多数のリクエストを並行処理）
- **構造化** … 異なる I/O 処理に別々のスレッドを割り当てて整理する
- **高速化** … 計算を複数スレッドに分割して並列に処理

#### スレッドの操作（プリミティブ）

- `thread_create(関数のエントリ, スタックのポインタ)` … 生成
- `thread_exit()` … 終了
- `thread_self()` … 自分の ID 取得
