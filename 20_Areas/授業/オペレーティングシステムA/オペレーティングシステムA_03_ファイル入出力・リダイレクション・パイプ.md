# オペレーティングシステムA：ファイル入出力・リダイレクション・パイプ

- 親ノート: [[オペレーティングシステムA]]
- 担当: 内田（UNIX・システムコール）

## Everything Is a File

- UNIX の設計思想の象徴が**すべてはファイル**という考え方である
- 正確には、通常ファイル、端末、パイプ、一部のデバイスなどを、`open/read/write/close` とファイル記述子による**共通インターフェースで扱う**という意味である
- すべての資源が通常ファイルと同一という意味ではない。対象固有の操作には `ioctl` なども使われる

## ファイル入出力の基本システムコール

- Linux でのファイル操作は少数のシステムコールに集約されている
- Linux におけるファイル入出力は、次の5つの関数（システムコール）で実行される
  - open（ファイルを開く）
  - read（ファイルから読む）
  - write（ファイルに書く）
  - close（ファイルを閉じる）
  - lseek（ファイルの位置決め）
- 基本は「開く → 読む／書く → 閉じる」という流れである

## ファイル記述子

- これらの操作の鍵が**ファイル記述子（file descriptor）**という整数である
- 既存のファイルを開いたり新たなファイルを生成すると、カーネルはプロセスに対してファイル記述子を返す
- ファイルの読み書きを行いたい場合は、`open` から返されたファイル記述子を `read` や `write` の引数として指定しなければならない

## システムコールと標準入出力ライブラリ

- これら低水準のシステムコールの上に、もう一段使いやすい層がある
- 標準入出力ライブラリは、低水準入出力であるシステムコールへの様々な用途のインターフェースを提供する
- 標準入出力ライブラリは、ファイル記述子の上に**バッファリングや文字単位・行単位の操作を提供するストリーム抽象**を重ねる
- Cではストリームを `FILE *` で表す。`FILE *` とファイル記述子は同じものではないが、通常はストリームの内部から対応するファイル記述子を利用する
- プログラムが実行されると、次の3つのファイルストリームが自動的に開かれる
  - `stdin`（標準入力、ファイル記述子 0 に対応）
  - `stdout`（標準出力、ファイル記述子 1）
  - `stderr`（標準エラー、ファイル記述子 2）
- この「0 が入力、1 が出力、2 がエラー」という対応関係が、次のリダイレクションを理解する前提である

## リダイレクション

- リダイレクションは、標準入出力の接続先を付け替える操作である
  - `./a.out < infile` … 標準入力を切り替える（入力リダイレクト）
  - `./a.out > outfile` … 標準出力を切り替える（出力リダイレクト）
  - 両方同時に書けば入力と出力の両方を切り替えられる
  - `./a.out 2> outfile` … 標準エラー出力を切り替える
- 内部の手順は3ステップである
  1. 読み込み用／書き込み用にファイルを開く
  2. `dup2` などで、そのオープン済みファイルを標準入力／標準出力のファイル記述子へ複製する
  3. 複製元として使った余分なファイル記述子を閉じる。複製先は同じオープンファイルを参照し続ける
- この「関連付け」を実現するのが **dup() / dup2()** システムコールである
  - 例：`close(1)` で標準出力の記述子1を空けてから、開いたファイルをそこに複製する
  - あるいは `dup2(fd, 1)` で記述子1を一気に目的のファイルに付け替える
- `> outfile` の正体は、まさにこの記述子の付け替えである

## パイプ

- パイプはプロセス間で通信（データ交換）するための仕組みである
- あるコマンドの標準出力を別のコマンドの標準入力に接続する
- 書式は `cmd1 | cmd2`
- `ls | wc -l` では、シェルが `ls` と `wc` を実行して2つのプロセスをそれぞれ生成し、これら2つのプロセスがパイプで接続される
- 重要なのは、各プログラムが接続先を特別扱いせず、**標準入出力のファイル記述子に対して読み書きするだけでよい**点である
  - シェルが `ls` の標準出力をパイプの書き込み端へ接続する
  - シェルが `wc` の標準入力をパイプの読み取り端へ接続する
  - `ls` と `wc` は、標準入出力の接続先が端末かパイプかを意識せずに動作できる
