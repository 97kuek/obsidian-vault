# オペレーティングシステムA：ファイルシステム

- 親ノート: [[オペレーティングシステムA]]
- 担当: 中島（OS内部・仮想化）

## ファイルシステムが解決する2つの問題

- ファイルシステムの仕事は、突き詰めると**2つの名前解決**である
  1. **ファイル名の変換**：人間が使う文字列の名前を、内部の ID に変換する（例：`/a/b/c` → `101020`）
  2. **ファイル内ポインタの変換**：ファイル内の「論理アドレス（先頭から何バイト目）」を、ストレージ上の「物理アドレス」に変換する
- ストレージは、メモリと同じように `0` から `0xffffff` まで続く**リニアなアドレス空間**として捉える

### 論理アドレス→物理アドレスのマッピング方式

- ファイルの中身がディスク上のどこにあるかを管理する方式が3つある
  - **連続割り当て**：各ファイルの「先頭アドレス」と「ファイルサイズ」を記録する。シンプルだが、空き領域の断片化に弱い
  - **リンク方式**：「先頭アドレス」＋各ブロックに「次のブロックへのポインタ」を持たせ、数珠つなぎにする。途中のブロックへ飛びにくい
  - **ディレクトリ（インデックス）方式**：ブロックの位置を一覧表で管理する

### ディレクトリの構造

- ファイルは**木構造**で整理される

```
/
├── bin
├── usr ── local ── bin ── ls, netscape
└── home ── tatsuo, reikona
```

### UnixファイルシステムのAPI

- デバイスのときと同じ API が、ファイルにもそのまま使える

```c
fd = open(file, mode);
read/write(fd, buf, size);
lseek(fd, offset, mode);    // ファイルポインタの位置を移動
fstat(fd, statbuf);         // ファイルの情報を取得
close(fd);
```

- `lseek` は「ファイルポインタ（次に読み書きする位置）」を動かす命令である

## Unixファイルシステムの全体像

```
プロセスのデータ構造
   ↓
ファイルのデータ構造（オフセットを持つ）
   ↓
i-nodeテーブル ──→ i-node ──→ ファイルブロック（実データ）

ストレージ上には：
スーパブロック ／ i-nodeリスト ／ ファイルブロック
```

### i-node（アイノード）とは

- **i-node** は、1つのファイルにつき1つ存在する「管理情報の塊」である
- ファイルの中身そのものではなく、ファイルに関する情報を持っている
  - 所有者
  - アクセス権（誰が読み書きできるか）
  - アクセスタイム・変更時間
  - そして**実データの場所を指す10個のエントリ**（ファイルブロックへのポインタ）
- つまり、「ファイル名」ではなく「i-node 番号」がファイルの本当の正体である

### ファイルデスクリプタとファイルデータ構造

- プロセスがファイルを開くと、`fd`（ファイルデスクリプタ＝0, 1, 4… のような番号）が返される
- この番号と実データの間に、**オフセット（今ファイルのどこを読んでいるか）を保持するファイルデータ構造**が挟まる
- この仕組みのおかげで、**プロセス A とプロセス B が同じファイルを開いても、それぞれ別々のオフセット**を持てる
- 逆に、共有させることもできる、という柔軟性がここで生まれる

### スーパブロックとフリーブロック管理

- **スーパブロック**：そのファイルシステム全体の属性（ブロックサイズなど）を記録した、いわば「目次・台帳」
- **フリーブロック管理**：空きブロックを**ビットマップ**（0/1 の並び）で管理し、「どこが空いていてどこが使用中か」を高速に判定する

### バッファキャッシュ

- ディスクアクセスはメモリに比べて圧倒的に遅いので、**一度読んだブロックはメモリ上にキャッシュとして残しておく**
  - よく使うブロックは、2回目以降ディスクへアクセスせずに済む → 高速化
  - キャッシュがいっぱいになったら、**LRU** アルゴリズムで入れ替える

### ディレクトリ管理

- ディレクトリの実体も、実は**名前と i-node 番号の対応表が書かれた特殊なファイル**である
- 例えばルートディレクトリの中身は：

```
名前        i-node番号
.       →   2     （自分自身）
..      →   2     （親、ルートなので自分）
usr     →   3
bin     →   10
```

- `/usr/local` をたどるときは、「ルートの i-node → usr（i-node 3）の中身を見る → local を探す…」と、**名前と i-node 番号を1段ずつたどっていく**ことでファイルにたどり着く
- `.`（自分）と `..`（親）も必ず登録されているのがポイントである

### ファイルシステムのマウント

- 複数のディスクを1つの木構造にまとめる仕組みが**マウント**である
- 実装としては、**マウントテーブル**で対応関係を管理する
- マウントポイントになっているディレクトリの **i-node に「マウントフラグ」が立っている**ので、そこに来たら別のディスクへ切り替える、という仕組みである
- これによりユーザーは複数のディスクを意識せず、ひとつながりのディレクトリツリーとして扱える
