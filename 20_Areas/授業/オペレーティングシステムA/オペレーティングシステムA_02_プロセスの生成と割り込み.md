# オペレーティングシステムA：プロセスの生成と割り込み

- 親ノート: [[オペレーティングシステムA]]
- 担当: 内田（UNIX・システムコール）

## fork()

- プロセス生成の中心は fork() である
- fork() を呼び出したプロセス（親プロセス）は新たなプロセス（子プロセス）を生成する
- fork() は子プロセス用の独立した仮想アドレス空間を作り、両プロセスは fork 文の直後の命令から実行を継続する
- 意味上は親のメモリ内容が子へ複製されるが、現代のUNIX系OSでは通常、物理ページをすぐ丸ごとコピーせず **Copy on Write（COW）** で共有し、どちらかが書き込んだページだけをコピーする
- 返り値は次のとおり
  - **子プロセスには 0** を返す
  - **親プロセスには子プロセスの PID** を返す
  - **エラー時には -1** を返す
- fork() が実行されると子プロセスは親プロセスのアドレス空間の複製を受け取る
  - fork 前に親の PID を代入しておいた変数は子プロセスにもそのままコピーされる
  - 子側ではその値が getpid() の返り値（＝子自身の PID）と一致しない
- つまり fork 時点の変数の値は引き継がれるが、その後の変更は原則として相手のアドレス空間へ反映されない

## fork()のユースケースとexec()

- fork() の重要な使い道として、1つのプロセスで異なるプログラムを実行したい場合がある
- これはシェルでよく用いられる
- この場合、子プロセスは fork() の直後に exec() を実行する
- シェルはまず fork() で自分の子プロセスを作り、その子プロセスが exec() で目的のプログラム（たとえば `ls`）に「中身を入れ替わる」
- 親であるシェル自身はそのまま残り、フォアグラウンド実行なら子の終了を待つ。バックグラウンド実行では、プロンプトへ戻って別の入力を受け付けられる
- `fork → exec → exit`、親は `wait` で待つ、という流れが標準パターンである

### wait() / waitpid()と終了状態の回収

- 子が終了しても、親が終了状態を受け取るまでは、PIDや終了コードなどの最小限の情報がカーネルに残る。この状態を**ゾンビプロセス**という
- `wait()` / `waitpid()` には次の2つの役割がある
  - 必要なら、対象の子が終了するまで親を待たせて同期する
  - 終了した子の終了状態を受け取り、カーネルに残った情報を回収する
- 親プロセスは `wait()` あるいは `waitpid()` を実行することで、子の終了を待つだけでなく終了理由や終了コードも取得できる
  - `wait()` … 子プロセスのうちの1つが終了するまで親を休止させる
  - `waitpid()` … 指定した子を待つ。オプションにより、終了していなければ直ちに戻る非ブロッキング動作も選べる

## シグナルによる非同期通知

- シグナルは、**カーネルからプロセスへイベントを通知するUNIXの仕組み**である
- デバイスがCPUへ通知する**ハードウェア割り込み**とは別の層の概念である。どちらも通常の実行へ非同期に介入する点は似ているが、通知先と実装が異なる

### シグナルとは何か

- プロセスに対して「どのイベントが起きたか」を通知する仕組みである
- 各シグナルには名前がついており、整数値、もしくは `<signal.h>` で定義されたシグナル名 `SIGxxxx` で指定される

### シグナルの種類とデフォルトの動作

- シグナルには種類ごとに「何も設定しなければこうなる」というデフォルトの動作が決まっている

|シグナル|説明|デフォルトの動作|
|---|---|---|
|SIGFPE|算術例外（代表例：整数の0除算）が起きた|コアダンプ終了|
|SIGSEGV|セグメンテーション違反（不正なメモリアクセス）が起きた|コアダンプ終了|
|SIGINT|キーボードからの割り込み（Ctrl-C）が届いた|終了|
|SIGTSTP|キーボードからの一時中断（Ctrl-Z）が届いた|一時中断|
|SIGKILL|プロセスの強制終了|終了|

### signal()

- デフォルト動作は signal() 関数を使って変更できる。ただし実用コードでは、挙動を細かく指定できる `sigaction()` の利用が一般に推奨される
- signal() はシグナル受信時の動作を設定する関数で、指定した番号のシグナルにシグナルハンドラーを設定する

```c
void (*signal(int sig, void (*handler)(int)))(int);
```

- ハンドラーの指定方法は3種類ある
  - **SIG_IGN** … そのシグナルを無視する
  - **SIG_DFL** … デフォルトの動作を行う（元に戻す）
  - **ユーザが指定した関数** … そのシグナルを受け取ったときにその関数が呼び出される
- `SIGKILL` と `SIGSTOP` は捕捉・無視・動作変更ができない
- 動作の流れが重要である
  - シグナル `sig` を受け取ると関数 `handler` が呼び出され、`handler` の引数には届いたシグナルの番号が渡される
  - `handler` が呼び出されるときは、それまで実行していたプログラムの実行はいったん一時中断され、`handler` から戻ると実行を再開する
- **通常の処理中にシグナルが配送されると、処理が一時的に脇へそれてハンドラが実行され、ハンドラから戻ると元の処理を再開する**。ただし、終了・停止がデフォルト動作のシグナルでは再開しない

### kill()

- ここまでは「受け取る側」の話だったが、シグナルを「送る側」のための関数が kill() である

```c
int kill(pid_t pid, int sig);
```

- kill() は、`pid` で指定したプロセスまたはプロセスグループに、`sig` で指定したシグナルを送出する。実際に送れるかどうかはユーザIDや権限によって制限される
- 名前は「kill（殺す）」だが、必ずしも終了させるとは限らない点に注意する
  - SIGKILL を送れば強制終了になる
  - SIGTSTP を送れば一時中断になる
  - ユーザ定義のハンドラが設定されていればその処理が走る
- kill() は、権限の許す対象へシグナルを送る汎用的な手段である
- 第1回で学んだ PID が、ここで「どのプロセスに送るか」の宛先として効いてくる
