# オペレーティングシステムA：仮想メモリ

- 親ノート: [[オペレーティングシステムA]]
- 担当: 中島（OS内部・仮想化）

## なぜ仮想化するのか

- ひとつは**仮想化**＝各プロセスに「大きなアドレス空間を持っているように見せる」ことである
- もうひとつは、その**実現方法**として、MMU で仮想記憶を物理メモリへマッピングし、さらにオンデマンドページングによって「物理メモリより多くのメモリ」をアプリに使わせることである

## 仮想アドレス空間と物理アドレス空間

- **仮想アドレス空間（Virtual Address Space）**：各プロセスが**それぞれ独自に**持つ。同じ仮想アドレスでも、プロセスが違えば別の物理ページへ対応づけられる
- アドレス範囲はCPUアーキテクチャやOSによって異なる。32ビット環境の `0x00000000`〜`0xffffffff` は説明用の一例であり、全範囲が実際に使用可能とは限らない
- **物理アドレス空間（Physical Address Space）**：コンピュータに**1つだけ**存在する、実際の RAM
- **MMU**：仮想アドレス（VA）を物理アドレス（PA）へ変換する
- App A も App B も App C も、それぞれの「0番地から始まる仮想空間」を持ち、MMU がそれを物理メモリ上のバラバラの場所へ振り分ける（`PA = f(Pi, VA)`、Pi はプロセス番号）
- これによって**プロセス同士が互いのメモリを壊せない**という重要な性質が生まれる
- MMU は CPU が提供する機能であり、ユーザモードのメモリアクセスも含めてアドレス変換を行う
- ページテーブルの設定やMMUの制御レジスタの変更は特権操作であり、カーネルだけが実行できる
- OS はページテーブル基点などの設定を切り替えることで、「現在どのアドレス空間を使うか」を指定する

## フラグメンテーション

- プロセスを**連続した領域**にそのまま置くと、Process A/B/C が並んだ後に B が終了して隙間が空いても、その隙間が新しいプロセスより小さければ「合計の空きは足りているのに連続領域が取れず入らない」という事態が起きる
- これを**外部フラグメンテーション**という
- 解決策の1つが**コンパクション**（プロセスを詰め直して隙間を埋める）だが、大量のメモリコピーが発生して非現実的である

## ページとページテーブル

- **仮想アドレス空間を固定長のページ、物理メモリを同じ大きさのページフレームに区切る**（代表例は4KiB。より大きなページを併用するCPUもある）
- 仮想ページを、物理メモリ上の任意のページフレームへマッピングする
- 連続して置く必要がなくなるので、外部フラグメンテーションが原理的に消える
- このマッピング表を**ページテーブル**という
- 各プロセスが自分のページテーブルを持ち、MMU レジスタが「今のプロセスのページテーブルの場所」を指す

## ページテーブルエントリ（PTE）の中身

- ページテーブルの1行（エントリ）には、物理ページ番号のほかに次のビットが入る
- これが後の最適化すべての土台になる
  - **Present / Valid bit**：その変換が現在利用可能か。名称と厳密な意味はアーキテクチャによって異なる
  - **Dirty bit (D)**：そのページが書き換えられたか（追い出すとき書き戻しが必要か判断）
  - **Reference bit (R)**：最近アクセスされたか（追い出し候補の判断に使う）
  - **Protection bit**：Read / Write / Execute の許可

## ページテーブルをどこに置くか／TLB

- ページテーブル自体の置き場所には一長一短がある
  - **CPU レジスタ**は速いが数が限られ、プロセスが増えると退避が必要
  - **物理メモリ**に置くのが基本だが、テーブルが大きいと場所を食う
  - **仮想メモリ**に置くと「ページテーブルをマップするためのページテーブル」が要る、という入れ子問題が出る
- そして大事なのが速度である
  - 毎回ページテーブルを引くと、メモリアクセスが2回（テーブル参照＋本体）になり遅い
  - そこで **TLB（Translation Lookaside Buffer）** という変換結果のキャッシュを CPU 内に置く（`CPU → TLB → ページテーブル`）
- ただし TLB には注意点がある
  - **ページテーブルを書き換えても古い変換がTLBに残る場合がある**ため、OSは対象エントリの無効化やTLBフラッシュを行う
  - 仮想領域のマップ／アンマップやアクセス権変更では、影響する変換を無効化する必要がある
  - コンテキストスイッチ時は、単純なCPUではTLBをフラッシュする。ASIDやPCIDのようにアドレス空間を識別する機能があれば、異なるプロセスのエントリを共存させられる

## ページテーブルが大きすぎる問題と改良

- 仮想記憶が大きくなるとページテーブルも巨大化する
- ページサイズ1KiBの単純な1段ページテーブルを仮定すると、1プロセス分のエントリ数は次のようになる
  - 32ビット：$2^{32}/2^{10}=2^{22}$ エントリ。PTEが4バイトなら約16MiBである
  - 64ビット：$2^{64}/2^{10}=2^{54}$ エントリ。全仮想空間分を平坦な表で用意するのは現実的でない
- 実際の64ビットCPUでは、使用する仮想アドレス幅が64ビット未満の場合も多い。それでも平坦な表は巨大になるため、階層化などが必要である
- このままでは破綻するので、3つの改良が示される

### (a) 階層化ページテーブル

- 仮想アドレスを「上位ビット／中位ビット／オフセット」に分け、ページテーブルを**ツリー状の段**にする
- 利点は、**実際に使われていない部分の下位テーブルを作らなくてよい**ことである
- プログラムが実際に使うのはコード・データ・スタックの周辺だけなので、テーブル全体をフルに用意する必要がなくなり、メモリを大幅に節約できる

### (b) セグメントアーキテクチャ

- 空間を「使用中／未使用」の**セグメント**単位で管理する
- セグメントテーブル経由でページテーブルを引く

### (c) インバーテッドページテーブル

- 発想を逆転させ、「仮想ページごと」ではなく**物理ページごと**に1エントリだけ持つ
- 各物理ページフレームのエントリに、対応する（プロセスID, 仮想ページ番号）などを記録する。物理ページ番号は表の位置そのものから分かる
- 変換時は（プロセスID, 仮想ページ番号）をハッシュで検索し、衝突時にはチェインなどで目的のエントリを探す
- テーブルサイズが**物理メモリ量に比例**して済むので、仮想空間がいくら大きくてもテーブルは肥大しない

## 内部フラグメンテーション

- ページは固定長なので、Code 100B / Data 20B / Stack 10B（合計 130B）しか使わなくても、それぞれ別ページに置くと **1024×3 = 3072B** を占有する
- ページ内の使い残しがムダになる
- これが**内部フラグメンテーション**である
- 外部フラグメンテーションと引き換えに、こちらが発生する

## ページフォールト

- **ページフォールト**とは、仮想アドレス変換が現在利用できない場合や、アクセス権に違反した場合にCPUが発生させる例外である
- 発生は**ハードウェア**が行い、処理は**OS**が担当する
- OS は適切なページを割り当てて Valid ビットを On にし、アプリは何事もなかったように続行できる
- ページが未配置なのか、不正なアドレスなのか、書き込み禁止ページへの書き込みなのかをOSが原因情報から判定する。回復不能ならプロセスへシグナルを送るなどして終了させる
- **ハンドラの呼ばれ方**
  - CPUは例外ベクタなどの仕組みに従ってカーネルの例外処理へ制御を移し、フォールトした仮想アドレスや原因をレジスタ等で伝える
  - OSがページを用意して変換を修復できれば、原因となった命令を再実行して処理を継続する
- この「**アクセスされるまで本当の処理を遅らせる（遅延評価／Lazy Evaluation）**」のが、以下すべての最適化の共通原理である

## ページフォールトを使った3つの最適化

### (a) On Demand Page Allocation（オンデマンド割り当て）

- `malloc` やスタック拡張でメモリを「確保」しても、**実際にアクセスされるまで物理ページを割り当てない**
- 初めて触られたときのフォールトで物理ページを与え、必要ならゼロクリアする
- プログラムが本当に使うページだけが割り当てられるので、**大きなスタックを宣言してもメモリのムダにならない**

### (b) Copy On Write（COW）

- `fork()` でプロセスを複製するとき、コード・データを丸ごとコピーするのは非常に高コストである
- そこで、最初は**親子で同じ物理ページを共有**し、PTE を **Read Only** にしておく
- どちらかが**書き込もうとした瞬間**にフォールトが起き、そのとき初めて**そのページだけを実コピー**して、書き込む側に書き込み可能（RW）な独立ページを与える
- **書き換えられないページは最後まで共有されたまま**なので、コピーコストを実際に変更が起きた分だけに抑えられる

### (c) Demand Paging（デマンドページング）

- 物理メモリより大きなプログラムを動かすための仕組みである
- 必要になったページだけを物理メモリへ読み込む。実行ファイルや共有ライブラリ由来のページは元ファイルから読み、追い出された匿名ページなどはスワップ領域から戻す
- ここで**3つのポリシ**が登場する
  - **ページフェッチポリシ**：いつディスクから物理メモリへコピーするか
  - **ページリプレースメントポリシ**：物理メモリが満杯のとき、どのページをスワップへ追い出すか
  - **ページ／フレーム割り当てポリシ**：各プロセスへどれだけの物理ページフレームを配分するか

## ページフェッチポリシとフリーページ管理

- **いつ運ぶか**には2方式ある
  - **オンデマンド（On Demand）**：アクセスされてから運ぶ
  - **プリフェッチ（Prefetch）**：先読みして前もって運ぶ
- ページの種類は、コードエリア／データエリア（Read Only か Read/Write）／スタックや動的に割り当てた領域、に分類される
- **フリーページの管理**
  - 未使用ページは**リスト**として繋いで管理し、新しいページが必要になったらリストの先頭から割り当てる（ゼロクリア済みページが必要なとき、ファイル/ディスクからページを読み込むとき、など）

## ページリプレースメント

- 物理メモリが満杯のとき、
  1. 追い出す物理ページを**選択**する
  2. 必要なら**スワップディスクへコピー**する（**Dirty なページだけ書き戻せばよい**。書き換えていないページは元がディスクにあるので破棄で OK）
  3. PTE の Valid を 1→0 に**変更**する
- 問題は「**どのページを選ぶか**」である
  - **FIFO（First-In First-Out）**：最初にフェッチしたページから追い出す。単純だが、よく使うページも順番が来れば追い出してしまう
  - **LRU（Least Recently Used）**：最近もっとも使われていないページを追い出す。**理想に近い**が、実装が重い ── アクセスのたびにキュー（アクセス時刻順のリスト）を並べ替える必要があり、その**オーバヘッドが現実的でない**
  - **理想的なポリシ（OPT）**：「次に参照される時点が最も遠いページ」を追い出す。未来の参照列が必要なので、実システムでは実装できず比較基準として使われる

## クロックアルゴリズム（LRUの近似）

- LRU を安く近似する実用解が**クロックアルゴリズム**である
- ページを円環状に並べ、針が回りながら**参照ビット**を見ていく
  1. 置換対象が必要になると、針の位置からページを調べる
  2. スキャン時、**R が ON ならば R を OFF にする**（猶予を1回与える）
  3. **R が OFF のページを追い出し候補**とする
  4. よく参照されるページは、再び調べられるまでにRがONへ戻るため生き残りやすい
- つまり「最近触られていないページ＝R がずっと off のまま」を追い出す
- 全ページのRがONでも、1周目でOFFにし、次の走査で再参照されなかったページを選べる。ただし走査量が増えると置換処理のコストが高くなる
- 改良版には、Rを落とす針と回収する針を分ける**2針方式**や、Dirty bitも考慮して書き戻しコストを抑える方式がある
